Security Hardening

1 ALB Security Group: Open vs Restricted Egress

Before -- Insecure
# Default SG: allows ALL outbound
egress {
  cidr_blocks = ["0.0.0.0/0"]
  protocol    = "-1"
  from_port   = 0
  to_port     = 0
}
After -- Hardened
# Restricted: only to Worker SG
resource "aws_vpc_security_group_egress_rule"
  "alb_to_workers" {
  security_group_id            = sg_alb
  referenced_security_group_id = sg_workers
  ip_protocol = "tcp"
  from_port   = 30000
  to_port     = 32767
}

What Changed: Replaced the default 0.0.0.0/0 egress (all traffic, all ports, all destinations) with a single rule targeting only the Worker Security Group on the Kubernetes NodePort range.

Why: If the ALB is compromised (e.g., via HTTP request smuggling), the attacker cannot pivot to databases, Redis, Kafka, or exfiltrate data to the internet. The blast radius is limited to the EKS worker tier.

Attack Prevented: Lateral movement from ALB to data tier

2 Worker Nodes SG: Wide Open vs Per-Service Egress

Before -- Over-Permissive
# Single wide-open egress rule
egress {
  cidr_blocks = ["0.0.0.0/0"]
  protocol    = "-1"
}
After -- Least Privilege
# 5 separate egress rules:
egress → sg_aurora     :5432
egress → sg_msk        :9092-9094
egress → sg_redis      :6379
egress → 0.0.0.0/0     :443  # NAT
egress → sg_vpce       :443

What Changed: Replaced one catch-all egress rule with five targeted rules, each referencing a specific Security Group and port range.

Why: A compromised pod can only reach the services it legitimately needs. It cannot connect to arbitrary internet destinations for C2 beaconing, crypto mining, or data exfiltration on non-443 ports.

Attack Prevented: C2 beaconing, lateral movement, data exfiltration

3 Aurora SG: VPC-Wide Access vs Workers-Only

Before -- CIDR-Based
ingress {
  cidr_blocks = ["10.0.0.0/16"]
  from_port   = 5432
  to_port     = 5432
  protocol    = "tcp"
}
After -- SG-Referenced
resource "aws_vpc_security_group_ingress_rule"
  "aurora_from_workers" {
  security_group_id            = sg_aurora
  referenced_security_group_id = sg_workers
  ip_protocol = "tcp"
  from_port   = 5432
  to_port     = 5432
}

What Changed: Replaced CIDR-based 10.0.0.0/16 ingress with a Security Group reference. Only ENIs attached to the Worker SG can connect.

Why: CIDR-based rules allow ANY resource in the VPC (including a compromised ALB or rogue EC2 instance) to connect to the database. SG-referencing ensures only EKS worker nodes have access, and autoscaling works without rule changes.

Attack Prevented: ALB-to-database pivot, rogue instance access

4 Redis SG: VPC-Wide vs Workers-Only

Before -- VPC-Wide Access
ingress {
  cidr_blocks = ["10.0.0.0/16"]
  from_port   = 6379
  to_port     = 6379
  protocol    = "tcp"
}
After -- Workers-Only
resource "aws_vpc_security_group_ingress_rule"
  "redis_from_workers" {
  security_group_id            = sg_redis
  referenced_security_group_id = sg_workers
  ip_protocol = "tcp"
  from_port   = 6379
  to_port     = 6379
}

What Changed: Same pattern as Aurora -- replaced VPC CIDR ingress with SG-referenced rule. Only Worker SG on port 6379.

Why: Redis stores CSRF tokens, rate-limiting state, and session data. A compromised database or ALB should never be able to dump session tokens or bypass rate limits.

Attack Prevented: Session hijacking via cache dump, rate-limit bypass

5 MSK SG: Adding Inter-Broker Replication

Before -- Missing Inter-Broker
# Only worker access, no self-ref
ingress {
  security_groups = [sg_workers]
  from_port = 9092; to_port = 9094
}
# Missing: inter-broker rule!
After -- Complete Rules
# Workers + self-referencing for replication
ingress → sg_workers :9092-9094
ingress → sg_msk     :9092-9094  # self
egress  → sg_msk     :9092-9094  # self

What Changed: Added self-referencing ingress and egress rules on ports 9092-9094 so Kafka brokers can replicate partitions across AZs.

Why: Without inter-broker rules, Kafka replication breaks silently. ISR (In-Sync Replicas) shrink, and a single broker failure risks data loss. This is a correctness fix, not just security.

Attack Prevented: Silent data loss from broken replication

6 No NACLs vs 3-Tier NACLs

Before -- Default NACL
# Using default NACL
# Allows ALL traffic in ALL
# directions between ALL subnets.
# No defense-in-depth.
After -- 3-Tier NACLs
# Public: 443/80 in, VPC out
# Private: VPC in, VPC+443 out
# Database: 5432 from private only
# ALL end with rule 32766: DENY ALL

What Changed: Created three custom NACLs (public, private, database) replacing the default allow-all NACL. Each tier has explicit allow rules and a final deny-all catch.

Why: Defense-in-depth. If a Security Group is accidentally misconfigured (e.g., someone adds 0.0.0.0/0 to Aurora SG), the database NACL still blocks non-5432 traffic from non-private subnets.

Attack Prevented: SG misconfiguration bypass, subnet-level lateral movement

7 No VPC Flow Logs vs Reject-Only Logging

Before -- Blind
# No flow logs configured.
# Blind to rejected connections,
# port scans, and lateral movement
# attempts.
After -- Reject Logging
resource "aws_flow_log" "reject" {
  vpc_id          = aws_vpc.main.id
  traffic_type    = "REJECT"
  log_destination = aws_cloudwatch_log_group
                      .flow_logs.arn
  max_aggregation_interval = 60
}

resource "aws_cloudwatch_log_group" "flow_logs" {
  name              = "/vpc/bookstore/flow-logs"
  retention_in_days = 30
}

What Changed: Enabled VPC Flow Logs with traffic_type = "REJECT", sending to CloudWatch Logs with 30-day retention.

Why: Without flow logs, you are blind to blocked connection attempts, port scans, and lateral movement. REJECT-only logging captures the security-relevant events at ~90% less cost than logging ALL traffic (~$0.50/GB).

Attack Prevented: Undetected reconnaissance and port scanning

8 No VPC Endpoints vs 6 Endpoints (PrivateLink)

Before -- Via Internet
# No VPC Endpoints configured.
# All AWS API calls (ECR, STS,
# CloudWatch) go through NAT GW
# → Internet → AWS service endpoint.
# Cost: $0.045/GB processing.
After -- PrivateLink
# 6 VPC Endpoints:
# S3 Gateway        — FREE
# ECR API/DKR       — $21.60/mo each
# STS               — $21.60/mo
# CloudWatch Logs   — $21.60/mo
# Secrets Manager   — $21.60/mo
resource "aws_vpc_endpoint" "s3" {
  vpc_endpoint_type = "Gateway"  # Free!
  route_table_ids   = private_rts
}

What Changed: Created 6 VPC Endpoints (1 Gateway for S3, 5 Interface for ECR/STS/CloudWatch/Secrets Manager). AWS API traffic stays on the AWS backbone, never traversing the public internet.

Why: Reduces attack surface (no internet exposure for AWS API calls), improves resilience (works even if NAT GW fails), and reduces NAT data processing charges. S3 Gateway Endpoint is free and handles the bulk of ECR image layer transfers.

Attack Prevented: API call interception via internet Cost Saved: NAT data charges reduced 80%+

9 No EKS Secrets Encryption vs KMS Envelope Encryption

Before -- Plaintext in etcd
# Default: no encryption_config
# K8s Secrets stored as base64 in etcd.
# echo "Q0hBTkdFX01F" | base64 -d
# → "CHANGE_ME"
# Anyone with etcd access reads all.
After -- KMS Envelope Encryption
resource "aws_eks_cluster" "main" {
  encryption_config {
    provider {
      key_arn = aws_kms_key.eks_secrets.arn
    }
    resources = ["secrets"]
  }
}

resource "aws_kms_key" "eks_secrets" {
  enable_key_rotation = true
}

What Changed: Added KMS envelope encryption to the EKS cluster. Each Secret gets a unique Data Encryption Key (DEK), which is itself encrypted by the KMS master key. Auto-rotation enabled.

Why: Without this, anyone with etcd access (or an etcd backup) can read all Kubernetes Secrets (DB passwords, API keys, Kafka credentials) by simply base64-decoding them. With KMS, secrets are unreadable without the KMS key.

Attack Prevented: Secret exfiltration from etcd backup/compromise

Cost Optimization

10 On-Demand x86 vs Spot + Graviton ARM

Before -- Expensive
instance_types = ["t3.xlarge"]
capacity_type  = "ON_DEMAND"
desired_size   = 5
After -- Optimized
# System: Graviton On-Demand
instance_types = ["t4g.medium"]
capacity_type  = "ON_DEMAND"
# App: Spot with diversity
instance_types = ["t3.large", "t3a.large",
                  "m5.large", "m7g.large"]
capacity_type  = "SPOT"

What Changed: Split into system (On-Demand Graviton) and app (Spot, diversified instance types) node groups. ARM processors cost 20% less; Spot saves 60-90%.

Why: Stateless application pods can tolerate 2-minute Spot eviction notices because Kubernetes reschedules them instantly. System components (CoreDNS, Istio control plane) stay on stable On-Demand nodes.

Cost Saved: $585 to $108/mo = 82% reduction ($477/mo saved)

11 Aurora Over-Provisioned vs Serverless v2

Before -- Fixed Capacity
serverlessv2_scaling_configuration {
  min_capacity = 2    # $0.12/ACU-hr x 2
  max_capacity = 8    # = $175/mo minimum
}
# x 4 clusters = $700/mo idle
After -- Auto-Scaling
serverlessv2_scaling_configuration {
  min_capacity = 0.5  # $44/mo minimum
  max_capacity = 8    # scales up under load
}
# x 4 clusters = $176/mo idle

What Changed: Reduced Aurora Serverless v2 minimum capacity from 2 ACU to 0.5 ACU. The database scales down to near-zero when idle and scales up automatically under load.

Why: At 2 ACU minimum, you pay $175/cluster/month even when idle. At 0.5 ACU, idle cost drops to $44/cluster. For 4 clusters, that is $524/mo saved during off-hours.

Cost Saved: ~$344 to ~$172/mo per cluster = 50% reduction

12 Redis Over-Sized vs Right-Sized

Before -- Massively Over-Provisioned
node_type = "cache.r6g.large"
# 13 GB RAM, 2 vCPU
# Actual usage: ~100 MB
# 99.2% of RAM is wasted
After -- Right-Sized
node_type = "cache.t4g.micro"
# 0.5 GB RAM, Graviton, 2 vCPU
# 5x headroom for 100 MB usage
# Production: t4g.small ($24/mo)

What Changed: Downgraded from cache.r6g.large (13 GB RAM) to cache.t4g.micro (0.5 GB RAM). The BookStore uses Redis for CSRF tokens, rate limiting, and sessions -- total usage is under 100 MB.

Why: Paying for 13 GB of RAM when you use 100 MB is a 130x over-provision. Even production with replication (t4g.small x2 = $48/mo) is 4x cheaper than the original single node.

Cost Saved: $200 to $8/mo (dev) = 96% reduction

13 MSK Over-Provisioned vs Right-Sized

Before -- Production Instance for Dev
broker_node_group_info {
  instance_type = "kafka.m5.large"
}
number_of_broker_nodes = 3
# $216/broker = $648/mo total
After -- Right-Sized / Serverless
# Dev: Serverless (pay-per-use)
resource "aws_msk_serverless_cluster" "dev" {}
# Prod: right-sized brokers
instance_type = "kafka.t3.small"
# or kafka.m7g.large (Graviton)

What Changed: Dev environments use MSK Serverless ($0 when idle). Production uses right-sized kafka.t3.small or Graviton kafka.m7g.large instead of over-provisioned kafka.m5.large.

Why: The BookStore Kafka topics handle modest throughput (order events, CDC). A kafka.m5.large broker can handle 100K messages/sec; actual usage is under 1K/sec.

Cost Saved: $648 to $150/mo (prod) = 77% reduction

14 3 NAT Gateways vs 1 NAT + VPC Endpoints

Before -- 3 NAT Gateways
# 3 NAT GWs (one per AZ)
resource "aws_nat_gateway" "main" {
  count = 3
}
# All AWS API traffic via NAT
# $32/mo fixed + $0.045/GB data
After -- 1 NAT + Endpoints
# 1 NAT GW for dev (3 for prod HA)
resource "aws_nat_gateway" "main" {
  count = var.env == "prod" ? 3 : 1
}
# S3 Gateway: FREE (bulk of traffic)
# Interface endpoints: ~$108/mo
# NAT data charges drop 80%+

What Changed: Dev uses 1 NAT Gateway instead of 3 (saves $90/mo). VPC Endpoints handle S3/ECR/STS/CloudWatch/Secrets Manager traffic without NAT, reducing data charges by 80%+. S3 Gateway Endpoint is free and carries the bulk of ECR image layer traffic.

Why: NAT Gateways are among the most expensive networking components. For dev, single-AZ NAT is acceptable (cluster survives, just loses outbound temporarily). VPC Endpoints improve both security (no internet exposure) and cost.

Cost Saved: $135 to ~$55/mo (dev) = 59% reduction on NAT

15 No Resource Tags vs Full Cost Attribution

Before -- Untagged
# Resources created without tags.
# Cost Explorer: one big number.
# No way to filter by team,
# environment, or service.
# No automated cleanup possible.
After -- Full Cost Attribution
provider "aws" {
  default_tags {
    tags = {
      Project     = "bookstore"
      Environment = var.environment
      Team        = "platform"
      ManagedBy   = "terraform"
      CostCenter  = var.cost_center
    }
  }
}

What Changed: Added default_tags to the AWS provider so every resource gets Project, Environment, Team, ManagedBy, and CostCenter tags automatically. No resource can be created untagged.

Why: Tags enable Cost Explorer filtering (which team, which env is expensive?), automated cleanup of dev resources, budget alerts per cost center, and compliance auditing. Without tags, a $3,000 bill is an opaque number with no accountability.

Cost Saved: Enables identification and elimination of waste
