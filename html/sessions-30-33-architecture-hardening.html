■ Session 30 — Container & Network Security

#ChangeImpact
1.dockerignore for all 3 servicesPrevents .git, .env, node_modules from entering images
2Gateway egress restrictedNo more open egress — only ecom(8080), inventory(8000), identity(8080), UI(80)
3Kafka ingress restricted to named podsOnly Debezium, Schema Registry, Kafka Exporter can reach Kafka
4Inventory CORS: removed DELETEReduces CORS attack surface — no DELETE endpoints exist
5Cert-dashboard RBAC trimmedRemoved create/delete on ClusterRoles (least privilege)
6Configurable logging via ${LOG_LEVEL:INFO}Runtime-configurable log level, no more hardcoded DEBUG
7PSS restricted on ecom & inventoryEnforces seccompProfile, non-root, drop ALL capabilities
Smaller, Safer Images

.dockerignore prevents secrets and dev artifacts from leaking into production container images

Network Segmentation

Gateway can only reach its backends — lateral movement to other namespaces is blocked

Kafka Isolation

Only CDC components (Debezium, Schema Registry) can access Kafka topics

Admission-Time Enforcement

PSS restricted means Kubernetes rejects non-compliant pods at creation time

■ Session 31 — Application Layer Security

#ChangeImpact
1JWT audience validation (ecom-service)JwtClaimValidator checks aud contains account
2Inventory verify_aud: False fixedBoth JWT decode paths now validate audience
3jwt_audience config settingConfigurable via environment variable (default: account)
4@Max(99) on CartRequest.quantityPrevents integer overflow and unreasonable orders
5@Max(99) on CartUpdateRequest.quantitySame protection for cart updates
6le=99 on ReserveRequest.quantityInventory rejects reserve > 99 at validation layer
Token Confusion Prevention

JWT audience validation prevents tokens issued for one client from being used against another service

Input Bounds Enforcement

Quantity bounds 1-99 eliminate edge cases with zero, negative, or absurdly large values

Defense in Depth

Validation at both API gateway (Spring Boot) and downstream service (FastAPI) layers
