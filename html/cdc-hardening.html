The CDC (Change Data Capture) pipeline replicates every database change from the ecom and inventory PostgreSQL databases into a central analytics database in real time. Debezium Server reads the PostgreSQL WAL (Write-Ahead Log), publishes change events to Kafka, where Apache Flink transforms and loads them into a star schema for Superset dashboards.

Problem: The existing pipeline had a solid foundation but several critical gaps that would cause data loss, silent corruption, or operational blindness at production scale. This session addresses all of them.

Phase 1: Prevent Data Loss

1. Flink Restart Strategy & Job Submission Resilience

Before

  • SQL Runner backoffLimit: 0 — if SQL Gateway is temporarily unavailable, pipeline never starts
  • No restart strategy — a transient JDBC sink failure kills streaming jobs permanently
  • Zero tolerance for checkpoint failures

After

  • SQL Runner backoffLimit: 3 — retries on cold-start timing issues
  • fixed-delay restart: 10 attempts, 30s delay between retries
  • Tolerates up to 3 consecutive checkpoint failures before aborting

Why: The Flink SQL Runner submits 4 streaming INSERT statements via SQL Gateway. During cold starts, the Gateway may not be ready when the Job runs. With backoffLimit: 0, the entire CDC pipeline fails permanently. The restart strategy prevents transient JDBC connection hiccups from killing long-running streaming jobs.

2. Kafka Producer Idempotency

Before

  • acks=all + retries=3 but no idempotent producer
  • Retries after network timeout could produce duplicate order.created events
  • No max.in.flight.requests.per.connection cap

After

  • enable.idempotence=true — broker deduplicates retried messages
  • max.in.flight.requests.per.connection=5 — optimal for idempotent mode
  • Exactly-once delivery from ecom-service producer to Kafka broker

Why: The ecom-service publishes order.created events on checkout. Without idempotency, a retry after a timeout could produce duplicate messages, leading to double-counted orders in analytics. Kafka's idempotent producer assigns sequence numbers for transparent deduplication.

Phase 2: Observability & Monitoring

3. Kafka Consumer Lag Monitoring

Before

  • No visibility into whether Flink is falling behind on consumption
  • No Kafka broker metrics in Prometheus at all
  • Stale analytics dashboards undetectable

After

  • kafka-exporter (v1.9.0) deployed in infra namespace
  • kafka_consumergroup_lag per group, topic, and partition
  • NetworkPolicy: Prometheus scrapes port 9308, exporter reads from Kafka 9092

4. Flink Prometheus Metrics Reporter

Before

  • Flink Web Dashboard (:32200) is visual only — no history, no alerting
  • No Flink metrics in Prometheus/Grafana
  • Checkpoint health invisible to monitoring

After

  • flink-metrics-prometheus-1.20.0.jar baked into Flink image
  • Prometheus reporter on port 9249 (JobManager + TaskManager)
  • All Flink metrics flow to Prometheus: checkpoints, throughput, GC, buffers

5. CDC End-to-End Latency Measurement

Before

  • synced_at DEFAULT NOW() column exists in analytics tables
  • No view or dashboard to measure DB-write → analytics-arrival latency

After

  • vw_cdc_latency view: latency_seconds = synced_at - created_at
  • Ready for P50/P95/P99 latency dashboarding in Grafana or Superset

CDC Alert Rules

Four new Prometheus alert rules provide automated detection of CDC pipeline failures:

FlinkJobNotRunning
Flink JobManager or TaskManager is down for 2+ minutes. CDC pipeline is completely stopped.

up{job=~"flink-*"} == 0

FlinkCheckpointsFailing
More than 3 checkpoint failures in 10 minutes. Potential data loss risk if jobs crash.

increase(...numberOfFailedCheckpoints[10m]) > 3

KafkaConsumerLagHigh
Flink analytics consumer is 1000+ messages behind for 5+ minutes. Dashboards showing stale data.

kafka_consumergroup_lag > 1000

DebeziumPodNotReady
Debezium Server pod is not ready for 2+ minutes. CDC events are not being captured from source databases.

kube_pod_status_ready == 0

Combined with existing alerts: These CDC-specific rules complement the existing ServiceDown, HighErrorRate, PodRestartLoop, and HighLatency alerts, providing full pipeline coverage from source databases through to analytics.

Phase 3: Availability

Debezium PodDisruptionBudgets

Before

  • No PDB for Debezium Server pods
  • Both pods could be evicted simultaneously during node drain
  • CDC capture stops entirely during cluster maintenance

After

  • minAvailable: 1 PDB for debezium-server-ecom
  • minAvailable: 1 PDB for debezium-server-inventory
  • At least one Debezium pod always available during voluntary disruptions

Bootstrap Port Conflict Resolution

Before

  • kind create cluster fails with opaque "port already allocated" error
  • Manual docker stop required to find and stop conflicting containers

After

  • free_required_ports() auto-detects containers blocking any of 11 required ports
  • Shows container names and their bound ports, prompts to stop them
  • Verifies all ports are free before proceeding with cluster creation

Debezium Metrics: Discovery

Debezium Server 3.4 limitation: The Debezium Server image bundles OpenTelemetry Prometheus exporter JARs but does not include the Quarkus Micrometer extension. Setting quarkus.micrometer.export.prometheus.enabled=true in application.properties is silently ignored — /q/metrics returns 404.

Mitigation: Consumer lag (the most critical metric) is now tracked via kafka-exporter. Pod readiness is monitored via kube_pod_status_ready. Full Debezium JMX metrics require a jmx_exporter sidecar container (deferred to a future session).
