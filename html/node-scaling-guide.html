Node Group Scaling Guide

Your cluster started with max_size = 5. Now you need more. Here's exactly what to do — and what NOT to do.

Zero Downtime EKS + AKS Terraform + CLI Industry Best Practice
Key fact: Increasing max_size on an existing node group is a non-disruptive, zero-downtime operation in both EKS and AKS. No pods are restarted. No traffic is interrupted. You're just telling the autoscaler it's allowed to provision more nodes.

The 3-Tier Scaling Strategy (Industry Standard)

Production Kubernetes clusters scale in three tiers. Always try the cheapest/fastest tier first:

1 Scale Pods (HPA) — Free, instant, within existing nodes Pod CPU > 70%? HPA adds replicas. No new nodes needed. ↓ If all nodes are full (pods stuck in Pending)... 2 Scale Nodes (update max_size) — Zero-downtime config change Increase max_size from 5 → 10. Autoscaler provisions new nodes. ↓ If workloads need different instance types... 3 Add New Node Groups — For specialized workloads GPU nodes for ML, high-memory for databases, Spot for batch jobs. Runs alongside existing node groups.

Tier 2: Scale Existing Node Groups

When to Do This

  • Pods are in Pending state (no available nodes with enough resources)
  • kubectl get events shows FailedScheduling: Insufficient cpu/memory
  • Cluster Autoscaler logs show max node group size reached

AWS EKS — Three Options

Option A: AWS CLI (immediate, no Terraform)

# Check current scaling config
aws eks describe-nodegroup \
  --cluster-name bookstore \
  --nodegroup-name bookstore-app-nodes \
  --query 'nodegroup.scalingConfig'

# Update max_size from 5 to 10 (zero downtime)
aws eks update-nodegroup-config \
  --cluster-name bookstore \
  --nodegroup-name bookstore-app-nodes \
  --scaling-config minSize=2,maxSize=10,desiredSize=5

# Verify
aws eks describe-nodegroup \
  --cluster-name bookstore \
  --nodegroup-name bookstore-app-nodes \
  --query 'nodegroup.scalingConfig'
This takes effect immediately. The autoscaler can now provision up to 10 nodes. No pods restart.

Option B: Terraform (recommended for GitOps)

# variables.tf — change the default
variable "app_node_max" {
  description = "Maximum nodes in app node group"
  type        = number
  default     = 10   # was 5
}

# Apply
terraform plan -var="app_node_max=10"
terraform apply -var="app_node_max=10"
Best practice: Always update Terraform so your infrastructure-as-code matches reality. CLI changes create drift.

Option C: eksctl (if using eksctl-managed clusters)

eksctl scale nodegroup \
  --cluster bookstore \
  --name bookstore-app-nodes \
  --nodes-max 10

Azure AKS — Three Options

Option A: Azure CLI (immediate)

# Check current config
az aks nodepool show \
  --resource-group bookstore-rg \
  --cluster-name bookstore \
  --name apppool \
  --query '{minCount:minCount, maxCount:maxCount, count:count}'

# Update max_count from 5 to 10
az aks nodepool update \
  --resource-group bookstore-rg \
  --cluster-name bookstore \
  --name apppool \
  --max-count 10

# Verify
az aks nodepool show \
  --resource-group bookstore-rg \
  --cluster-name bookstore \
  --name apppool \
  --query '{minCount:minCount, maxCount:maxCount}'

Option B: Terraform (recommended)

# variables.tf
variable "app_node_max_count" {
  description = "Maximum nodes in app pool (autoscaler)"
  type        = number
  default     = 10   # was 5
}

terraform apply -var="app_node_max_count=10"

Option C: Azure Portal (GUI)

  1. Navigate to your AKS cluster → Node pools
  2. Click the node pool → Scale
  3. Update Maximum node count to 10
  4. Click Apply

EKS: Add a New Managed Node Group

# Terraform
resource "aws_eks_node_group" "kafka_nodes" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "bookstore-kafka-nodes"
  node_role_arn   = aws_iam_role.node.arn
  subnet_ids      = var.private_subnet_ids

  instance_types = ["r7g.xlarge"]

  scaling_config {
    desired_size = 2
    min_size     = 2
    max_size     = 4
  }

  labels = {
    "workload" = "streaming"
  }

  taint {
    key    = "workload"
    value  = "streaming"
    effect = "NO_SCHEDULE"
  }
}

AKS: Add a New Node Pool

# Terraform
resource "azurerm_kubernetes_cluster_node_pool" "kafka" {
  name                  = "kafkapool"
  kubernetes_cluster_id = azurerm_kubernetes_cluster.main.id
  vm_size              = "Standard_E4as_v5"
  zones                = ["1", "2", "3"]

  min_count            = 2
  max_count            = 4
  enable_auto_scaling  = true

  node_labels = {
    "workload" = "streaming"
  }

  node_taints = ["workload=streaming:NoSchedule"]
}

Deploy to the New Node Group

# Add nodeSelector + tolerations to your Kafka deployment
spec:
  nodeSelector:
    workload: streaming
  tolerations:
    - key: "workload"
      operator: "Equal"
      value: "streaming"
      effect: "NoSchedule"

Cluster Autoscaler vs Karpenter

FeatureCluster AutoscalerKarpenter
How it worksAdjusts ASG/VMSS desired countProvisions individual EC2 instances directly
Cloud supportEKS + AKS + GKEEKS only (AKS: preview as AKS Karpenter Provider)
Instance type flexibilityOne type per node groupMixes types per NodePool (e.g., m7g + c7g + r7g)
Scale-up speed30-60 seconds10-20 seconds
Spot + On-Demand mixSeparate node groupsSingle NodePool with capacity-type weights
Zone awarenessVia ASG AZ configNative topologySpreadConstraints
Right-sizingNo (fixed instance type)Yes (chooses optimal type per pod requirements)
Setup complexityLow (Helm chart)Medium (CRDs + IAM)
MaturityStable (5+ years)Stable (graduated 2024)
Recommendation: Use Cluster Autoscaler for AKS (only option) and for simple EKS setups. Use Karpenter for EKS when you need fast scaling, Spot/On-Demand mixing, or multi-instance-type flexibility.

Install Cluster Autoscaler (EKS)

helm repo add autoscaler https://kubernetes.github.io/autoscaler
helm install cluster-autoscaler autoscaler/cluster-autoscaler \
  --namespace kube-system \
  --set autoDiscovery.clusterName=bookstore \
  --set awsRegion=us-east-1 \
  --set rbac.create=true \
  --set rbac.serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::ACCOUNT:role/cluster-autoscaler

Install Cluster Autoscaler (AKS)

AKS has the autoscaler built in. Just set enable_auto_scaling = true on your node pool. No separate installation needed.

What NOT to Do

Never delete and recreate a node group to change max_size. This drains all pods, causes downtime, and is completely unnecessary. Just update the scaling config.
Never set desired_size = max_size permanently. This disables autoscaling down, wasting money on idle nodes. Let the autoscaler manage desired count.
Don't scale nodes without checking pod resource requests. If pods request 4 CPU but only use 0.5 CPU, adding nodes won't help — you're wasting capacity. Right-size pod requests first.

Monitoring: How to Know When to Scale

# Check for Pending pods (need more nodes)
kubectl get pods -A --field-selector=status.phase=Pending

# Check node resource utilization
kubectl top nodes

# Check autoscaler status (EKS)
kubectl -n kube-system logs deploy/cluster-autoscaler --tail=20 | grep -i "scale"

# Check node group current vs max
aws eks describe-nodegroup --cluster-name bookstore --nodegroup-name bookstore-app-nodes \
  --query 'nodegroup.scalingConfig'

# AKS equivalent
az aks nodepool show --resource-group bookstore-rg --cluster-name bookstore \
  --name apppool --query '{count:count, minCount:minCount, maxCount:maxCount}'
