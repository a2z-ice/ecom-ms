1. The Problem: 6 Recurring Test Failures

Six E2E tests failed repeatedly across multiple test runs. Every failure traced back to the same root cause: Flink CDC streaming jobs were not running.

Impact: Any Docker Desktop restart, Flink pod restart, or cluster recovery would kill the streaming jobs permanently — causing persistent test failures until someone manually resubmitted the pipeline.

How the Pipeline Is Deployed

The 4 Flink streaming jobs are submitted via a Kubernetes Job (flink-sql-runner) that:

  1. Waits for the Flink SQL Gateway to be ready (init container with curl polling)
  2. Submits 4 INSERT INTO streaming SQL statements via sql-client.sh gateway
  3. Each statement becomes a long-running Flink streaming job that reads from Kafka and writes to the analytics DB
  4. The Job completes (exit 0) after submission — the streaming jobs continue running independently

3. Root Cause Analysis

Why Flink Jobs Disappear

The flink-sql-runner Kubernetes Job is a one-shot submission. It submits 4 streaming SQL statements and exits. The streaming jobs run inside the Flink TaskManager.

When any of these events occur, the streaming jobs are permanently lost:

  • Docker Desktop restart — all pods restart, Flink TaskManager loses in-memory job state
  • Flink JobManager/TaskManager pod restart — no checkpoint/savepoint recovery configured
  • Kind cluster recovery — pods get new IPs, connections break

The Kubernetes Job has status: Complete, so kubectl apply won't re-run it. The streaming jobs are gone, and nothing triggers resubmission.

Normal startup: flink-sql-runner Job → submits 4 INSERT INTO → Job completes → 4 streaming jobs RUNNING After Docker restart: Flink pods restart → streaming jobs lost → Job still "Complete" → 0 streaming jobs, nobody resubmits Cascade effect: No Flink jobs → No data flows to analytics DB → Superset has no data → 6 E2E tests fail

Why Superset Dashboard Tests Fail

The Superset "Book Store Analytics" dashboard displays charts built from dim_books, fact_orders, and fact_inventory tables. When the Flink pipeline is down:

  • Tables exist but contain stale or no data
  • Charts render as empty — no .chart-container elements appear in the DOM
  • The test waits 45 seconds for chart containers, then times out

5. How to Fix Manually

If you've diagnosed that Flink jobs are missing, resubmit the pipeline:

# Step 1: Delete the completed Job (required — K8s won't re-run a completed Job)
kubectl delete job flink-sql-runner -n analytics

# Step 2: Re-apply the Job manifest (ConfigMap + Job)
kubectl apply -f infra/flink/flink-sql-runner.yaml

# Step 3: Wait for completion (the Job submits SQL, then exits)
kubectl wait --for=condition=complete job/flink-sql-runner -n analytics --timeout=120s

# Step 4: Verify 4 jobs are RUNNING
curl -s http://localhost:32200/jobs | python3 -m json.tool
Expected: 4 streaming jobs transition to RUNNING state within 30 seconds of Job completion.

7. The Self-Healing Solution

The core fix is in e2e/global-setup.ts — a function called ensureFlinkJobsRunning() that runs before every test session:

ensureFlinkJobsRunning() Flow: 1 curl http://localhost:32200/jobs ↓ 2 Count jobs with status === "RUNNING" ↓ 3 If ≥ 4: HEALTHY → log and return If < 4: UNHEALTHY → continue to recovery ↓ 4 kubectl delete job flink-sql-runner -n analytics --ignore-not-found ↓ 5 kubectl apply -f infra/flink/flink-sql-runner.yaml ↓ 6 kubectl wait --for=condition=complete --timeout=90s ↓ 7 Verify: curl /jobs → count RUNNING ≥ 4 ↓ 8 waitForAnalyticsData() → poll dim_books until count > 0 (up to 60s) ↓ RECOVERED — all tests can proceed

Recovery Timing

PhaseDurationWhat Happens
Health check~1scurl Flink /jobs API
Job deletion~2sRemove completed K8s Job
Job creation~1sApply ConfigMap + Job manifest
Init container~5-15sWait for SQL Gateway readiness
SQL submission~10-20sSubmit 4 INSERT INTO statements
Data propagation~10-30sDebezium snapshot → Kafka → Flink → analytics DB
Total recovery~30-70sFull pipeline restored with data
When pipeline is healthy: The health check takes ~1 second and logs "Flink CDC pipeline healthy (4/4 jobs RUNNING)". No recovery needed — zero overhead on normal runs.

8. Superset Dashboard Test Fix

The Superset dashboard test had two problems:

Problem 1: Stale/Missing Data

Root cause: Flink pipeline was down, so analytics tables had no data. Charts rendered as empty containers (or not at all).

Fix: The ensureFlinkJobsRunning() recovery in global-setup ensures data is available before any test runs. The waitForAnalyticsData() function polls dim_books to confirm CDC data has propagated.

Problem 2: Fragile Chart Selectors

Root cause: Superset renders charts using different DOM structures depending on the chart type and version. The original selectors missed some rendering modes.

Fix: Added broader selectors (.dashboard-chart, .slice_container), switched from waitForLoadState('load') to waitForLoadState('networkidle'), and increased timeouts.

9. Flaky UI Test Fix

The "minus button removes cart item" test was flaky due to two issues:

Issue 1: CSRF Retry Matching

The waitForResponse matched any /ecom/cart response, including the intermediate 403 CSRF retry. This meant the test proceeded before the actual successful response arrived.

// BEFORE: matches 403 CSRF retry
page.waitForResponse(r => r.url().includes('/ecom/cart'))

// AFTER: only matches successful response
page.waitForResponse(r => r.url().includes('/ecom/cart') && r.status() < 400)

Issue 2: DOM Re-render Timing

After the server confirms item removal, React needs time to re-render the cart table and remove the row. The 10-second timeout was sometimes insufficient.

// BEFORE: 10s timeout
await expect(rows).toHaveCount(rowCountBefore - 1, { timeout: 10000 })

// AFTER: 15s timeout
await expect(rows).toHaveCount(rowCountBefore - 1, { timeout: 15000 })
