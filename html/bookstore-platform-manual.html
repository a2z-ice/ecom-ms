Platform Overview
Architecture, technology stack, and system topology

Infrastructure Architecture

The BookStore platform runs on a local Kubernetes (kind) cluster with Istio Ambient Mesh providing zero-trust networking. All external traffic is encrypted via TLS with certificates managed by cert-manager. The diagram below shows the complete infrastructure topology.

Live Data Flow

Data flows from the user's browser through TLS-terminated gateway endpoints, across mTLS-encrypted service mesh, through event-driven CDC pipelines, and into real-time analytics dashboards.

Bootstrap the Cluster

The smart startup script auto-detects the current state and takes the appropriate action: fresh bootstrap if no cluster exists, recovery if the cluster is degraded, or health verification if everything is running.

2Run the bootstrap script

# First time — full bootstrap (creates cluster, builds images, deploys everything)
bash scripts/up.sh

# Force a clean rebuild from scratch
bash scripts/up.sh --fresh

# Skip confirmation prompts
bash scripts/up.sh --fresh --yes

The bootstrap process takes approximately 8-12 minutes and performs the following steps:

  1. Creates a 3-node kind cluster with NodePort mappings
  2. Installs Istio Ambient Mesh and Kubernetes Gateway API
  3. Builds and loads all Docker images into the cluster
  4. Deploys all infrastructure (databases, Kafka, Redis, Flink)
  5. Imports the Keycloak realm configuration
  6. Registers Debezium CDC connectors
  7. Runs a full smoke test to verify the deployment

3Trust the self-signed CA (optional, for browser access)

# Extract CA cert and add to macOS Keychain
bash scripts/trust-ca.sh --install

This extracts the self-signed CA certificate from the cluster and adds it to your macOS Keychain. After trusting the CA, browsers will show a green padlock for all https://*.net:30000 URLs. For curl commands, use the -sk flag instead.

Verification

After bootstrap, verify that all components are healthy.

4Run the smoke test

# Quick health check — pods, HTTPS routes, Kafka, Debezium
bash scripts/smoke-test.sh

# Comprehensive check — all of the above plus detailed pod status
bash scripts/sanity-test.sh

# Verify CDC pipeline end-to-end — inserts a test row, polls analytics DB
bash scripts/verify-cdc.sh

5Verify HTTPS endpoints manually

# Storefront
curl -sk https://myecom.net:30000/ | head -20

# E-Commerce API
curl -sk https://api.service.net:30000/ecom/books | jq .

# Inventory API
curl -sk https://api.service.net:30000/inven/health

# Keycloak
curl -sk https://idp.keycloak.net:30000/realms/bookstore/.well-known/openid-configuration | jq .issuer
After Docker Desktop Restart: If Docker Desktop is restarted, simply run bash scripts/up.sh again. It auto-detects the degraded state and runs full recovery (ztunnel restart, pod rolling restart, Debezium re-registration).

Book Store Application
Customer experience — browsing, login, cart, and checkout

1Open the Store

Navigate to https://myecom.net:30000 in your browser.

First-time users see the book catalog with cover images, prices, and real-time stock badges. No login is required to browse the catalog.

2Login via Keycloak

Click Login in the navbar. You are redirected to the Keycloak OIDC provider:

https://idp.keycloak.net:30000/realms/bookstore/protocol/openid-connect/auth?...

Customer credentials: user1 / CHANGE_ME

The login uses Authorization Code Flow with PKCE (S256) — no client secret is needed. Tokens are stored in memory only (never localStorage).

3Browse & Add to Cart

After login, your username appears in the navbar and a Logout button is shown.

Stock badges indicate availability in real time:

  • In Stock — sufficient inventory
  • Only X left — low stock warning
  • Out of Stock — unavailable, Add to Cart disabled

Click Add to Cart on any in-stock book — the cart badge in the navbar updates immediately. Use the search bar for title/author lookup.

4Checkout

Navigate to the Cart page to verify items with quantities and availability.

Click Checkout (enabled only when all items have sufficient stock). Behind the scenes:

  1. Order created in the ecom database
  2. Stock reserved via mTLS call to inventory service
  3. order.created event published to Kafka
  4. Cart cleared on success

Idempotency: Include an Idempotency-Key header to prevent double orders. The server deduplicates requests with the same key within a time window.

5Guest Cart

Guest users (unauthenticated) can browse the catalog and add items to their cart without logging in. The guest cart is stored in localStorage under the key bookstore_guest_cart.

  • On login, guest cart items are automatically merged into the server-side cart
  • Checkout always requires authentication — guests are prompted to log in

Security Architecture

Defense-in-depth from network edge to container runtime

Before & After Comparison

LayerBefore (Day 0)After (Session 35)
External trafficPlaintext HTTPHTTPS with auto-rotating certs (cert-manager)
Internal trafficPlaintextIstio mTLS STRICT (all namespaces)
AuthenticationNoneKeycloak OIDC + PKCE (S256)
AuthorizationNoneJWT + RBAC + AuthorizationPolicy
Network isolationOpen (any pod to any)Default-deny NetworkPolicies per namespace
Container securityRoot, writable FSNon-root + readOnlyRootFS + seccomp + drop ALL
Rate limitingNonePer-user Bucket4j + Redis (60 cart, 10 checkout/min)
CSRF protectionNoneRedis-backed CSRF tokens
Token storageN/AMemory only (never localStorage — prevents XSS theft)
API docsPublicly accessibleDisabled in production
JWT validationN/ASignature + issuer + audience claim
SecretsHardcodedKubernetes Secrets + secretKeyRef only
Image tags:latest (mutable)Git SHA tags (immutable references)
IdempotencyNoneIdempotency-Key header (prevents double orders)
Why defense-in-depth? Every security layer operates independently. Even if one layer is bypassed, the remaining layers prevent escalation. mTLS authenticates pods. JWT authenticates users. NetworkPolicies isolate namespaces. RBAC restricts operations. This defense-in-depth approach means a compromised component cannot access resources outside its authorized scope.
