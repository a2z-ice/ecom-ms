1. The Problem: 30-Minute Attack Window

The CSRF service stores tokens in Redis with a fixed 30-minute TTL. Once generated, the token lives for 30 minutes regardless of user activity. This creates a significant security concern:

Attack Scenario

T+0:00
User logs in. CSRF token generated, stored in Redis with 30min TTL.
T+0:05
Attacker steals the token (via XSS, log leak, or network sniffing).
T+0:05 to T+0:30
25-minute attack window. Attacker can use the stolen token for any mutating request.
T+0:30
Token expires. Attack window closes.

Additionally, after a POST request consumes the single-use token, the next POST requires a 3-step retry:

Old flow (3 requests): 1 POST /ecom/cart → 403 Forbidden (token expired/consumed) 2 GET /csrf/token → 200 {"token":"<new-uuid>"} (extra round trip) 3 POST /ecom/cart → 200 OK (retry with new token)

Two Problems to Solve

ProblemImpactSolution
30-minute fixed TTLWide attack window for stolen tokensSliding TTL (10 min, resets on activity)
Extra round trip on retryLatency on every token expiration/consumptionAuto-regeneration in 403 response body

Standards Alignment

StandardRecommendationOur Implementation

NIST SP 800-63B15-minute idle timeout for AAL2 (multi-factor)10-minute idle TTL (more conservative)
OWASP ASVS v4Tokens must be unique per session and short-livedUUID v4, single-use, 10min sliding with auto-regen

Security Properties

Why This Is Safe

  • Active users unaffected: Every GET request (page load, navigation) refreshes the TTL. Normal browsing keeps the token alive indefinitely.
  • Idle tokens expire 3x faster: 10 min vs 30 min — significantly reduces stolen token viability.
  • Auto-regen is CORS-safe: The 403 response body with the new token cannot be read cross-origin. Only same-origin JavaScript (our UI) can access it.
  • Token still user-bound: New token is stored under csrf:{userID} — only usable with that user's JWT.
  • Single-use remains primary defense: Tokens are consumed on first successful mutating request, regardless of TTL.

3. Enhancement 1: Sliding TTL

Every authenticated safe method request (GET, HEAD, OPTIONS) through the Istio ext_authz gateway automatically refreshes the CSRF token's TTL in Redis.

Sliding TTL Flow (ext_authz safe method path): Browser → GET /ecom/books → Envoy ext_authz → csrf-service ↓ Is method safe? YES Is JWT present? YES ↓ Redis EXPIRE csrf:{sub} 600 (fire-and-forget goroutine) ↓ 200 OK → request proceeds to backend Latency impact: ZERO (goroutine runs after response is sent)

How It Works

  1. Envoy ext_authz sends the request to csrf-service for every HTTP method
  2. For safe methods (GET/HEAD/OPTIONS), the handler returns 200 OK immediately
  3. New: If CSRF_SLIDING_TTL=true and JWT is present, a fire-and-forget goroutine runs Redis EXPIRE csrf:{sub} 600
  4. The EXPIRE command is O(1) — it just updates the key's TTL without reading the value
  5. If the key doesn't exist (no token), EXPIRE returns 0 — a harmless no-op

Timeline Example

T+0:00
Login → CSRF token generated. Redis: csrf:user123 = uuid TTL 600s
T+3:00
Browse catalog (GET /ecom/books) → TTL refreshed to 600s
T+5:30
Search books (GET /ecom/books/search) → TTL refreshed to 600s
T+8:00
View book detail (GET /ecom/books/1) → TTL refreshed to 600s
T+9:00
Add to cart (POST /ecom/cart) → Token validated & consumed. TTL irrelevant.

As long as the user is browsing (making GET requests), the CSRF token never expires.

4. Enhancement 2: Auto-Regeneration on 403

When a mutating request fails CSRF validation but the JWT is valid, the csrf-service automatically generates a new token and includes it in the 403 response body.

Auto-Regeneration Flow: Browser → POST /ecom/cart (expired/missing CSRF token) → csrf-service ↓ CSRF validation fails JWT valid? YES ↓ Generate new UUID token Store in Redis: csrf:{sub} = new-uuid TTL 600s ↓ 403 Forbidden Body: {"...","token":"<new-uuid>"} ↓ Browser reads token from 403 body → stores in memory → retries POST → 200 OK

Response Format

HTTP/1.1 403 Forbidden
Content-Type: application/json

{
  "type": "about:blank",
  "title": "Forbidden",
  "status": 403,
  "detail": "Invalid or missing CSRF token",
  "token": "d30fdd4d-82d2-44b0-a593-c9102cc718c6"    ← NEW: regenerated token
}
Key insight: The token field is only present when the JWT is valid (authenticated user). Unauthenticated requests pass through to the backend without CSRF checks (backend handles auth).

5. Before vs After Comparison

Token Lifecycle

BEFORE
  • Fixed 30-minute TTL
  • No refresh on safe methods
  • Token expires after 30 min regardless of activity
  • 403 retry requires separate GET /csrf/token call
  • 3 HTTP requests on expiry: 403 + GET + retry
AFTER
  • 10-minute sliding TTL
  • Refreshed on every authenticated GET/HEAD/OPTIONS
  • Active users' tokens never expire
  • 403 response includes new token in body
  • 2 HTTP requests on expiry: 403 (with token) + retry

Attack Window

BEFORE

30 minutes

A stolen token is valid for up to 30 minutes, regardless of whether the legitimate user is active.

AFTER

10 minutes (idle only)

A stolen token expires in 10 minutes of user inactivity. If the user is browsing, the token's TTL keeps resetting — but it's also single-use, so the first mutating request by either party consumes it.

Server Code: ext_authz Safe Method Handler

BEFORE
if SafeMethods[r.Method] {
    h.Metrics.RequestsTotal.
        WithLabelValues("authz_safe", "ok").Inc()
    w.WriteHeader(http.StatusOK)
    return
}
// No TTL refresh. Token slowly expires.
AFTER
if SafeMethods[r.Method] {
    h.Metrics.RequestsTotal.
        WithLabelValues("authz_safe", "ok").Inc()
    if h.SlidingTTL {
        authHeader := r.Header.Get("Authorization")
        claims := jwt.ExtractClaims(authHeader)
        if claims.Sub != "" {
            go func() {
                ctx, cancel := context.WithTimeout(
                    context.Background(), 1*time.Second)
                defer cancel()
                h.Store.RefreshTTL(ctx, claims.Sub)
            }()
        }
    }
    w.WriteHeader(http.StatusOK)
    return
}

6. Request Flow Diagrams

Active User Flow (No Expiry)

Active user — token never expires: T+0:00 Login → GET /csrf/token → token=abc123, TTL=600s T+2:00 GET /ecom/books → sliding TTL → TTL reset to 600s T+4:00 GET /ecom/books/1 → sliding TTL → TTL reset to 600s T+5:00 POST /ecom/cart → 200 OK (token consumed) T+5:01 GET /ecom/books → sliding TTL → no token to refresh (no-op) T+7:00 POST /ecom/checkout → 403 + token=xyz789 (auto-regen) T+7:01 POST /ecom/checkout → 200 OK (retry with xyz789)

Idle User Returning

Idle user — token expires after 10 min: T+0:00 Login → token=abc123, TTL=600s T+1:00 GET /ecom/books → TTL reset to 600s ... (user goes idle for 12 minutes) T+13:00 POST /ecom/cart → 403 + token=newtoken (auto-regen!) T+13:01 POST /ecom/cart → 200 OK (retry, transparent to user)

Stolen Token Scenario

Attacker steals token while user is active: T+0:00 Legitimate user gets token=abc123 T+2:00 Attacker steals abc123 T+3:00 User browses (GET) → TTL refreshed T+5:00 User POSTs (add to cart) → abc123 CONSUMED (deleted from Redis) T+5:30 Attacker tries abc123 → 403 (token already consumed!) Single-use defense: first mutation wins, token is gone. Even if attacker is faster: they get one use, user gets auto-regen on next attempt.

7. Implementation Details

Redis Store: RefreshTTL Method

// internal/store/redis.go

// RefreshTTL extends the TTL of an existing token without reading its value.
// Uses Redis EXPIRE command — O(1), does not read the value.
// Returns nil if key doesn't exist (no token to refresh).
func (s *RedisStore) RefreshTTL(ctx context.Context, userID string) error {
    _, err := s.client.Expire(ctx, keyPrefix+userID, s.ttl).Result()
    if err != nil {
        slog.Warn("Failed to refresh CSRF token TTL", "user", userID, "error", err)
        return err
    }
    return nil
}

Handler: writeForbiddenWithNewToken

// internal/handler/authz.go

func (h *Handler) writeForbiddenWithNewToken(
    w http.ResponseWriter, ctx context.Context,
    userID, origin, detail string,
) {
    newToken, err := h.Store.Generate(ctx, userID, origin)
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusForbidden)
    if err == nil && newToken != "" {
        fmt.Fprintf(w,
            `{"type":"about:blank","title":"Forbidden","status":403,"detail":"%s","token":"%s"}`,
            detail, newToken)
        h.Metrics.RequestsTotal.WithLabelValues("authz_mutate", "regenerated").Inc()
    } else {
        fmt.Fprintf(w,
            `{"type":"about:blank","title":"Forbidden","status":403,"detail":"%s"}`,
            detail)
    }
}

9. UI Client Changes

Only one file changed: ui/src/api/client.ts. The 403-retry block now reads the regenerated token from the response body before falling back to fetchCsrfToken().

// ui/src/api/client.ts — 403 retry block

if (resp.status === 403 && isMutating && !_csrfRetried) {
  try {
    const body = await resp.json()
    if (body.token) {
      _csrfToken = body.token  // Use regenerated token from 403 body
    } else {
      await fetchCsrfToken()   // Fallback: separate GET /csrf/token
    }
  } catch {
    await fetchCsrfToken()     // Fallback: response not JSON
  }
  return request<T>(url, options, true)
}
Backward compatible: If the server doesn't include a token field (e.g., older version), the client falls back to the existing fetchCsrfToken() flow.

11. Manual Testing Guide

Step 1: Get a JWT

TOKEN=$(curl -sk -X POST \
  "https://idp.keycloak.net:30000/realms/bookstore/protocol/openid-connect/token" \
  -d "grant_type=password&client_id=ui-client&username=user1&password=CHANGE_ME" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

Step 2: Generate a CSRF Token

CSRF=$(curl -sk -H "Authorization: Bearer $TOKEN" \
  https://api.service.net:30000/csrf/token \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")
echo "CSRF token: $CSRF"

Step 3: Verify Sliding TTL

# Get user sub from JWT
SUB=$(echo $TOKEN | cut -d. -f2 | base64 -d 2>/dev/null | python3 -c "import sys,json; print(json.load(sys.stdin)['sub'])")

# Check initial TTL (~600 seconds)
kubectl exec -n infra deploy/redis -- redis-cli -a CHANGE_ME TTL "csrf:$SUB"
Expected: A value between 590 and 600 (10 minutes in seconds)
# Wait 3 seconds, then make an authenticated GET
sleep 3
curl -sk -H "Authorization: Bearer $TOKEN" https://api.service.net:30000/ecom/books > /dev/null

# Check TTL again — should be back near 600 (sliding refresh!)
sleep 1
kubectl exec -n infra deploy/redis -- redis-cli -a CHANGE_ME TTL "csrf:$SUB"
Expected: TTL back near 600 (not 596). The GET refreshed it.

Step 4: Test Auto-Regeneration

# POST without CSRF token — should get 403 with new token in body
curl -sk -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"bookId":"00000000-0000-0000-0000-000000000001","quantity":1}' \
  https://api.service.net:30000/ecom/cart
Expected: 403 response with "token":"<new-uuid>" in the JSON body
# Extract the regenerated token and use it
REGEN=$(curl -sk -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"bookId":"00000000-0000-0000-0000-000000000001","quantity":1}' \
  https://api.service.net:30000/ecom/cart \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")

# Use regenerated token — should succeed
curl -sk -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -H "X-CSRF-Token: $REGEN" \
  -d '{"bookId":"00000000-0000-0000-0000-000000000001","quantity":1}' \
  https://api.service.net:30000/ecom/cart
Expected: 200 OK (item added to cart using the regenerated token)
