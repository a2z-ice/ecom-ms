Overall risk: LOW-MEDIUM. The primary risk is PgAdmin becoming a pivot point if compromised. Attack surface is limited: localhost-only access (NodePort 31111), non-root container, per-service DB users with minimal privileges, and network access restricted to only the 4 DB pods.

3. Security Controls in Place

Network Layer (Defense in Depth)

Layer 1: ztunnel mTLS (HBONE)

All traffic between PgAdmin and DB pods is wrapped in HTTP/2 CONNECT tunnel with mutual TLS. SPIFFE-based identity verification. PeerAuthentication mode: STRICT. Cipher: TLS 1.3 with ECDHE.

Layer 2: PostgreSQL TLS 1.3

CNPG enforces ssl=on and ssl_min_protocol_version=TLSv1.3. Server certificates auto-rotated by CNPG. Even if ztunnel mTLS is bypassed, PostgreSQL connection is encrypted.

Layer 3: AuthorizationPolicy (Istio L4 RBAC)

ztunnel checks SPIFFE identity of source pod. Only specific namespaces permitted: service's own ns, infra (Debezium), cnpg-system (operator), admin-tools (PgAdmin). All others rejected with TCP RST.

Layer 4: NetworkPolicy (Kubernetes CNI)

Kernel-level packet filtering. Restricts DB ingress to the same set of namespaces. Defense-in-depth: even if Istio is compromised, CNI policy still blocks unauthorized traffic.

Blast Radius Analysis

If PgAdmin is compromised, the attacker CAN: Read and write data in all 4 PostgreSQL databases. Execute SQL as the respective service user on each database.

The attacker CANNOT: Access application pods (ecom-service, inventory-service). Access Kafka, Redis, or other infrastructure. Access the Kubernetes API with elevated permissions. Escape the container (non-root, read-only FS, capabilities dropped). Reach any service outside the 4 DB pods. Access the host network or other nodes.

4. Recommendations

Priority: HIGH

Recommendation 1: Rotate DB Passwords

The current passwords are placeholder values (CHANGE_ME). In any environment beyond local development, replace with strong random passwords using openssl rand -base64 32. Update PgAdmin's server configuration and all Kubernetes Secrets to match.

Priority: MEDIUM

Recommendation 2: Add PgAdmin Authentication Hardening

Enable PgAdmin's built-in login attempt limiting (MAX_LOGIN_ATTEMPTS=5). Restrict access to specific source IPs if deployed beyond localhost. Set PGADMIN_CONFIG_LOGIN_BANNER to warn that this is an administrative tool.

Priority: MEDIUM

Recommendation 3: Enable Audit Logging

PgAdmin logs all executed SQL queries -- ensure these logs are collected and retained. Additionally, enable pgaudit on the PostgreSQL side (CNPG supports this) to log all SQL statements at the database level regardless of client.

Priority: MEDIUM

Recommendation 4: Create Read-Only DB Users

For safer day-to-day browsing, create pgadmin_readonly users with SELECT-only privileges on each database. Configure PgAdmin to use read-only users by default, with full-privilege users available only when write access is explicitly needed.

Priority: LOW

Recommendation 5: Restrict admin-tools Egress

Add an egress NetworkPolicy to the admin-tools namespace. Ensure PgAdmin can only reach the 4 DB services on port 5432 and DNS (kube-dns on port 53). Block all other egress traffic to limit lateral movement if compromised.

5. Verdict

APPROVED for Local Development

The security posture is acceptable. Double encryption (ztunnel mTLS + PostgreSQL TLS 1.3), SCRAM-SHA-256 authentication, namespace-scoped access, dual-layer policy enforcement, and minimal-privilege container provide adequate protection.

Justification

  1. Double encryption -- ztunnel mTLS (HBONE) + PostgreSQL TLS 1.3 protect data in transit
  2. SCRAM-SHA-256 authentication -- the strongest PostgreSQL auth method; no plaintext passwords on the wire
  3. Namespace-scoped access -- PgAdmin is allowed to reach only the 4 DB pods, not the entire cluster
  4. Dual-layer enforcement -- both AuthorizationPolicy (Istio) and NetworkPolicy (CNI) must pass; compromising one layer does not bypass the other
  5. Minimal privilege container -- non-root (UID 5050), read-only filesystem, all capabilities dropped
  6. Localhost-only exposure -- PgAdmin is on NodePort 31111, not routed through the HTTPS gateway
