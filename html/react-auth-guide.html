2. OAuth2 & OpenID Connect Fundamentals

What is OAuth2?

OAuth2 is an authorization framework that lets a user grant a third-party application limited access to their resources without sharing their password.

ActorIn Our AppRole
Resource OwnerThe user (customer/admin)Owns the data
ClientReact SPA (ui-client)Wants access to user's data
Authorization ServerKeycloakIssues tokens after authentication
Resource Serverecom-service, inventory-serviceProtects APIs, validates tokens

What is OpenID Connect (OIDC)?

OIDC is an identity layer built on top of OAuth2. While OAuth2 only handles authorization ("can this app access my data?"), OIDC adds authentication ("who is this user?"). It introduces the ID Token (JWT with user claims), UserInfo Endpoint, and Discovery Document (.well-known/openid-configuration).

Authorization Code Flow with PKCE

This app uses the most secure OAuth2 flow for SPAs: Authorization Code Flow with Proof Key for Code Exchange (PKCE).

Step 1: Generate PKCE Challenge React SPA generates: code_verifier (random string) computes: code_challenge = SHA256(code_verifier) Step 2: Authorization Request GET /realms/bookstore/protocol/openid-connect/auth ?response_type=code &client_id=ui-client &redirect_uri=https://localhost:30000/callback &scope=openid profile email roles &code_challenge=<hash> &code_challenge_method=S256 Keycloak shows login form → user authenticates Redirects to: /callback?code=<authorization_code> Step 3: Token Exchange (in browser via oidc-client-ts) POST /realms/bookstore/protocol/openid-connect/token grant_type=authorization_code &code=<authorization_code> &code_verifier=<original_random_string> &redirect_uri=https://localhost:30000/callback &client_id=ui-client Keycloak verifies: SHA256(code_verifier) == code_challenge Returns: { access_token, id_token, refresh_token }
Why PKCE? Without PKCE, an attacker who intercepts the authorization code could exchange it for tokens. PKCE ensures only the original requester (who knows the code_verifier) can complete the exchange.

JWTs (JSON Web Tokens)

header.payload.signature
  |       |        |
  |       |        +-- Cryptographic signature (verified by backend services)
  |       +-- Claims: { sub, email, roles, exp, iss, ... }
  +-- Algorithm info: { alg: "RS256", typ: "JWT" }

Important: The React app decodes the payload (for UI decisions like showing admin links) but never verifies the signature. Signature verification happens server-side — the backend services use Keycloak's JWKS (JSON Web Key Set) endpoint to verify every token.

3. OIDC Configuration (oidcConfig.ts)

This file configures the UserManager from the oidc-client-ts library — the engine that drives the entire OIDC flow.

// File: ui/src/auth/oidcConfig.ts

import { UserManager, WebStorageStateStore } from 'oidc-client-ts'

// All config from Vite env vars (injected at build time via ConfigMap in k8s)
const AUTHORITY = import.meta.env.VITE_KEYCLOAK_AUTHORITY
const CLIENT_ID = import.meta.env.VITE_KEYCLOAK_CLIENT_ID

// redirect_uri derived from current origin so callback returns to same host
const REDIRECT_URI = `${window.location.origin}/callback`

export const userManager = new UserManager({
  authority: AUTHORITY,         // Keycloak realm URL
  client_id: CLIENT_ID,         // OAuth2 client identifier
  redirect_uri: REDIRECT_URI,   // Where Keycloak sends user after auth
  response_type: 'code',        // Authorization Code Flow (not 'token')
  scope: 'openid profile email roles',

  // PKCE enabled by default when response_type is 'code'

  // Tokens in sessionStorage -- cleared on tab close (never localStorage)
  userStore: new WebStorageStateStore({ store: window.sessionStorage }),

  // Silent token refresh via hidden iframe
  automaticSilentRenew: true,
  silent_redirect_uri: `${window.location.origin}/silent-renew.html`,

  // Load user info from ID token claims (not userinfo endpoint)
  loadUserInfo: false,
})

Line-by-Line Breakdown

  • import.meta.env.VITE_* — Vite's way of exposing environment variables. Only VITE_ prefixed vars reach client code (security measure)
  • window.location.origin — returns https://myecom.net:30000 or https://localhost:30000. Dynamic origin means no hardcoded URLs
  • response_type: 'code' — selects Authorization Code Flow. The alternative 'token' (Implicit Flow) is deprecated
  • scope: 'openid profile email roles' — openid is required for OIDC; roles is a custom scope mapped in Keycloak
  • sessionStorage — data cleared when tab closes; not shared between tabs. Limits XSS exposure
  • automaticSilentRenew: true — refreshes access token before expiry via hidden iframe
  • loadUserInfo: false — user claims read from ID token directly (saves one network request)

4. Authentication Context (AuthContext.tsx)

The AuthContext is the central nervous system of authentication in the React app. It provides auth state and methods to every component via React Context.

Interface Definition

interface AuthContextValue {
  user: User | null              // Authenticated user, or null
  isLoading: boolean             // True during initial auth check
  isAdmin: boolean               // Derived from JWT claims
  login: (returnPath?: string) => void   // Initiates OIDC login
  logout: () => Promise<void>    // Ends session
  getAccessToken: () => string | null   // Raw JWT for API calls
}

Login Method

const login = useCallback((returnPath?: string) => {
  // PKCE requires crypto.subtle (Web Crypto API)
  // Only available in "secure contexts" (HTTPS or localhost)
  if (typeof crypto === 'undefined' || !crypto.subtle) {
    // Redirect to localhost (always secure) for PKCE
    window.location.href =
      `https://localhost:30000/login?return=${encodeURIComponent(window.location.href)}`
    return
  }

  const resolvedPath = returnPath ?? (window.location.pathname + window.location.search)
  userManager.signinRedirect({ state: { returnUrl: resolvedPath } })
}, [])

Logout Method (Back-Channel)

const logout = useCallback(async () => {
  const currentUser = await userManager.getUser()

  if (currentUser?.refresh_token) {
    const authority = userManager.settings.authority
    try {
      await fetch(`${authority}/protocol/openid-connect/logout`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
        body: new URLSearchParams({
          client_id: userManager.settings.client_id,
          refresh_token: currentUser.refresh_token,
        }),
      })
    } catch {
      // Best-effort: clear local state even if Keycloak unreachable
    }
  }

  await userManager.removeUser()  // Clear sessionStorage
  setUser(null)                   // Clear React state
  window.location.href = window.location.origin + '/'
}, [])

Admin Role Detection

const isAdmin = (() => {
  if (!user?.access_token) return false
  try {
    const payload = JSON.parse(atob(user.access_token.split('.')[1]))
    //   split('.')[1] = JWT payload   atob() = base64 decode
    const roles: string[] = payload.roles ?? []
    return roles.includes('admin')
  } catch {
    return false
  }
})()
Important: This is a UI-only check. The backend services independently validate the JWT and enforce authorization. Even if someone modified the client-side isAdmin flag, API calls would still be rejected by the server.

6. The Callback Page (CallbackPage.tsx)

The most complex component in the auth flow. It handles OIDC callback, CSRF token acquisition, guest cart merging, and cross-origin relay.

export default function CallbackPage() {
  const navigate = useNavigate()

  useEffect(() => {
    userManager.signinRedirectCallback()   // Exchange code for tokens
      .then(async (user) => {
        const state = user.state as { returnUrl?: string }
        const returnUrl = state?.returnUrl || '/'

        // 1. Fetch CSRF token before any mutations
        let csrfToken: string | null = null
        const csrfResp = await fetch('/csrf/token', {
          headers: { Authorization: `Bearer ${user.access_token}` },
        })
        if (csrfResp.ok) {
          csrfToken = (await csrfResp.json()).token
          setCsrfToken(csrfToken)
        }

        // 2. Merge guest cart to server cart
        const pending = getGuestCart()
        if (pending.length > 0) {
          await Promise.allSettled(         // allSettled, not all!
            pending.map(item =>
              fetch('/ecom/cart', {
                method: 'POST',
                headers: {
                  'Content-Type': 'application/json',
                  'Authorization': `Bearer ${user.access_token}`,
                  ...(csrfToken ? { 'X-CSRF-Token': csrfToken } : {}),
                },
                body: JSON.stringify({ bookId: item.bookId, quantity: item.quantity }),
              })
            )
          )
          clearGuestCart()
        }

        // 3. Navigate to return URL or cart
        navigate(pending.length > 0 && returnUrl === '/' ? '/cart' : returnUrl)
      })
  }, [navigate])

10. Route Protection

ProtectedRoute — Requires Authentication

export default function ProtectedRoute({ children }: { children: React.ReactNode }) {
  const { user, isLoading, login } = useAuth()
  const location = useLocation()

  useEffect(() => {
    if (!isLoading && !user) {
      login(location.pathname + location.search)  // Save where to return
    }
  }, [isLoading, user, login, location])

  if (isLoading || !user) {
    return <div>Redirecting to login...</div>
  }
  return <>{children}</>
}

AdminRoute — Requires Admin Role

export default function AdminRoute({ children }: { children: React.ReactNode }) {
  const { user, isLoading, isAdmin, login } = useAuth()

  if (isLoading) return <div>Loading...</div>

  if (!user) { login('/admin'); return null }

  if (!isAdmin) return <div>Access Denied</div>

  return <>{children}</>
}

11. CSRF Token Management

What is CSRF?

Cross-Site Request Forgery is an attack where a malicious website tricks the user's browser into making unwanted requests. The CSRF token proves the request originated from our UI, not a malicious site.

React App Istio Gateway CSRF Service (Go) | | | | GET /csrf/token | | | (JWT in header) | | |--------------------->| | | | Generate UUID token | | | Store in Redis (10min)| | { token: "uuid" } | | |<---------------------| | | | | | POST /ecom/cart | | | X-CSRF-Token: uuid | | | Authorization: JWT | | |--------------------->| | | | ext_authz check | | | Validate in Redis | | | (timing-safe compare) | | | | | | Token valid → forward to backend

CSRF Auto-Retry on 403

When a CSRF token expires, the csrf-service includes a new token in the 403 response body. The API client reads it and retries immediately — saving a round trip:

if (resp.status === 403 && isMutating && !_csrfRetried) {
  try {
    const body = await resp.json()
    if (body.token) {
      _csrfToken = body.token           // Use auto-regenerated token
    } else {
      await fetchCsrfToken()            // Fallback: fetch fresh token
    }
  } catch {
    await fetchCsrfToken()              // JSON parse failed
  }
  return request<T>(url, options, true)  // Retry once
}

12. The API Client (client.ts)

// Module-level state (not React state -- persists across renders)
let _getToken: (() => string | null) | null = null
let _csrfToken: string | null = null

const MUTATING_METHODS = new Set(['POST', 'PUT', 'DELETE', 'PATCH'])

async function request<T>(
  url: string,
  options: RequestInit = {},
  _csrfRetried = false,
): Promise<T> {
  const token = _getToken?.()
  const method = (options.method ?? 'GET').toUpperCase()
  const isMutating = MUTATING_METHODS.has(method)

  const headers: HeadersInit = {
    'Content-Type': 'application/json',
    ...(options.headers ?? {}),                              // Keep caller headers
    ...(token ? { Authorization: `Bearer ${token}` } : {}),  // JWT if authenticated
    ...(isMutating && _csrfToken ? { 'X-CSRF-Token': _csrfToken } : {}), // CSRF if mutation
  }

  const resp = await fetch(url, { ...options, headers })

  // ... CSRF retry logic (shown above) ...

  if (!resp.ok) throw new Error(`HTTP ${resp.status}: ${await resp.text()}`)
  if (resp.status === 204) return undefined as T
  return resp.json()
}

export const api = {
  get:    <T>(url: string) => request<T>(url),
  post:   <T>(url: string, body: unknown) => request<T>(url, { method: 'POST', body: JSON.stringify(body) }),
  put:    <T>(url: string, body: unknown) => request<T>(url, { method: 'PUT', body: JSON.stringify(body) }),
  delete: <T>(url: string) => request<T>(url, { method: 'DELETE' }),

20. TypeScript Generics — The <T> Syntax

Generics let you write reusable code that works with any type while preserving type safety.

The Problem Generics Solve

// WITHOUT generics -- repetitive for every return type:
async function getBooks(url: string): Promise<Page<Book>> { return (await fetch(url)).json() }
async function getCart(url: string): Promise<CartItem[]> { return (await fetch(url)).json() }
async function getStock(url: string): Promise<StockResponse> { return (await fetch(url)).json() }

// WITH generics -- one function for ALL types:
async function request<T>(url: string): Promise<T> { return (await fetch(url)).json() }

How Callers Specify the Type

// T = Page<Book>
api.get<Page<Book>>('/ecom/books')       // Returns Promise<Page<Book>>

// T = CartItem[]
api.get<CartItem[]>('/ecom/cart')        // Returns Promise<CartItem[]>

// T = void
api.delete<void>(`/ecom/cart/${id}`)     // Returns Promise<void>

Nested Generics

interface Page<T> {
  content: T[]              // Array of whatever T is
  totalElements: number
  totalPages: number
}

// Page<Book> becomes: { content: Book[], totalElements: number, ... }
// api.get<Page<Book>>(...) means:
//   Outer <T> of request = Page<Book>
//   Inner <T> of Page = Book

Utility Types with Generics

// Omit<Type, Keys> -- removes specified keys
function addToGuestCart(item: Omit<GuestCartItem, 'quantity'>)
// Creates: { bookId: string, title: string, price: number }
// (quantity removed -- callers don't specify it)

21. Promises & Async/Await Deep Dive

Promise States

+---- PENDING ----+ | | ↓ ↓ FULFILLED REJECTED (value) (error)

Three Syntax Styles

// STYLE 1: .then() chains
fetch('/ecom/books')
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error(error))

// STYLE 2: async/await
async function getBooks() {
  try {
    const response = await fetch('/ecom/books')
    const data = await response.json()
  } catch (error) {
    console.error(error)
  }
}

// STYLE 3: Mixed (.then with async callback) -- used in CallbackPage
userManager.signinRedirectCallback()
  .then(async (user) => {
    const csrfResp = await fetch(...)
  })
  .catch(err => console.error(err))

Promise.allSettled vs Promise.all

// Promise.all -- FAILS FAST (one rejection kills everything)
await Promise.all([
  fetch('/cart/item1'),    // success
  fetch('/cart/item2'),    // FAILS → everything fails, item3 result lost!
  fetch('/cart/item3'),    // success (but result lost)
])

// Promise.allSettled -- WAITS FOR ALL (used in CallbackPage for cart merge)
await Promise.allSettled([
  fetch('/cart/item1'),    // { status: 'fulfilled', value: Response }
  fetch('/cart/item2'),    // { status: 'rejected',  reason: Error }
  fetch('/cart/item3'),    // { status: 'fulfilled', value: Response }
])
Why allSettled for cart merge? If one item is out of stock, we still want the others to merge. Promise.all would discard all results if any single item fails.

The Async IIFE Pattern

// React useEffect cannot be async directly
useEffect(async () => { ... }, [])  // INVALID -- returns Promise, not void!

// Solution: Immediately Invoked Function Expression (IIFE)
useEffect(() => {
  ;(async () => {                    // Define async function
    const u = await userManager.getUser()
    setUser(u)
  })()                               // Invoke it immediately
}, [])

Optional Chaining on Function Calls

const token = _getToken?.()
// If _getToken is null/undefined → returns undefined (no error)
// If _getToken is a function     → calls it, returns result

22. Spread & Destructuring — The ... Syntax

1. Object Spread (Most Common)

const base = { a: 1, b: 2 }
const extended = { ...base, c: 3 }       // { a: 1, b: 2, c: 3 }
const override = { ...base, b: 99 }      // { a: 1, b: 99 } -- later overrides
const same = { ...base, ...{} }          // { a: 1, b: 2 } -- empty spread = noop

2. The Conditional Spread Pattern

// The key pattern used throughout this codebase:
...(condition ? { key: value } : {})

// Example from client.ts:
...(token ? { Authorization: `Bearer ${token}` } : {})
// token = "abc123" → spreads { Authorization: "Bearer abc123" } → added
// token = null     → spreads {} → nothing added

3. Complete Header Build Example

const headers = {
  'Content-Type': 'application/json',          // Always present
  ...(options.headers ?? {}),                    // Preserve caller headers
  ...(token ? { Authorization: `Bearer ${token}` } : {}),    // JWT if auth
  ...(isMutating && _csrfToken ? { 'X-CSRF-Token': _csrfToken } : {}), // CSRF
}

// Step-by-step with token="jwt..." and csrfToken="uuid..." and POST:
// 1. { 'Content-Type': 'application/json' }
// 2. + caller headers (if any)
// 3. + { Authorization: 'Bearer jwt...' }
// 4. + { 'X-CSRF-Token': 'uuid...' }

23. Advanced React Patterns

Pattern 1: Context + Custom Hook (Provider Pattern)

The Problem: Prop Drilling

// WITHOUT Context -- passing props through every layer:
<App user={user}>
  <Layout user={user}>                  // Just passing through!
    <NavBar user={user} login={login}>
      ...
    <CartPage user={user}>              // Just passing through!
      <CartItem user={user} />          // Finally needs it

The Solution

// 1. Create Context
const AuthContext = createContext<AuthContextValue | null>(null)

// 2. Provider manages state
function AuthProvider({ children }) { ... }

// 3. Custom hook consumes it
function useAuth() {
  const ctx = useContext(AuthContext)
  if (!ctx) throw new Error('useAuth must be used within AuthProvider')
  return ctx
}

// 4. Use anywhere -- no prop drilling!
function NavBar() {
  const { user, login } = useAuth()   // Direct access
}

Pattern 2: Module-Level State (Singleton)

// React state: tied to component lifecycle, triggers re-renders
const [token, setToken] = useState(null)  // Destroyed on unmount

// Module-level state: persists for page lifetime, no re-renders
let _token = null  // Created once, shared across all imports

The API client uses module-level state because it needs to: persist across component lifecycles, be shared as a singleton, and NOT trigger re-renders when it changes.

Pattern 3: useCallback for Stable References

// WITHOUT useCallback: new function on every render
const login = (path) => { ... }  // New reference each time!

// WITH useCallback: same reference across renders
const login = useCallback((path) => { ... }, [])
//                                              ^^
//  [] = never recreate (no dependencies)
//  [user] = recreate only when user changes

Prevents unnecessary re-renders of child components that receive these functions as context values.

Pattern 4: useEffect Cleanup

useEffect(() => {
  const handler = () => fetchCartCount()
  window.addEventListener('cartUpdated', handler)   // Subscribe

  return () => {                                      // Cleanup function
    window.removeEventListener('cartUpdated', handler) // Unsubscribe
  }
}, [user])  // Re-runs when user changes

Cleanup runs: (a) before the effect re-runs (when deps change), and (b) when the component unmounts. Without cleanup, each render adds another listener — a memory leak.

Pattern 5: Lazy Loading with React.lazy

// Vite creates a separate chunk for each lazy import
const AdminDashboard = React.lazy(() => import('./pages/admin/AdminDashboard'))

// Suspense shows fallback while chunk downloads
<Suspense fallback={<div>Loading...</div>}>
  <AdminDashboard />
</Suspense>

Admin pages are code-split — regular users never download admin JavaScript. The chunk is fetched on first navigation to /admin.

Pattern 6: Custom DOM Events

// CartPage dispatches:
window.dispatchEvent(new Event('cartUpdated'))

// NavBar listens:
window.addEventListener('cartUpdated', fetchCount)

Minimal coupling between components. NavBar doesn't need to be a child of CartPage — they communicate through the DOM event system.

Pattern 7: Conditional Rendering

// 1. Logical AND -- show or hide
{isAdmin && <Link to="/admin">Admin</Link>}

// 2. Ternary -- show one OR the other
{user ? <button onClick={logout}>Logout</button>
      : <button onClick={login}>Login</button>}

// 3. Early return -- different entire renders
if (isLoading) return <div>Loading...</div>
if (!user) { login(); return null }
if (!isAdmin) return <div>Access Denied</div>
return <>{children}</>

Pattern 8: Nullish Coalescing & Optional Chaining

// Optional chaining (?.) -- safe property/call access
user?.access_token              // undefined if user is null (no error)
_getToken?.()                   // undefined if null, calls if function

// Nullish coalescing (??) -- default for null/undefined only
options.method ?? 'GET'          // 'GET' only if method is null/undefined

// vs logical OR (||) -- default for any falsy value
'' ?? 'default'   // '' (empty string is NOT nullish)
'' || 'default'   // 'default' (empty string IS falsy)
0 ?? 'default'    // 0 (zero is NOT nullish)
0 || 'default'    // 'default' (zero IS falsy)

Security Checklist

  • Tokens stored in memory / sessionStorage (never localStorage)
  • PKCE enabled on Authorization Code Flow
  • CSRF token required on all mutating requests
  • JWT validated server-side (never trust client claims)
  • Route guards prevent unauthorized UI access
  • Admin role checked in both UI and backend
  • Logout terminates both local and server sessions
  • Silent renewal keeps tokens fresh without user interaction
  • Cross-origin relay uses URL hash (not sent to servers)
  • Hash cleared immediately from browser history
  • Error boundaries prevent auth failures from crashing the app
  • Guest cart uses localStorage only for non-sensitive data
