Claude Code Skills Reference

15 operator shortcuts (.claude/skills/*/SKILL.md) that turn common platform operations into one-line commands. This document explains why each skill exists, how to invoke it, what it does, and the concrete benefits it provides.

15 skills production-ready 1011 lines total all idempotent

What Are Skills?

Claude Code skills are reusable prompt-templates stored in .claude/skills/<name>/SKILL.md. Each skill has YAML frontmatter declaring its name, description, allowed tools, and argument hints. When invoked as /skill-name [args], Claude Code expands the SKILL.md file into the conversation as instructions.

Skills differ from slash-commands in that they are in-repo, version-controlled, and expand into context rather than executing directly. They're ideal for multi-step operations that combine kubectl, curl, docker, and psql — exactly the kind of glue a 15-service platform needs.

Why We Built These Skills

This project is a 15-service microservices platform with Istio Ambient mesh, Kafka, Debezium, Flink, Keycloak, CNPG PostgreSQL, and 5 observability stacks. Common operations — "check stack health", "run E2E tests", "fix Debezium after failover" — touch 5-10 components each and are error-prone if typed by hand.

Three forces drove skill creation:

  1. Repetition: The same diagnostic sequences were run many times. Every manual repetition risks a typo.
  2. Shared vocabulary: Team members and future-Claude-sessions need a consistent way to say "recover after Docker Desktop restart" without re-deriving the 6-step procedure.
  3. Onboarding compression: A new operator can read .claude/skills/ and see every high-value operation the platform supports, in ~1000 lines total.

Lifecycle & Bootstrap Skills

lifecycle bootstrap

argument-hint: [--fresh] [--yes]

Why it exists: The full stack bootstrap touches a kind cluster, Istio, KGateway, cert-manager, 4 CNPG clusters, Kafka, Redis, Debezium, Flink, Keycloak, Superset, Grafana, Prometheus, OTel, and 5+ custom services. Typing the sequence by hand is unreliable.

/bootstrap [--fresh] [--yes]

What it does: Wraps scripts/up.sh which is smart — it auto-detects cluster state (no cluster → bootstrap, degraded → recover, healthy → verify) and runs the correct path. --fresh forces a full teardown and rebuild.

Benefit: One command replaces ~20 manual steps. Same entrypoint for "fresh install" and "recovery" scenarios. Idempotent — safe to re-run.
Tools: BashRuns: scripts/up.shTime: 8-12 min fresh

lifecycle teardown

argument-hint: [--data] [--images] [--all] [--yes]

Why it exists: Fully removing a kind cluster involves stopping Docker containers, deleting PVCs, optionally wiping ./data/, and optionally removing all bookstore/* Docker images. The flags make each choice explicit.

/teardown [--data] [--images] [--all] [--yes]

What it does: Delegates to scripts/down.sh. By default only deletes the cluster (fastest re-bootstrap). --data wipes host-mounted PVC data. --images removes images to reclaim ~5GB. --all does both.

Benefit: Prevents accidental data loss (no default --data). Supports CI automation via --yes. Each mode is visible at invocation time.
Tools: BashRuns: scripts/down.sh

lifecycle deploy

argument-hint: <service-name>

Why it exists: After changing code in one service (e.g. ecom-service), the cycle is: build JAR/binary → docker build → kind load docker-image → kubectl rollout restart → wait. Each service has its own build tool (Maven, Poetry, Go, Vite). This skill knows each one.

/deploy ecom # Spring Boot /deploy inventory # FastAPI /deploy ui # React/Vite (needs VITE_ build args) /deploy csrf # Go /deploy flink # custom Flink image

What it does: Reads the service name, picks the right build tool, loads the image into kind, and does a rolling restart. For ui, it passes the required VITE_ build args.

Benefit: No need to remember 5 different build/push/restart sequences. UI's VITE_ gotcha is baked in (build-time vars, not runtime). Works with kind's lack of a container registry.
Tools: Bash, Read, Grep

lifecycle recover

Why it exists: After Docker Desktop sleeps/restarts, the Istio Ambient HBONE plumbing breaks at ztunnel level. Pods must restart in dependency order (DBs → apps). Debezium re-snapshots from Kafka offset topics. This is non-obvious and was painful to derive the first time.

/recover

What it does: Runs the smart up.sh which detects degraded state automatically, or falls back to scripts/restart-after-docker.sh. Restarts ztunnel first, then all pods in dependency order, then verifies Debezium health.

Benefit: Turns a multi-hour debugging session ("why are all my services returning connection reset?") into a 2-minute recovery. Encodes the "restart ztunnel FIRST" rule.
Tools: BashRuns: scripts/up.sh or restart-after-docker.sh

Observability & Testing Skills

observability status

argument-hint: [namespace]

Why it exists: The platform spans 10+ namespaces. A single kubectl get pods -A dump is too noisy. This skill produces a curated view of what matters.

/status /status ecom # focused on one namespace

What it does: Shows pod readiness by namespace, external endpoint health (HTTPS 30000, Keycloak 32400, Debezium 32300/32301, Kiali 32100, Grafana 32500, Superset 32000, Flink 32200, PgAdmin 31111, Cert Dashboard 32600), Flink job count/status, Debezium /q/health status, and cluster name.

Benefit: 30-second overview of the entire platform. Instantly spots pods in CrashLoopBackOff, stopped Flink jobs, or down endpoints.
Tools: Bash

observability smoke

Why it exists: "Is everything working?" needs a bulletproof yes/no answer. 31 individual checks cover pods, HTTP, Kafka, Debezium, admin endpoints.

/smoke

What it does: Runs scripts/smoke-test.sh which verifies pods in all app namespaces, tests HTTPS endpoints, checks Kafka connectivity, polls Debezium health, exercises admin routes. Exits non-zero on failure (usable in CI).

Benefit: Single green/red signal. Part of the standard bootstrap verification flow. Catches silent degradation (e.g. a pod that's Running but failing liveness).
Tools: Bash31 checks

observability health-check

Why it exists: Deeper than smoke — includes observability stack (Loki, Tempo, Grafana dashboards), AlertManager silences, and cert-manager certificate expiry.

/health-check

What it does: Runs all the checks from smoke plus queries Loki for recent error logs, checks Grafana dashboard availability, verifies AlertManager has no firing alerts, confirms cert-manager certificates are not expiring within 7 days.

Benefit: Catches issues that smoke misses. Proactive — flags expiring certs before they break the platform.
Tools: Bash

observability e2e

argument-hint: [test-file] [--headed] [--ui]

Why it exists: Playwright E2E tests are the acceptance gate for every feature. The invocations vary (full suite vs single file vs UI mode) and the working directory matters.

/e2e # all tests /e2e cart.spec.ts # one file /e2e checkout.spec.ts --headed # watch it run /e2e --ui # Playwright UI mode

What it does: Runs Playwright from the e2e/ directory. Supports grep-by-name, headed browser, UI mode. Prints a summary of pass/fail/skip.

Benefit: Always runs with the correct working directory. Exposes headed/UI modes for debugging without needing to remember flags.
Tools: Bash533 tests total

observability logs

argument-hint: <service> [--tail N] [--follow]

Why it exists: Each service is in a different namespace. Typing kubectl logs -n <ns> deploy/<name> -f for the N-th time invites mistakes.

/logs ecom --follow /logs kafka --tail 100 /logs debezium-ecom /logs keycloak

What it does: Maps friendly service names to {namespace, deployment} pairs. Supports --tail N and --follow.

Benefit: No need to memorize which namespace each service is in. Consistent flag syntax across all services.
Tools: BashServices: ecom, inventory, ui, kafka, debezium-*, flink, keycloak, csrf

observability cdc-verify

Why it exists: The CDC pipeline spans 5 components (ecom-db → Debezium → Kafka → Flink SQL → analytics-db). Verifying end-to-end requires seeding a row, polling the analytics DB, and checking each hop.

/cdc-verify

What it does: Runs scripts/verify-cdc.sh. Inserts a test book into ecom-db, then polls analytics-db for up to 30 seconds waiting for the row to appear in dim_books. Also checks Debezium health, Flink job statuses, and Kafka topic presence.

Benefit: Verifies the hardest part of the stack. Catches silent CDC breakages (stuck Flink job, Debezium offset drift) that pods-Running would not.
Tools: BashTimeout: 30s

Debugging & Fix Skills

debug fix-stability

Why it exists: After up.sh finishes, 7 components can be silently broken (kafka-exporter missing, Kiali can't reach Prometheus, cert-dashboard not deployed, cart + not flushing, gateway xDS stale, workload certs expired). These are documented in up.sh Stability Fixes.

/fix-stability

What it does: Runs 7 targeted diagnostic checks matching each known failure mode. For each broken component, provides the exact fix command (idempotent, safe to re-run). Includes an all-in-one diagnostic summary.

Benefit: Compresses a ~4-hour debugging session (happened to me on 2026-04-05) into a 30-second diagnostic. Encodes the tricky bits: Istio Ambient non-mesh AuthZ denials, Hibernate 7 saveAndFlush(), gateway xDS staleness.
Tools: Bash, Read, Edit7 checksSee: up-sh-stability-fixes.md

debug fix-debezium

Why it exists: After a CNPG HA failover, Debezium Server enters CrashLoopBackOff because the new primary doesn't have the logical replication slot, and the Kafka-stored offset references a WAL position that no longer exists.

/fix-debezium

What it does: Identifies which Debezium instance is failing (ecom or inventory), connects to the new CNPG primary, recreates the missing debezium_ecom_slot / debezium_inventory_slot, clears the stale offset topic in Kafka, and restarts Debezium. Instance re-snapshots from scratch.

Benefit: Turns a deep-PostgreSQL-logical-replication debug session into a 3-minute recovery. Captures the CNPG-specific failover pattern that isn't documented upstream.
Tools: BashTargets: CrashLoopBackOff after failover

Development & Data Skills

dev db

argument-hint: <database> <query>

Why it exists: Connecting to each of the 4 CNPG databases (ecom, inventory, analytics, keycloak) requires knowing the pod name, credentials, and the CNPG "connect via TCP not unix socket" gotcha (psql -h localhost).

/db ecom "SELECT count(*) FROM cart_items" /db analytics "SELECT * FROM dim_books LIMIT 5" /db keycloak "\\dt"

What it does: Looks up the CNPG primary pod, extracts credentials from the K8s secret, runs the query via kubectl exec … psql. Abstracts away the peer-auth / socket quirks.

Benefit: One consistent syntax for all 4 databases. No need to remember pod names or the -h localhost CNPG gotcha.
Tools: BashDatabases: ecom, inventory, analytics, keycloak

dev operator

argument-hint: <action> [--no-cache]

Why it exists: The cert-dashboard operator (Go + operator-sdk + OLM) has a build/test/deploy/verify lifecycle that spans 3 images, 1 CRD, and 2 deployments. Actions like "rebuild operator image, reload to kind, restart CR" are multi-step.

/operator build /operator deploy /operator status /operator logs /operator rebuild --no-cache /operator uninstall

What it does: Dispatches on action: builds Go binary + operator image + dashboard image, runs go tests, loads images to kind, applies CRD, installs OLM if needed, creates CR, verifies dashboard at NodePort 32600. rebuild does the whole cycle.

Benefit: The longest skill (183 lines) because it's the most complex workflow. Hides OLM's quirks and the double-image (operator + dashboard) nature of the setup.
Tools: Bash, Read, Grep, GlobActions: build, test, deploy, rebuild, status, logs, verify, uninstall

dev diagrams

argument-hint: [architecture|data-flow|all]

Why it exists: The architecture diagrams are authored as SVGs and need to be exported to PNG/GIF for LinkedIn sharing, docs, and the HTML index page. The export pipeline uses rsvg-convert + ffmpeg.

/diagrams all /diagrams architecture /diagrams data-flow

What it does: Converts SVG → PNG at fixed resolution, assembles multi-frame animated GIFs via ffmpeg, drops outputs into docs/images/.

Benefit: Reproducible diagram rendering. New diagrams just need an SVG dropped in the right place — the skill handles format conversion.
Tools: BashOutputs: PNG, GIF

Strengths of This Skill System

1. Encodes Tribal Knowledge

Each skill captures an "aha" moment from debugging. recover encodes "ztunnel must restart FIRST". fix-stability encodes "Istio Ambient AuthZ denies non-mesh callers". fix-debezium encodes "CNPG failover breaks logical replication slots". Without skills, this knowledge lives in one person's head.

2. Idempotent by Design

Every skill can be run multiple times safely. bootstrap auto-detects what's already deployed. deploy does a rolling restart (no downtime). fix-* skills check before fixing. This makes experimentation cheap.

3. Composable

Skills call other skills naturally. bootstrap ends with smoke. recover ends with a health verification. deploy runs status afterwards. Workflows compose from small pieces.

4. Documentation That Stays Current

Skills are executed, so they can't drift from reality like prose docs. If the underlying script or binary path changes, the skill breaks immediately and gets fixed. CLAUDE.md references skills for current usage.

5. Safe Tool Boundaries

Every skill declares allowed-tools in its frontmatter. Most only allow Bash. fix-stability allows Bash, Read, Edit. operator adds Grep, Glob for code search. This keeps skills from accidentally gaining unsafe capabilities.

When to Add a New Skill

Add a skill when any of these apply:

  • You've run the same multi-step command sequence 3+ times.
  • You just debugged something non-obvious and want the fix pattern reusable.
  • A workflow spans multiple tools (kubectl + curl + psql + docker) and has gotchas.
  • Onboarding a new operator — they will ask "how do I check X?" and a skill is the answer.

Do NOT add a skill for a one-off command, or for something that's better as a shell alias or script. Skills are for operations that benefit from Claude's context expansion (multi-step diagnosis with interpretation).

How Skills Integrate With CLAUDE.md

CLAUDE.md is the platform's architectural reference — it documents what exists. Skills are the platform's operational layer — they document how to work with what exists. Together:

  • CLAUDE.md says "Debezium Server runs in infra namespace, health at /q/health, port 32300/32301".
  • .claude/skills/fix-debezium/SKILL.md says "when Debezium is CrashLoopBackOff, do these 6 steps".
  • .claude/skills/status/SKILL.md says "show me Debezium health as part of the cluster status".

The MANDATORY RULE in the project's memory: "After any change that affects workflows, scripts, services, ports, or commands, update the relevant skill(s) in .claude/skills/." This keeps skills in lockstep with the platform.
