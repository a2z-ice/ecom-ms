What the Schema Registry Provides

CapabilityDescription
Schema CatalogCentral repository of all event schemas — single source of truth for CDC and application events
VersioningEvery schema change creates a new version; full history retained for auditing
CompatibilityBACKWARD compatibility enforced — new schemas can always read data written by old schemas
ValidationREST API allows producers/consumers to validate messages against registered schemas
DocumentationEach schema includes title, description, field types, constraints, and required fields

Deployment: Confluent Schema Registry 7.8.0 runs as a ClusterIP service in the infra namespace at schema-registry.infra.svc.cluster.local:8081. It uses Kafka as its backend store (topic: _schemas). NetworkPolicies restrict access to infra-namespace pods only.

Step-by-Step Manual Testing Guide

Prerequisites: Cluster running (bash scripts/up.sh) and Schema Registry pod in Running state (kubectl get pod -n infra -l app=schema-registry).

1
Verify Schema Registry is Healthy
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects

Expected: JSON array of subject names (e.g. ["ecom-connector.public.orders-value", ...])

2
Register Schemas (if not already done)
bash infra/schema-registry/register-schemas.sh

Expected: "6 registered, 0 failed"

3
List All Registered Subjects
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects | python3 -m json.tool

Expected: 6 subjects listed alphabetically

4
View Schema Versions
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects/ecom-connector.public.orders-value/versions

Expected: [1]

5
Retrieve Latest Schema Content
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects/ecom-connector.public.orders-value/versions/latest \
  | python3 -c "import sys,json; schema=json.loads(json.load(sys.stdin)['schema']); print(json.dumps(schema, indent=2))"

Expected: Full JSON Schema with properties (before, after, op), $defs (OrderRow)

6
Check Global Compatibility Mode
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/config

Expected: {"compatibilityLevel":"BACKWARD"}

7
Test Backward Compatibility (Schema Evolution)

Adding an optional field should be backward-compatible:

kubectl exec -i -n infra deploy/schema-registry -- \
  curl -sf -X POST \
  http://localhost:8081/compatibility/subjects/order.created-value/versions/latest \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{"schemaType":"JSON","schema":"{\"type\":\"object\",\"properties\":{\"orderId\":{\"type\":\"string\"},\"userId\":{\"type\":\"string\"},\"items\":{\"type\":\"array\"},\"total\":{\"type\":\"number\"},\"timestamp\":{\"type\":\"string\"},\"newField\":{\"type\":\"string\"}},\"required\":[\"orderId\",\"userId\",\"items\",\"total\",\"timestamp\"]}"}'

Expected: {"is_compatible":true}

8
Retrieve Schema by Global ID
# Get schema ID
SCHEMA_ID=$(kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects/ecom-connector.public.orders-value/versions/latest \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")

# Fetch by ID
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/schemas/ids/$SCHEMA_ID | python3 -m json.tool

Expected: Schema JSON with schemaType and schema fields

9
Verify Idempotent Re-registration
# Run again
bash infra/schema-registry/register-schemas.sh

# Check versions — should still be [1]
kubectl exec -n infra deploy/schema-registry -- \
  curl -sf http://localhost:8081/subjects/ecom-connector.public.orders-value/versions

Expected: [1] (no new version created)

10
Run Automated E2E Tests
cd e2e && npx playwright test schema-registry.spec.ts --reporter=list

Expected: 24 tests passed
