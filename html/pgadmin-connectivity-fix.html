PgAdmin Database Connectivity Fix

How AuthorizationPolicy + NetworkPolicy blocked PgAdmin, and the systematic debugging process

Root Cause Found 2 Policy Layers 4 Databases Fixed SSL + SCRAM Auth

1. The Problem

PgAdmin at http://localhost:31111 was fully functional at the web UI level. Login worked, the "BookStore" server group displayed all 4 pre-configured servers, and the interface rendered correctly.

But clicking any server to establish a database connection failed immediately:

connection to server at "ecom-db-rw.ecom.svc.cluster.local" (10.96.x.x), port 5432 failed:
  server closed the connection unexpectedly
  This probably means the server terminated abnormally before or while processing the request.

At the wire protocol level:

[Errno 104] Connection reset by peer

This happened for all 4 databases: ecom-db, inventory-db, analytics-db, and keycloak-db.

2. Initial Misdiagnosis

Several theories were investigated and ruled out before reaching the root cause.

WRONG

Theory 1: Istio Ambient ztunnel Breaks PostgreSQL L4

Hypothesis: ztunnel's HBONE tunneling corrupts or interferes with PostgreSQL's binary wire protocol. Disproved because Debezium Server (in infra namespace) connects cross-namespace to ecom-db using the same PostgreSQL wire protocol and it works perfectly through ztunnel.

WRONG

Theory 2: portLevelMtls PERMISSIVE Needed

Hypothesis: the PostgreSQL port needs portLevelMtls: PERMISSIVE because PgAdmin sends plaintext. Disproved because ztunnel handles mTLS transparently for ambient-to-ambient connections. Both source and destination are in the mesh; the application sees plaintext in both directions.

WRONG

Theory 3: PgAdmin Must Be Outside the Mesh

Hypothesis: remove ambient mesh label from admin-tools so PgAdmin bypasses ztunnel. This made things WORSE. Non-HBONE traffic is rejected by the DB-side ztunnel when the namespace has mtls.mode: STRICT. Being inside the mesh is required, not optional.

PARTIALLY WRONG

Theory 4: SSLRequest Causes the Reset

Hypothesis: PgAdmin's initial SSLRequest packet confuses ztunnel. While PgAdmin does send SSLRequest, even direct StartupMessages (bypassing SSLRequest) resulted in the same reset. The RST came before any application-layer protocol exchange.

WRONG

Theory 5: NodePort Bypass Needed

Hypothesis: expose PostgreSQL directly via NodePort to bypass ztunnel. NodePorts do not bypass ztunnel when the destination pod is in the ambient mesh. And this would be a massive security hole.

3. The Breakthrough

Key observation: Debezium Server (in the infra namespace) connects cross-namespace to ecom-db and it WORKS. PgAdmin (in the admin-tools namespace) connects cross-namespace to ecom-db and it FAILS. If ztunnel itself was the problem, Debezium would also fail. So ztunnel is NOT the issue.

This shifted the investigation from "is ztunnel breaking the protocol?" to "what is different about admin-tools vs infra?"

The answer: the AuthorizationPolicy and NetworkPolicy on the database pods explicitly allow infra but do NOT mention admin-tools.

4. Root Cause

Two security policies blocked PgAdmin's traffic. Both must allow a connection for it to succeed.

AuthorizationPolicy (Before)

  • ecom namespace -- allowed
  • infra namespace -- allowed (Debezium)
  • cnpg-system namespace -- allowed (operator)
  • admin-tools -- NOT listed, BLOCKED

AuthorizationPolicy (After)

  • ecom namespace -- allowed
  • infra namespace -- allowed (Debezium)
  • cnpg-system namespace -- allowed (operator)
  • admin-tools -- added, ALLOWED

AuthorizationPolicy (ecom namespace example)

apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: ecom-db-policy
  namespace: ecom
spec:
  selector:
    matchLabels:
      cnpg.io/cluster: ecom-db
  rules:
  - from:
    - source:
        namespaces: [ecom]          # ecom-service
  - from:
    - source:
        namespaces: [infra]         # Debezium
  - from:
    - source:
        namespaces: [cnpg-system]   # CNPG operator
  # admin-tools NOT listed -- PgAdmin BLOCKED

NetworkPolicy (ecom namespace example)

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: ecom-db-policy
  namespace: ecom
spec:
  podSelector:
    matchLabels:
      cnpg.io/cluster: ecom-db
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: ecom-service
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: infra
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: cnpg-system
  # admin-tools NOT listed -- PgAdmin BLOCKED

Dual-layer block: The same pattern existed for all 4 database namespaces: ecom, inventory, analytics, and identity. Both AuthorizationPolicy and NetworkPolicy had to be updated for each.

5. The Fix

Step 1: Ensure admin-tools Is in the Ambient Mesh

kubectl label namespace admin-tools istio.io/dataplane-mode=ambient --overwrite

Step 2: Update AuthorizationPolicy (all 4 namespaces)

# ecom namespace
kubectl -n ecom patch authorizationpolicy ecom-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/rules/-","value":{"from":[{"source":{"namespaces":["admin-tools"]}}]}}]'

# inventory namespace
kubectl -n inventory patch authorizationpolicy inventory-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/rules/-","value":{"from":[{"source":{"namespaces":["admin-tools"]}}]}}]'

# analytics namespace
kubectl -n analytics patch authorizationpolicy analytics-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/rules/-","value":{"from":[{"source":{"namespaces":["admin-tools"]}}]}}]'

# identity namespace
kubectl -n identity patch authorizationpolicy keycloak-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/rules/-","value":{"from":[{"source":{"namespaces":["admin-tools"]}}]}}]'

Step 3: Update NetworkPolicy (all 4 namespaces)

# ecom namespace
kubectl -n ecom patch networkpolicy ecom-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/ingress/-","value":{"from":[{"namespaceSelector":{"matchLabels":{"kubernetes.io/metadata.name":"admin-tools"}}}]}}]'

# inventory namespace
kubectl -n inventory patch networkpolicy inventory-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/ingress/-","value":{"from":[{"namespaceSelector":{"matchLabels":{"kubernetes.io/metadata.name":"admin-tools"}}}]}}]'

# analytics namespace
kubectl -n analytics patch networkpolicy analytics-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/ingress/-","value":{"from":[{"namespaceSelector":{"matchLabels":{"kubernetes.io/metadata.name":"admin-tools"}}}]}}]'

# identity namespace
kubectl -n identity patch networkpolicy keycloak-db-policy --type=json \
  -p '[{"op":"add","path":"/spec/ingress/-","value":{"from":[{"namespaceSelector":{"matchLabels":{"kubernetes.io/metadata.name":"admin-tools"}}}]}}]'

  v

9. Lessons Learned

Lesson 1: Dual-layer defense. AuthorizationPolicy (Istio L4 RBAC) and NetworkPolicy (CNI packet filtering) are independent layers. Both must allow the traffic. Missing either one blocks the connection, even if the other allows it.

Lesson 2: "Connection reset by peer" = AuthorizationPolicy denial. When ztunnel's RBAC rejects a connection, the symptom is a TCP RST. This is NOT a protocol mismatch, NOT an mTLS issue, and NOT a PostgreSQL problem. Always check AuthorizationPolicy first when you see this error in an ambient mesh.

Lesson 3: Compare with a known-working service. Debezium's successful cross-namespace PostgreSQL connection proved that ztunnel and the wire protocol worked correctly. The only variable was the source namespace. This eliminated 90% of possible causes in one step.

Lesson 4: New namespaces need explicit access. When deploying a new service in a new namespace, always check AuthorizationPolicy and NetworkPolicy on the target services. Zero-trust networking means no implicit access -- every cross-namespace connection must be explicitly authorized at both layers.

Lesson 5: The error message is misleading. "Connection to server closed unexpectedly" suggests PostgreSQL crashed. In reality, PostgreSQL never saw the connection. The RST came from ztunnel sitting between PgAdmin and PostgreSQL. Always consider the full network path, not just the endpoints.
