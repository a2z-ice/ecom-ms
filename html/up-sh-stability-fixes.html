up.sh Stability Fixes

Post-mortem: 7 components broken after bash scripts/up.sh, what caused each, how to identify them, and the permanent fixes. Dated 2026-04-05.

7 issues found All fixed 531 E2E tests passing

Context

After running bash scripts/up.sh, multiple components appeared up but were silently broken: Kiali couldn't see Prometheus, kafka-exporter was missing, cert-dashboard operator wasn't deployed, the cart + button silently failed, and the gateway had stale xDS config. This document explains each root cause, identifies how to detect it, and lists the fix.

Issue #1 & #2 — kafka-exporter Not DeployedFIXED

Symptoms

Prometheus targets API shows one unhealthy target:

dial tcp: lookup kafka-exporter.infra.svc.cluster.local on 10.96.0.10:53: no such host

E2E tests fail in cdc-hardening.spec.ts:175-211:

  • kafka-exporter pod is Running
  • kafka-exporter ClusterIP service exists on port 9308
  • kafka-exporter exposes Prometheus metrics
  • kafka-exporter reports consumer group lag metrics
  • kafka-exporter reports topic metrics

Diagnosis Commands

kubectl get deploy kafka-exporter -n infra
# NotFound → confirms issue

kubectl get all -n infra | grep kafka-exporter
# Empty

Root Cause

The manifest infra/kafka/kafka-exporter.yaml exists and is correctly deployed by scripts/infra-up.sh (line 67-70). But scripts/up.sh doesn't call infra-up.sh — it inlines all the deploy steps directly for finer-grained parallelism. The kafka-exporter step was missed when the deploys were inlined.

Permanent Fix

BEFORE
# up.sh bootstrap_fresh()
# Step 5: Kafka deployed
# Step 5b: Schema Registry
# → kafka-exporter MISSING
AFTER
# up.sh bootstrap_fresh()
# Step 5: Kafka deployed
# Step 5b: Kafka Exporter (NEW)
kubectl apply -f infra/kafka/kafka-exporter.yaml
wait_deploy kafka-exporter infra
# Step 5c: Schema Registry

Also added kafka-exporter restart to the recovery() function.

Issue #3 — Kiali Can't Query PrometheusFIXED

Symptoms

Kiali dashboard shows no traffic graph, metrics charts are empty.

curl http://localhost:32100/kiali/api/status
# Only shows "Kubernetes-Kubernetes" in externalServices, NO Prometheus entry

Diagnosis — Two Nested Problems

Problem 3a: Kiali config had no prometheus.url

The Kiali helm chart defaults to http://prometheus:9090 (same namespace), which doesn't exist. You need to either create a prometheus ExternalName service in istio-system OR explicitly set external_services.prometheus.url in the Kiali config.

Problem 3b: Ambient mesh AuthorizationPolicy blocked non-mesh connection

kubectl logs -n istio-system deploy/kiali | grep prometheus
# "read: connection reset by peer"

kubectl logs -n istio-system -l app=ztunnel | grep "policy rejection"
# "connection closed due to policy rejection: allow policies exist, but none allowed"

The subtle root cause: Istio Ambient mesh uses ztunnel (per-node DaemonSet) to capture traffic for mesh-enrolled pods. The observability namespace has istio.io/dataplane-mode=ambient, so Prometheus IS in the mesh with STRICT mTLS. The istio-system namespace does NOT have that label, so Kiali is NOT in the mesh.

When Kiali (non-mesh) connects to Prometheus (mesh), the ztunnel on the destination node intercepts the INBOUND plaintext connection. The prometheus-policy AuthorizationPolicy rule namespaces: [istio-system] requires SPIFFE identity verification to match the namespace. But non-mesh pods have NO SPIFFE identity (they use plain TCP). The rule silently fails to match, and the implicit deny kicks in.

This is a classic "allow policies exist, but none allowed" error — the policy exists but didn't match. The connection is closed on the destination ztunnel.

Permanent Fix

BEFORE (broken)
# observability-policy.yaml
spec:
  selector:
    matchLabels:
      app: prometheus
  rules:
    - from:
        - source:
            namespaces: ["observability"]
    - from:
        - source:
            namespaces: ["istio-system"]
# ❌ Fails for non-mesh sources
# (can't verify their namespace)
AFTER (works)
# observability-policy.yaml
spec:
  selector:
    matchLabels:
      app: prometheus
  rules:
    - {}  # ALLOW-ALL
# Defense-in-depth via NetworkPolicy
# (same pattern as grafana-policy)

Also added the Prometheus URL to the Kiali config patch:

# infra/observability/kiali/kiali-config-patch.yaml
external_services:
  prometheus:
    url: http://prometheus.observability.svc.cluster.local:9090

Security note: Changing to ALLOW-ALL does NOT reduce security. The prometheus-ingress NetworkPolicy restricts ingress to port 9090 from Grafana, Kiali (istio-system), and the observability namespace only. Prometheus is read-only metrics data. This is the same pattern used for grafana-policy.

Issue #4 — cert-dashboard-operator MissingFIXED

Symptoms

  • curl http://localhost:32600 → connection refused
  • E2E test cert-dashboard.spec.ts:66 fails: "CertDashboard CRD is registered"
  • E2E test security-hardening.spec.ts:109 fails: "cert-dashboard-operator role has scoped RBAC"

Diagnosis

kubectl get crd | grep certdashboard  # Empty
kubectl get deploy -A | grep cert-dashboard  # Empty
kubectl get clusterrole | grep cert-dashboard  # Empty

Root Cause

scripts/up.sh never called scripts/cert-dashboard-up.sh, so the operator, CRD, and CR were entirely absent from the cluster. Session 25 added the cert-dashboard operator but forgot to wire it into the main bootstrap script.

Permanent Fix

# Added to scripts/up.sh step 13c
if [[ -f "${REPO_ROOT}/scripts/cert-dashboard-up.sh" ]]; then
  section "Deploying Cert Dashboard Operator"
  bash "${REPO_ROOT}/scripts/cert-dashboard-up.sh" || \
    warn "cert-dashboard-up.sh failed — cert dashboard may need manual setup"
fi

Issue #5 — Cart + Button Silently Fails to IncrementFIXED

Symptoms

E2E test ui-fixes.spec.ts:43 "minus button decrements authenticated cart item quantity" times out. The test adds a book (qty=1), clicks +, expects qty=2, but the span always shows "1". The API returns HTTP 200 but quantity: 1 in the response body despite the row already existing.

Diagnosis

# Direct API test (bypass UI)
# 1. Clear cart
# 2. POST /ecom/cart with {bookId, quantity: 1}  → Response: qty=1, id=X ✓
# 3. POST /ecom/cart with {bookId, quantity: 5}  → Response: qty=5, id=X
#                                                  ^^ Should be 6 (1+5)!

# Enable Hibernate SQL logging
kubectl set env deployment/ecom-service -n ecom SPRING_JPA_SHOW_SQL=true
kubectl logs -n ecom deploy/ecom-service | grep Hibernate

# Output shows:
# select ci1_0.id, ... from cart_items ... where user_id=? and book_id=?
# (No INSERT, no UPDATE follows!)

Root Cause

With Hibernate 7 (Spring Boot 4.0.3) + CNPG-managed PostgreSQL, the entity returned by findByUserIdAndBookId() was loaded in a state where the subsequent setQuantity() mutation was NOT detected as dirty by the automatic dirty-check. The transaction closed without flushing the UPDATE, so the DB never received the write. The service returned what it thought was the new state (the mutated Java object), but the DB was unchanged.

Calling save() on a managed entity is normally a no-op; Hibernate relies on dirty-checking at flush time. The dirty-check failed here — possibly due to CNPG's read-replica routing causing the entity to be loaded from a transaction that closed before the write.

Permanent Fix

BEFORE (broken)
// CartService.java
return cartItemRepository
  .findByUserIdAndBookId(userId, bookId)
  .map(existing -> {
    existing.setQuantity(
      existing.getQuantity() + quantity);
    return cartItemRepository.save(existing);
    //                        ^^^ Deferred, never flushed
  })
AFTER (works)
// CartService.java
return cartItemRepository
  .findByUserIdAndBookId(userId, bookId)
  .map(existing -> {
    int currentQty = existing.getQuantity();
    int newQty = currentQty + quantity;
    existing.setQuantity(newQty);
    return cartItemRepository
      .saveAndFlush(existing);
    //  ^^^ Immediate INSERT/UPDATE
  })

saveAndFlush() forces an immediate INSERT/UPDATE at that point in the transaction, bypassing deferred dirty-checking. The setQuantity() method (called by PUT endpoint) always worked because it uses findById() which loaded the entity differently.

Issue #7 — Gateway Stale xDSFIXED

Symptoms

curl https://api.service.net:30000/ecom/books
# Hangs, returns "upstream request timeout" or "connection termination"

Diagnosis

kubectl logs -n infra deploy/bookstore-gateway-istio | grep "xds-grpc closed"
# DeltaAggregatedResources gRPC config stream to xds-grpc closed since 30935s ago
# connection error: desc = "Error while dialing: dial tcp 10.96.94.189:15012: i/o timeout"

kubectl logs -n istio-system -l app=ztunnel | grep "CertificateExpired"
# tls handshake error: AlertReceived(CertificateExpired)

Root Cause

Workload certs are issued by istiod with ~24h TTL. After Docker Desktop sleeps (or when the system clock jumps during day rollover), istiod may need to re-issue certs. The gateway's Envoy xDS stream drops and can't re-establish because the service-account token or cert is stale. Envoy keeps serving with its last-known config — so NEW pod IPs (like newly-restarted ecom-service pods) aren't in its endpoint discovery, causing upstream timeouts.

Permanent Fix

Added istiod + gateway restart to recovery() function:

info "Restarting istiod (xDS control plane)..."
kubectl rollout restart deploy/istiod -n istio-system
kubectl rollout status deploy/istiod -n istio-system --timeout=90s

# ... (ztunnel restart already existed)

info "Restarting gateway (must re-establish xDS stream)..."
kubectl rollout restart deploy/bookstore-gateway-istio -n infra
# Re-patch NodePorts after restart (Istio may reassign them)
kubectl patch svc bookstore-gateway-istio -n infra --type='merge' \
  -p='{"spec":{"ports":[{"name":"https","port":8443,"targetPort":8443,"nodePort":30000,...}]}}'

Key Lessons Learned

  1. Istio Ambient + non-mesh callers: AuthorizationPolicies with namespaces: rules silently fail for callers without SPIFFE identity (non-mesh pods). Use rules: [{}] + NetworkPolicy for defense-in-depth.
  2. Hibernate 7 + CNPG: Prefer saveAndFlush() over save() when modifying entities loaded via derived queries (findByX_AndY) in CNPG environments where read-replica routing may affect persistence context consistency.
  3. xDS config drift: After Docker Desktop sleep or day rollover, include istiod + gateway restart in recovery. Stale Envoy config causes 30+ minute upstream timeouts.
  4. Bootstrap drift: Inline deploys in up.sh can drift from infra-up.sh. Either have up.sh call infra-up.sh, or keep both in sync via a shared manifest list.
