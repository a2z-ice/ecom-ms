1 Azure Regions & Availability Zones

What is it?

An Azure Region is a physical location in the world where Microsoft operates one or more datacenters. Examples: eastus (Virginia, USA), westeurope (Netherlands), southeastasia (Singapore). Each region is a self-contained collection of datacenters connected by a high-speed private network.

Within each region, Availability Zones (AZs) are physically separate buildings with independent power, cooling, and networking. Most regions have three zones, labeled Zone 1, Zone 2, and Zone 3. If Zone 1 loses power, Zones 2 and 3 keep running.

Analogy: A region is a city. Availability Zones are three separate power grids within that city. If one power grid fails, the other two keep the lights on. You spread your critical systems across all three grids so your application survives any single failure.

Why do we need it?

Without zones, all your resources sit in one datacenter. A single fire, flood, or power failure takes everything down. By deploying across zones, you get 99.99% uptime SLA instead of 99.95%. This is the foundation of high availability in Azure.

How does it work?

  • Zone-redundant resources — Azure automatically spreads replicas across all 3 zones. Examples: zone-redundant PostgreSQL, zone-redundant Public IP. You just enable a flag; Azure handles placement.
  • Zonal resources — You pin a resource to a specific zone (e.g., "put this VM in Zone 2"). You manage the redundancy yourself by creating copies in other zones.
  • Regional resources — Some resources (like VNets) span all zones automatically with no configuration needed.
Terraform resource:
azurerm_resource_group — you set the region via the location attribute. Availability Zones are selected per-resource (e.g., zones = ["1", "2", "3"] on a VM Scale Set).
Common mistake: Deploying everything into a single zone (or no zone at all). If that zone goes down, your entire application is offline. Always use zone-redundant resources or spread zonal resources across all three zones.
Cost: Availability Zones themselves are free. However, cross-zone data transfer costs $0.01/GB. For a database with synchronous replication across zones, this adds up. Budget ~$5–15/month for a typical AKS workload.

2 Resource Groups

What is it?

A Resource Group is a logical container that holds related Azure resources together. Every single Azure resource (VM, database, network, AKS cluster) must belong to exactly one resource group. Think of it as a folder on your computer — it does not affect how the resource works, but it organizes everything.

Analogy: A resource group is like a shipping container at a port. All the items for one project go into one container. When you are done with the project, you can delete the entire container (and everything inside) in one action.

Why do we need it?

  • Lifecycle management — Delete the resource group, and ALL resources inside are deleted. Perfect for tearing down dev/test environments.
  • Billing — Azure tracks costs per resource group. You can see exactly how much your AKS environment costs vs. your staging environment.
  • Access control — RBAC (role-based access control) can be scoped to a resource group. Give a developer "Contributor" on the dev resource group without touching production.
  • Organization — One resource group per environment (dev, staging, prod) or per project.

How does it work?

When you create a resource group, you choose a region for it. This region is where the resource group's metadata is stored — but the resources inside can be in any region. In practice, you put the resource group in the same region as its resources for simplicity.

Terraform resource: 
resource "azurerm_resource_group" "main" {
  name     = "rg-bookstore-prod"
  location = "eastus"
  tags = {
    environment = "production"
    project     = "bookstore"
  }
}
Common mistake: Putting dev and prod resources in the same resource group. When someone accidentally deletes the resource group, both environments are gone. Always separate environments into distinct resource groups.
Cost: Resource groups themselves are completely free. They are purely organizational. Create as many as you need.

3 VNet (Virtual Network)

What is it?

A Virtual Network (VNet) is your own isolated, private network inside Azure. It is a software-defined network that mimics a physical network — your resources inside can talk to each other, but nothing from the outside internet can reach them unless you explicitly allow it.

Every VNet has an address space expressed in CIDR notation. For example, 10.0.0.0/16 gives you 65,536 IP addresses (10.0.0.0 through 10.0.255.255). All resources deployed into the VNet receive an IP address from this range.

Analogy: A VNet is your own private floor in Azure's datacenter building. You control who can enter (firewalls), how rooms are divided (subnets), and which rooms can talk to each other (routing rules). Nobody on other floors can see your traffic.

Why do we need it?

  • Isolation — Your AKS pods, databases, and caches communicate over private IPs. No public internet exposure.
  • Security — Network Security Groups (NSGs) filter traffic at the subnet level. Only allowed traffic gets through.
  • Connectivity — VNet peering connects VNets across regions. VPN Gateway connects to on-premises networks.

How does it work?

Key difference from AWS VPC: In Azure, a VNet automatically spans ALL Availability Zones in the region. Subnets are NOT zone-scoped. A single subnet can have resources in Zone 1, 2, and 3 simultaneously. In AWS, each subnet is locked to one AZ. This is a fundamental architectural difference.

Terraform resource: 
resource "azurerm_virtual_network" "main" {
  name                = "vnet-bookstore"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  address_space       = ["10.0.0.0/16"]
}
Common mistake: Choosing an address space that is too small. A /24 gives only 256 IPs. With Azure CNI (where every pod gets a VNet IP), you can exhaust that in minutes. Always start with /16 and subdivide into subnets.
Cost: VNets are free. You only pay for resources deployed inside them and for data transfer (VNet peering: $0.01/GB same-region, $0.035/GB cross-region).
Section 4

4 Subnets

What is it?

A subnet is a range of IP addresses carved out of your VNet. It partitions the VNet into logical sections, each with its own security rules (NSGs), routing tables, and policies. Resources in different subnets can still communicate (they are in the same VNet), but you can control the traffic between them with NSG rules.

Analogy: If the VNet is your private floor in the building, subnets are individual rooms on that floor. The AKS room, the database room, the load balancer room. Each room has its own lock (NSG). People in any room can walk to another room, but the locks decide who gets in.

Why do we need it?

A production AKS deployment typically needs at least three subnets:

  • AKS subnet (/22) — Holds the AKS nodes AND all pod IPs. Why so large? Azure CNI assigns a real VNet IP to every pod. With 30 nodes and 30 pods each, that is 900+ IPs. A /22 gives 1,024 IPs.
  • Database subnet (/24) — Delegated to PostgreSQL Flexible Server for VNet integration. Isolated from other traffic.
  • App Gateway subnet (/24) — Azure Application Gateway requires its own dedicated subnet. No other resources can be placed here.

How does it work?

Critical Azure difference: Subnets in Azure are NOT zone-scoped. A single subnet spans all Availability Zones. This is fundamentally different from AWS, where each subnet lives in exactly one AZ. In Azure, zone placement is decided per-resource, not per-subnet.

Terraform resource: 
resource "azurerm_subnet" "aks" {
  name                 = "snet-aks"
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = ["10.0.0.0/22"]
}

resource "azurerm_subnet" "db" {
  name                 = "snet-db"
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = ["10.0.4.0/24"]
  delegation {
    name = "postgresql-delegation"
    service_delegation {
      name = "Microsoft.DBforPostgreSQL/flexibleServers"
    }
  }
}

resource "azurerm_subnet" "appgw" {
  name                 = "snet-appgw"
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = ["10.0.5.0/24"]
}
Common mistake: Making the AKS subnet too small. A /24 (256 IPs) seems fine until Azure CNI assigns one IP per pod. With autoscaling, you can hit "InsufficientSubnetSize" errors and new pods fail to schedule. Use at least /22 for production.
Cost: Subnets are free. They are logical divisions of your VNet. The resources inside them have costs, but the subnet itself does not.

5 Service Delegation

What is it?

Service Delegation is when you give an Azure managed service (like PostgreSQL Flexible Server) permission to inject its resources directly into one of your subnets. Normally, you control everything in your subnet. With delegation, you say: "Azure, I am handing this subnet over to you so you can place your managed database servers here."

Analogy: Imagine you own an apartment building (your VNet). Service Delegation is like leasing one apartment (subnet) to a specialized company (Azure PostgreSQL). They bring their own furniture and equipment, but the apartment is still inside your building and connected to your hallways (network). You cannot put your own stuff in that apartment while it is leased.

Why do we need it?

Azure PostgreSQL Flexible Server needs VNet integration for private access. Without delegation, PostgreSQL cannot inject its network interfaces into your subnet. Delegation also prevents you from accidentally deploying other resources into the database subnet, which could cause IP conflicts.

How does it work?

  1. You add a delegation block to your subnet definition, specifying Microsoft.DBforPostgreSQL/flexibleServers
  2. Azure takes ownership of the subnet for that service type
  3. When you create a PostgreSQL Flexible Server, you point it to this delegated subnet
  4. Azure injects the database server's NIC into the subnet, giving it a private IP (e.g., 10.0.4.5)
  5. Your AKS pods can now reach the database at that private IP — no public internet involved
Terraform resource: Delegation is part of the azurerm_subnet resource (shown in Section 4 above). The key block: 
delegation {
  name = "postgresql-delegation"
  service_delegation {
    name    = "Microsoft.DBforPostgreSQL/flexibleServers"
    actions = ["Microsoft.Network/virtualNetworks/subnets/join/action"]
  }
}
Common mistake: Forgetting to delegate the subnet before creating the PostgreSQL server. The deployment will fail with a cryptic "SubnetMissingServiceDelegation" error. Also, you cannot delegate a subnet that already has non-delegated resources in it — it must be empty or only contain resources of the delegated type.
Cost: Service Delegation is free. It is a network configuration, not a billable resource.
Section 6

6 NAT Gateway

What is it?

A NAT Gateway (Network Address Translation Gateway) gives your private resources a way to reach the public internet for outbound traffic — without exposing them to inbound traffic. Your AKS pods need to pull container images, call external APIs, or download updates. They have private IPs (like 10.0.1.42) that the public internet cannot route to. The NAT Gateway translates those private IPs to a single public IP for outbound connections.

Analogy: NAT Gateway is a one-way door. Your resources can walk out (reach the internet), but nobody from outside can walk in through that door. All outbound traffic appears to come from one public IP address, hiding the internal structure of your network.

Why do we need it?

  • Container image pulls — AKS nodes need to pull images from Docker Hub, GitHub Container Registry, or your ACR (if not using private endpoints)
  • External API calls — Your services may call third-party payment APIs, email services, or webhooks
  • OS/package updates — Node VMs need access to update repositories
  • Consistent source IP — Third-party APIs that whitelist your IP address get a single, stable public IP from the NAT Gateway

How does it work?

Key Azure difference from AWS: In AWS, you need one NAT Gateway per Availability Zone (3 AZs = 3 NAT Gateways = 3x the cost). In Azure, a single NAT Gateway with a zone-redundant public IP serves the entire subnet across all zones. This is simpler and cheaper.

  1. You create a NAT Gateway resource and associate it with a public IP (or public IP prefix)
  2. You attach the NAT Gateway to one or more subnets
  3. Any resource in those subnets that initiates an outbound connection will use the NAT Gateway's public IP
  4. Return traffic (responses) is allowed back in through the same translated connection
Terraform resources: 
resource "azurerm_public_ip" "nat" {
  name                = "pip-nat-bookstore"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  allocation_method   = "Static"
  sku                 = "Standard"
  zones               = ["1", "2", "3"]  # zone-redundant
}

resource "azurerm_nat_gateway" "main" {
  name                = "natgw-bookstore"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  sku_name            = "Standard"
}

resource "azurerm_nat_gateway_public_ip_association" "main" {
  nat_gateway_id       = azurerm_nat_gateway.main.id
  public_ip_address_id = azurerm_public_ip.nat.id
}

resource "azurerm_subnet_nat_gateway_association" "aks" {
  subnet_id      = azurerm_subnet.aks.id
  nat_gateway_id = azurerm_nat_gateway.main.id
}
Common mistake: Not attaching a NAT Gateway to the AKS subnet. AKS nodes will fall back to Azure's default SNAT (which uses ephemeral IPs that can change). Third-party IP whitelists break, and you may hit SNAT port exhaustion under load (max 1,024 ports per VM without NAT Gateway).
Cost: NAT Gateway: ~$32/month (fixed) + $0.045/GB processed data. Much cheaper than AWS (where 3 NAT Gateways cost ~$96/month). Budget ~$40–50/month for a typical AKS workload.

7 Network Security Groups (NSGs)

What is it?

A Network Security Group (NSG) is a firewall that filters network traffic to and from Azure resources. It contains a list of security rules that allow or deny traffic based on source, destination, port, and protocol. NSGs can be attached to subnets (applies to all resources in the subnet) or to individual network interfaces (applies to one specific resource).

Analogy: An NSG is a bouncer at a nightclub door. The bouncer has a list (rules) that says who can come in (inbound) and who can leave (outbound). Each rule has a priority number — the bouncer checks the highest-priority rule first. If someone matches an "allow" rule, they get in. If they match a "deny" rule, they are turned away. If no rule matches, the default is deny.

Why do we need it?

  • Defense in depth — Even if someone gets into your VNet, NSGs prevent lateral movement between subnets
  • Compliance — Auditors want to see explicit firewall rules at the network layer
  • Least privilege — The database subnet only accepts traffic from the AKS subnet on port 5432. Everything else is blocked.

How does it work?

Each NSG rule has these fields:

FieldExampleDescription
Priority100Lower number = higher priority. Range: 100–4096
DirectionInboundInbound (incoming) or Outbound (outgoing)
Source10.0.0.0/22IP range, service tag, or ASG
Destination10.0.4.0/24IP range, service tag, or ASG
Port5432TCP/UDP port number
ActionAllowAllow or Deny the traffic

Azure evaluates rules in priority order (100 first, then 200, etc.). The first matching rule wins. Three default rules exist in every NSG (you cannot delete them): Allow VNet inbound, Allow Azure Load Balancer, Deny all other inbound.

Terraform resources: 
resource "azurerm_network_security_group" "db" {
  name                = "nsg-db-subnet"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
}

resource "azurerm_network_security_rule" "allow_aks_to_pg" {
  name                        = "allow-aks-postgres"
  priority                    = 100
  direction                   = "Inbound"
  access                      = "Allow"
  protocol                    = "Tcp"
  source_address_prefix       = "10.0.0.0/22"
  source_port_range           = "*"
  destination_address_prefix  = "10.0.4.0/24"
  destination_port_range      = "5432"
  resource_group_name         = azurerm_resource_group.main.name
  network_security_group_name = azurerm_network_security_group.db.name
}

resource "azurerm_subnet_network_security_group_association" "db" {
  subnet_id                 = azurerm_subnet.db.id
  network_security_group_id = azurerm_network_security_group.db.id
}
Common mistake: Putting an NSG on the AKS subnet that blocks ports AKS needs internally (kube-apiserver:443, kubelet:10250, DNS:53). AKS nodes fail health checks, pods cannot schedule, and kubectl commands time out. Always check Microsoft's required ports list before locking down AKS subnet NSGs.
Cost: NSGs are completely free. You can create as many rules as you want (limit: 1,000 rules per NSG, 5,000 NSGs per subscription). No traffic processing charges.

8 Private Endpoints

What is it?

A Private Endpoint projects an Azure PaaS service (like PostgreSQL, Redis, Key Vault, or Event Hubs) into your VNet by creating a network interface with a private IP address. Instead of connecting to mydb.postgres.database.azure.com over the public internet, your traffic goes to 10.0.4.20 — a private IP inside your VNet. The data never leaves Azure's backbone network.

Analogy: Imagine a government building (Azure PaaS service) that normally serves citizens through a public window facing the street. A Private Endpoint is like building a private tunnel from your office building directly into the government building. You get the same service, but your paperwork never crosses a public street.

Why do we need it?

  • Security — Traffic stays within your VNet. No exposure to the public internet means no attack surface from the internet.
  • Compliance — Regulations (PCI-DSS, HIPAA) often require that sensitive data never traverse the public internet.
  • Performance — Private traffic on Azure's backbone is faster and more reliable than going through public internet routing.
  • Data exfiltration prevention — You can disable public access entirely, so the only way to reach the service is through the Private Endpoint.

How does it work?

  1. You create a Private Endpoint for the target service (e.g., PostgreSQL server) in one of your subnets
  2. Azure creates a network interface (NIC) with a private IP from that subnet
  3. You create a Private DNS Zone (e.g., privatelink.postgres.database.azure.com) and link it to your VNet
  4. Azure adds a DNS record: mydb.postgres.database.azure.com → 10.0.4.20
  5. When your app resolves the hostname, it gets the private IP and traffic stays in the VNet
Terraform resources: 
resource "azurerm_private_endpoint" "postgres" {
  name                = "pe-postgres-bookstore"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  subnet_id           = azurerm_subnet.db.id

  private_service_connection {
    name                           = "psc-postgres"
    private_connection_resource_id = azurerm_postgresql_flexible_server.main.id
    subresource_names              = ["postgresqlServer"]
    is_manual_connection           = false
  }

  private_dns_zone_group {
    name                 = "default"
    private_dns_zone_ids = [azurerm_private_dns_zone.postgres.id]
  }
}

resource "azurerm_private_dns_zone" "postgres" {
  name                = "privatelink.postgres.database.azure.com"
  resource_group_name = azurerm_resource_group.main.name
}

resource "azurerm_private_dns_zone_virtual_network_link" "postgres" {
  name                  = "link-postgres"
  resource_group_name   = azurerm_resource_group.main.name
  private_dns_zone_name = azurerm_private_dns_zone.postgres.name
  virtual_network_id    = azurerm_virtual_network.main.id
}
Common mistake: Creating the Private Endpoint but forgetting the Private DNS Zone and VNet link. Without DNS, your application resolves the hostname to the public IP instead of the private IP. Traffic goes out to the internet and back in, defeating the entire purpose. Always create the DNS zone + link.
Cost: Private Endpoint: ~$7.30/month per endpoint + $0.01/GB inbound/outbound data processed. For 4 services (PostgreSQL, Redis, Event Hubs, Key Vault): ~$30/month. Significantly cheaper than the security risk of public exposure.

9 Service Endpoints vs Private Endpoints

What is it?

Azure offers two ways to connect your VNet to PaaS services privately. Service Endpoints and Private Endpoints solve the same problem (avoid public internet) but work very differently. Understanding when to use each is critical for architecture decisions.

Analogy: Service Endpoints are like an express lane on the highway — you still use the public road system, but Azure gives you a dedicated fast lane to the service. The service still has a public address; you just reach it faster. Private Endpoints are like building a private road from your house directly to the destination — nobody else can see or use that road, and the destination gets a private address in your neighborhood.

Why does this distinction matter?

Choosing the wrong one can create security gaps or unnecessary costs. Here is the complete comparison:

FeatureService EndpointPrivate Endpoint
How it worksOptimized route over Azure backbonePrivate IP injected into your subnet
Service gets a private IP?No — still public IPYes — real private IP in your VNet
DNS resolutionResolves to public IPResolves to private IP (via Private DNS Zone)
On-premises accessOnly from VNet (not from on-prem via VPN)Works from on-prem via VPN/ExpressRoute
Data exfiltration riskCan still reach other tenants' servicesLocked to your specific resource instance
CostFree~$7.30/month per endpoint
ComplexitySimple — one line in subnet configModerate — requires DNS zone, NIC, link
Recommended forDev/test, non-sensitive dataProduction, sensitive/regulated workloads

How does each work?

Service Endpoint: You enable a service endpoint on your subnet (e.g., Microsoft.Sql). Azure adds a route that sends traffic destined for that service type over the Azure backbone instead of the public internet. The service firewall can restrict access to "only from this VNet." But the service itself still has a public IP — the traffic just takes a faster path.

Private Endpoint: Azure creates a NIC with a private IP in your subnet, connected via Private Link to the specific resource. DNS resolves to the private IP. Traffic never leaves your VNet. You can disable the public endpoint entirely.

Terraform resource (Service Endpoint): Service Endpoints are a one-line addition to the subnet: 
resource "azurerm_subnet" "aks" {
  name                 = "snet-aks"
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = ["10.0.0.0/22"]
  service_endpoints    = ["Microsoft.Sql", "Microsoft.Storage"]
}
Private Endpoint Terraform was shown in Section 8.
Common mistake: Using Service Endpoints in production and assuming traffic is "private." It is NOT. The service still has a public IP. Other Azure tenants could theoretically route traffic to it (though firewall rules help). For databases with PII/PHI data, always use Private Endpoints. Service Endpoints are a cost-saving measure for non-sensitive services only.
Cost: Service Endpoints: Free. Private Endpoints: ~$7.30/month each + data processing. For production, the $7.30/month per service is negligible compared to the security benefit. Use Private Endpoints for all production databases and secret stores.

10 AKS (Azure Kubernetes Service)

What is it?

Kubernetes is an open-source system for automating the deployment, scaling, and management of containerized applications. Instead of running your app directly on a server, you package it into a container (a lightweight, portable box with your code and all its dependencies), and Kubernetes decides which server to run it on, restarts it if it crashes, and scales it up when traffic increases.

AKS (Azure Kubernetes Service) is Azure's managed Kubernetes offering. "Managed" means Azure handles the hard parts — upgrading the Kubernetes control plane, patching the OS on nodes, integrating with Azure networking and identity. You focus on deploying your applications; Azure keeps the underlying infrastructure healthy.

Analogy: Kubernetes is like a shipping port manager. Containers (your apps) arrive, and the port manager decides which dock (server) to place them on, monitors them for damage, and moves them if a dock breaks. AKS is like hiring a professional port management company (Azure) instead of running the port yourself.

Why do we need it?

  • Self-healing — If a container crashes, Kubernetes restarts it automatically. If a node (VM) dies, Kubernetes reschedules all its containers to healthy nodes.
  • Scaling — Horizontal Pod Autoscaler adds more container replicas during traffic spikes. Cluster Autoscaler adds more VMs when pods cannot be scheduled.
  • Rolling updates — Deploy new versions with zero downtime. Kubernetes gradually replaces old containers with new ones.
  • Service discovery — Containers find each other by name (e.g., ecom-service.ecom.svc.cluster.local) without hardcoding IPs.

How does it work?

AKS has two planes:

  1. Control Plane (managed by Azure, free on Free tier) — the "brain" that makes decisions
  2. Data Plane (your VMs in your VNet) — the worker nodes that actually run your containers

AKS pricing tiers:

TierSLACostUse Case
FreeNone$0Dev/test, learning
Standard99.95% (with AZs: 99.99%)$0.10/hr ($73/mo)Production workloads
Premium99.99%$0.16/hr ($117/mo)Mission-critical, long-term support
Terraform resource: 
resource "azurerm_kubernetes_cluster" "main" {
  name                = "aks-bookstore"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  dns_prefix          = "bookstore"
  sku_tier            = "Standard"
  kubernetes_version  = "1.30"

  default_node_pool {
    name           = "system"
    node_count     = 3
    vm_size        = "Standard_D2s_v3"
    vnet_subnet_id = azurerm_subnet.aks.id
    zones          = ["1", "2", "3"]
  }

  identity { type = "SystemAssigned" }

  network_profile {
    network_plugin = "azure"   # Azure CNI
    service_cidr   = "10.1.0.0/16"
    dns_service_ip = "10.1.0.10"
  }
}
Common mistake: Using the Free tier for production. Free tier has NO SLA — Azure can restart your control plane during maintenance with no notice. Production workloads need Standard tier ($73/month) for the 99.95% uptime guarantee.
Cost: Control plane: Free/$73/$117 per month depending on tier. Worker nodes: you pay for the VMs. Three Standard_D2s_v3 nodes (2 vCPU, 8 GB each) cost ~$210/month. Total AKS: ~$283/month (Standard tier + 3 nodes).
Section 11

11 AKS Control Plane (Master Nodes)

What is it?

The Control Plane is the brain of Kubernetes. It consists of several components that make all the scheduling and management decisions. In AKS, Microsoft fully manages the control plane — you cannot SSH into it, you cannot see the VMs running it, and you do not pay directly for the compute it uses (beyond the tier fee).

Why do we need it?

Without the control plane, there is no Kubernetes. The control plane answers questions like: "Where should this new pod run?", "This node just died — what do we do with its pods?", "A developer just applied a new Deployment manifest — make it so."

How does it work?

The control plane has four critical components:

etcdThe database. Stores all cluster state: what pods exist, their desired replicas, node health, secrets. If etcd is lost, the cluster is lost.
SchedulerDecides which node runs each new pod. Considers CPU/memory requests, node taints, affinity rules, and zone distribution.
Controller ManagerRuns control loops that watch cluster state and make corrections. "Deployment says 3 replicas but only 2 exist → create 1 more pod."

Where does the control plane live? On Azure-managed infrastructure, completely outside your VNet. It connects to your worker nodes through a managed tunnel (or, with Private Cluster enabled, through a Private Endpoint in your VNet). You interact with it via the API Server using kubectl or Terraform.

Terraform resource: The control plane is created as part of azurerm_kubernetes_cluster (shown in Section 10). For a private cluster (API server only accessible from your VNet):

  # kubectl must be run from a VM inside the VNet (or via VPN)
}
Common mistake: Enabling private_cluster_enabled = true without planning for access. Once enabled, you cannot run kubectl from your laptop (it cannot reach the private API endpoint). You need a jump box VM inside the VNet, a VPN connection, or use Azure's "Run Command" feature. Set this up BEFORE enabling private cluster.
Cost: The control plane cost is included in the AKS tier fee (Free/$73/$117 per month). You do not pay for the master node VMs — Azure absorbs that cost. This is a major advantage over self-managed Kubernetes where you pay for 3 master VMs (~$300/month).
Section 12

12 Node Pools

What is it?

A Node Pool is a group of virtual machines (VMs) with the same configuration that serve as worker nodes in your AKS cluster. Each node pool has a specific VM size, OS, and purpose. You can have multiple node pools with different configurations, allowing you to match the right hardware to the right workload.

Analogy: Think of node pools as different teams in a warehouse. The "system team" (system pool) handles building operations — lighting, HVAC, security cameras. The "workers team" (user pool) handles the actual product packaging and shipping. You would not put the HVAC engineer on the packaging line, and you would not have the warehouse workers maintaining the security system.

Why do we need it?

  • System pool — Runs Kubernetes infrastructure: CoreDNS, kube-proxy, Istio control plane, metrics-server. These are lightweight but critical. Typically 2–3 nodes with smaller VMs.
  • Application pool — Runs your actual services: ecom-service, inventory-service, UI. Can scale independently based on traffic. Larger VMs with more CPU/memory.
  • Specialized pools — GPU pool for ML workloads, high-memory pool for analytics, spot pool for batch jobs (cheaper but can be evicted).

How does it work?

Kubernetes uses taints and tolerations to control which pods land on which nodes. The system pool has a taint CriticalAddonsOnly=true:NoSchedule — only system pods (which have the matching toleration) can run there. Your application pods are directed to the user pool.

AKS Cluster Autoscaler can independently scale each pool: if the app pool is at capacity, it adds nodes to the app pool (not the system pool).

Terraform resource: 
# System pool is the default_node_pool (in azurerm_kubernetes_cluster)
# App pool is a separate resource:
resource "azurerm_kubernetes_cluster_node_pool" "app" {
  name                  = "app"
  kubernetes_cluster_id = azurerm_kubernetes_cluster.main.id
  vm_size               = "Standard_D4s_v3"
  min_count             = 3
  max_count             = 10
  auto_scaling_enabled  = true
  zones                 = ["1", "2", "3"]
  vnet_subnet_id        = azurerm_subnet.aks.id

  node_labels = { "workload" = "app" }
  node_taints = []  # No taints — app pods welcome here
}
Common mistake: Running application pods on the system pool. System components (CoreDNS, kube-proxy) compete with your app for resources. During a traffic spike, your app can OOM-kill CoreDNS, causing DNS failures across the entire cluster. Always use at least two pools and taint the system pool.

• Standard_D4s_v3 (4 CPU, 16 GB): ~$140/month per node
System pool (3 x D2s): ~$210/month. App pool (3 x D4s): ~$420/month. Total: ~$630/month for nodes alone. Use Reserved Instances (1-year: ~35% off, 3-year: ~55% off) for production.

13 Azure CNI Networking

What is it?

CNI (Container Network Interface) defines how pods get their IP addresses. Azure CNI is Azure's native implementation: every pod gets a real IP address directly from your VNet subnet. This means pods are first-class citizens on your network — they can be reached by any other resource in the VNet without any NAT or overlay translation.

Analogy: In a hotel, each guest (pod) can get either a room phone (overlay network — internal extension only, needs the front desk to connect external calls) or a direct phone line (Azure CNI — a real phone number anyone can dial directly). Azure CNI gives every pod its own direct line.

Why do we need it?

  • Simplicity — No overlay network to debug. Pod IPs are real VNet IPs. NSGs, route tables, and Private Endpoints all work natively with pod traffic.
  • Performance — No encapsulation/decapsulation overhead. Traffic goes directly between pods at line speed.
  • Azure integration — Pods can directly reach Private Endpoints, Service Endpoints, and peered VNets without extra configuration.

How does it work?

Three networking models are available for AKS:

ModelPod IPs FromIP UsageBest For
Azure CNIVNet subnetHigh (1 IP per pod)Standard production workloads
Azure CNI OverlayPrivate CIDR (overlay)Low (subnet IPs only for nodes)Large clusters needing IP conservation
KubenetPrivate 10.244.x.x rangeLowestDev/test only (limited features)

Why /22 subnet? With Azure CNI, if you have 10 nodes and each can run 30 pods, that is 300 pod IPs + 10 node IPs = 310 IPs minimum. Azure pre-allocates IPs, so you need headroom. A /22 gives 1,024 IPs, enough for growth and autoscaling.

Terraform resource: CNI is configured in the network_profile block of azurerm_kubernetes_cluster: 
network_profile {

  dns_service_ip      = "10.1.0.10"
}
Common mistake: Choosing kubenet for production. Kubenet does not support Network Policies, Azure Network Policy Manager, or Windows node pools. It also does not work with Private Endpoints natively. Always use Azure CNI (standard or overlay) for production.
Cost: Azure CNI is free (it is a networking mode, not a billed resource). However, it consumes more IP addresses from your subnet, so you need a larger subnet — which is also free. The trade-off is IP space, not money.
Section 14

14 Pods, Deployments, Services

What is it?

These are the three core Kubernetes building blocks for running applications:

  • Pod — The smallest deployable unit. Contains one or more containers that share networking and storage. A pod gets one IP address. Usually one container per pod.
  • Deployment — Manages a set of identical pods. "I want 3 copies of ecom-service always running." If a pod dies, the Deployment creates a replacement.
  • Service — A stable network endpoint (DNS name + IP) that routes traffic to the pods behind it. Pods come and go; the Service stays constant.
Analogy: A Pod is a single worker at a counter. A Deployment is the staffing policy: "always keep 3 workers at the counter." A Service is the counter number that customers use to find an available worker — customers do not need to know which specific worker will serve them.

Why do we need it?

Without Deployments, you would have to manually create and track each pod. Without Services, every time a pod restarts (and gets a new IP), all its clients would lose connectivity.

How does it work (Azure-specific)?

  • Azure Disk — PersistentVolumeClaims (PVCs) in AKS are backed by Azure Managed Disks (SSD by default). The CSI driver automatically provisions and attaches disks.
  • Azure Load Balancer — A Service of type: LoadBalancer creates an Azure Standard Load Balancer with a public or internal IP. For internal services, use the annotation service.beta.kubernetes.io/azure-load-balancer-internal: "true".
  • In our architecture — We use Application Gateway as the external load balancer and only ClusterIP Services internally (routed by Istio).
Terraform resource: Pods, Deployments, and Services are Kubernetes resources, not Azure resources. You deploy them with kubectl apply or the Terraform kubernetes_deployment / kubernetes_service providers. The AKS-specific integration (Azure Disk, Load Balancer) is automatic via AKS's built-in CSI drivers and cloud controller.
Common mistake: Not setting resource requests and limits on pods. Without limits, a single pod can consume all CPU/memory on a node, starving other pods. Without requests, the scheduler cannot make intelligent placement decisions. Always set both.
Cost: Pods themselves are free — they consume the VM resources you already pay for via node pools. Azure Managed Disks (PVCs): Standard SSD 32 GB = ~$3/month, Premium SSD 32 GB = ~$5/month. Azure Load Balancer: ~$18/month (if you use one directly, but with App Gateway ingress, you may not need it).

15 Workload Identity (Azure's IRSA Equivalent)

What is it?

Workload Identity allows Kubernetes pods to authenticate to Azure services (Key Vault, PostgreSQL, ACR, Blob Storage) using Azure AD without static credentials. Instead of storing a password or API key in a Kubernetes Secret, the pod presents a short-lived token that Azure validates. This is the Azure equivalent of AWS IRSA (IAM Roles for Service Accounts).

Analogy: Instead of giving your employee a master key to the building (static credentials), you give them an ID badge (Workload Identity). The badge is scanned at each door, and the building's security system checks whether that specific employee is authorized for that specific room. The badge expires daily and cannot be copied.

Why do we need it?

  • No static credentials — Passwords in Kubernetes Secrets can be leaked, stolen, or accidentally committed to Git. Workload Identity eliminates them.
  • Least privilege — Each pod (via its ServiceAccount) gets only the permissions it needs. The ecom-service pod can read from Key Vault but cannot delete databases.
  • Automatic rotation — Tokens are short-lived (hours, not months). No manual rotation needed.
  • Audit trail — Azure AD logs which pod accessed which resource, when, and from where.

How does it work?

  1. AKS creates an OIDC Issuer endpoint (a URL that Azure AD trusts)
  2. You create a User-Assigned Managed Identity in Azure (this is the "role")
  3. You create a Federated Identity Credential that links the Managed Identity to a specific Kubernetes ServiceAccount in a specific namespace
  4. In your pod spec, you reference that ServiceAccount
  5. When the pod starts, the Workload Identity webhook injects environment variables and a projected token volume
  6. Azure SDKs automatically use the token to authenticate — no code changes needed
Terraform resources: 
# 1. Enable OIDC Issuer on AKS
resource "azurerm_kubernetes_cluster" "main" {
  ...
  oidc_issuer_enabled       = true
  workload_identity_enabled = true
}

# 2. Create a Managed Identity
resource "azurerm_user_assigned_identity" "ecom" {
  name                = "mi-ecom-service"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
}

# 3. Federated Credential (links K8s SA to Azure identity)
resource "azurerm_federated_identity_credential" "ecom" {
  name                = "fic-ecom-service"
  resource_group_name = azurerm_resource_group.main.name
  parent_id           = azurerm_user_assigned_identity.ecom.id
  audience            = ["api://AzureADTokenExchange"]
  issuer              = azurerm_kubernetes_cluster.main.oidc_issuer_url
  subject             = "system:serviceaccount:ecom:sa-ecom"
}

# 4. Grant the identity access (e.g., Key Vault Secrets User)
resource "azurerm_role_assignment" "ecom_kv" {
  scope                = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets User"
  principal_id         = azurerm_user_assigned_identity.ecom.principal_id
}
Common mistake: Getting the subject wrong in the Federated Identity Credential. It must exactly match system:serviceaccount:<namespace>:<serviceaccount-name>. A typo means the token exchange silently fails, and your pod gets "DefaultAzureCredential authentication failed" errors with no clear cause.
Cost: Workload Identity and Managed Identities are completely free. Token exchanges, OIDC validation — all free. This is one of the best security features in Azure with zero cost.

16 Azure Container Registry (ACR)

What is it?

Azure Container Registry (ACR) is a private Docker registry hosted on Azure. When you build a Docker image of your application, you push it to ACR. When AKS needs to run that application, it pulls the image from ACR. Think of it as a private storage locker for your container images.

Analogy: Docker Hub is like a public library — anyone can borrow books (images). ACR is like your company's private document vault — only authorized employees (AKS clusters) can access the materials inside, and everything is scanned for security before being stored.

Why do we need it?

  • Security — Your images stay private. No one outside your Azure AD tenant can access them.
  • Speed — ACR is in the same Azure region as your AKS cluster. Image pulls are fast (same datacenter network) vs. pulling from Docker Hub (public internet).
  • Integration — AKS can pull from ACR without Docker login credentials — just assign the AcrPull role to the AKS managed identity.
  • Vulnerability scanning — Premium tier includes Microsoft Defender for Containers, which scans images for known CVEs.

How does it work?

  1. CI/CD pipeline builds Docker image: docker build -t myacr.azurecr.io/ecom-service:v1.2 .
  2. Pipeline pushes to ACR: docker push myacr.azurecr.io/ecom-service:v1.2
  3. Kubernetes Deployment references the image: image: myacr.azurecr.io/ecom-service:v1.2
  4. AKS kubelet pulls the image using its managed identity (no password needed)
Terraform resources: 
resource "azurerm_container_registry" "main" {
  name                = "acrbookstore"  # must be globally unique
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
  sku                 = "Standard"
  admin_enabled       = false  # Use managed identity, not admin password
}

  role_definition_name = "AcrPull"
  principal_id         = azurerm_kubernetes_cluster.main.kubelet_identity[0].object_id
}
Common mistake: Enabling admin_enabled = true and using the admin username/password. This is a single shared credential with full push/pull access. If it leaks, anyone can push malicious images to your registry. Use Managed Identity + AcrPull role instead. Always set admin_enabled = false.
Cost: Basic: $5/month (10 GB storage). Standard: $20/month (100 GB, webhooks). Premium: $50/month (500 GB, geo-replication, private link). Standard is the sweet spot for most production workloads.

21 Application Gateway (v2)

What is it?

Azure Application Gateway is a Layer 7 (HTTP/HTTPS) load balancer with a built-in Web Application Firewall (WAF). It receives all incoming user traffic, inspects it for attacks (SQL injection, XSS, OWASP Top 10), terminates TLS, and routes requests to the correct backend service based on the URL path or hostname.

Analogy: Application Gateway is the reception desk and security checkpoint at a corporate campus. Every visitor (request) must pass through. The receptionist (routing rules) checks who you are visiting (hostname/path) and directs you to the right building. The security guard (WAF) checks your bag for prohibited items (malicious payloads) and turns you away if something is suspicious.

Why do we need it?

  • WAF protection — Blocks SQL injection, cross-site scripting, and other OWASP Top 10 attacks before they reach your application code
  • TLS termination — Handles HTTPS certificates centrally. Backend pods communicate over HTTP internally (Istio mTLS handles internal encryption).
  • Host-based routing — myecom.net goes to the UI service, api.service.net goes to ecom-service. One gateway, multiple backends.
  • Zone redundancy — Automatically spans all 3 Availability Zones for high availability

How does it work?

Application Gateway has four core concepts:

22 Azure DNS

What is it?

Azure DNS is a managed DNS hosting service. DNS (Domain Name System) translates human-readable domain names (like myecom.net) into IP addresses (like 20.72.1.50) that computers use to find each other. Azure DNS hosts your DNS zone and answers queries from anywhere in the world using Azure's global network of DNS servers.

Analogy: DNS is like a phone book for the internet. When someone types myecom.net, DNS looks up the "phone number" (IP address) of your Application Gateway. Azure DNS is a hosted phone book service — you provide the entries, Azure distributes copies worldwide so lookups are fast everywhere.

Why do we need it?

  • Human-readable URLs — Users type myecom.net instead of 20.72.1.50
  • High availability — Azure DNS uses anycast routing across hundreds of global servers. 100% uptime SLA.
  • Terraform integration — DNS records are managed as code, version-controlled, and automatically updated when infrastructure changes.

How does it work?

  1. You register a domain (e.g., myecom.net) with any registrar (GoDaddy, Namecheap, etc.)
  2. You create a DNS Zone in Azure for that domain
  3. You point your registrar's nameservers to Azure DNS nameservers (one-time setup)
  4. You create A records pointing to your Application Gateway public IP
  5. When a user visits myecom.net, the DNS query goes to Azure DNS, which returns the App Gateway IP
Terraform resource: 
resource "azurerm_dns_zone" "main" {
  name                = "myecom.net"
  resource_group_name = azurerm_resource_group.main.name
}

resource "azurerm_dns_a_record" "root" {
  name                = "@"  # root domain

  ttl                 = 300
  target_resource_id  = azurerm_public_ip.appgw.id  # alias record
}

}
Common mistake: Creating the DNS zone but forgetting to update your domain registrar's nameservers to Azure DNS. Until the registrar points to Azure's nameservers (ns1-xx.azure-dns.com), all DNS queries go to the old provider and Azure DNS is never queried. This takes up to 48 hours to propagate.
Cost: DNS Zone: $0.50/zone/month. Queries: $0.40 per million queries. For a bookstore with 100k daily users: ~$1/month total. DNS is extremely cheap.

23 Managed TLS Certificates

What is it?

TLS certificates enable HTTPS — the padlock icon in your browser. They encrypt traffic between the user's browser and your server, and they prove your server is who it claims to be (not an impersonator). In Azure, you have multiple ways to manage these certificates, from fully automated to manually uploaded.

Analogy: A TLS certificate is like a notarized ID card for your website. When a browser connects, the website presents its ID card. The browser checks with the notary (Certificate Authority) to confirm it is legitimate. If the ID checks out, the browser establishes an encrypted conversation. If not, the user sees a scary "Your connection is not private" warning.

Why do we need it?

  • Encryption — Without TLS, anyone on the network can read passwords, credit card numbers, and personal data in plaintext
  • Trust — Browsers show a padlock for HTTPS sites. Chrome marks HTTP sites as "Not Secure" since 2018.
  • SEO — Google ranks HTTPS sites higher than HTTP sites
  • Compliance — PCI-DSS requires TLS for any site handling payment information

24 Azure Key Vault

What is it?

Azure Key Vault is a centralized secrets management service. It stores three types of sensitive data: Secrets (database passwords, API keys), Keys (encryption keys for data-at-rest), and Certificates (TLS certs). All stored in FIPS 140-2 Level 2 validated hardware security modules (HSMs).

Analogy: Key Vault is a bank safety deposit vault for your application's secrets. Instead of hiding your house key under the doormat (hardcoded passwords in code), you store it in the bank vault. Only people with verified identity (Managed Identities) and the right authorization (RBAC roles) can open the vault and retrieve specific items.

Why do we need it?

  • No secrets in code — Database passwords, Kafka connection strings, and API keys are stored in Key Vault, not in Git repos or Kubernetes manifests
  • Audit trail — Every access is logged: who read which secret, when, from which IP
  • Rotation — Secrets can be versioned. Update a secret, and applications pick up the new version automatically via the External Secrets Operator
  • Kubernetes integration — External Secrets Operator (ESO) syncs Key Vault secrets into Kubernetes Secrets, so pods access them normally via env or volumeMount

How does it work?

The flow from Key Vault to your pods:

  1. DevOps engineer stores the database password in Key Vault as secret ecom-db-password
  2. External Secrets Operator (running in AKS) watches for ExternalSecret custom resources
  3. ESO authenticates to Key Vault using Workload Identity (no static credentials)
  4. ESO creates a Kubernetes Secret with the value from Key Vault
  5. Pod reads the Kubernetes Secret via secretKeyRef in its env configuration
  6. When the Key Vault secret changes, ESO automatically updates the Kubernetes Secret
Terraform resource: 
resource "azurerm_key_vault" "main" {
  name                       = "kv-bookstore"
  location                   = azurerm_resource_group.main.location
  resource_group_name        = azurerm_resource_group.main.name
  tenant_id                  = data.azurerm_client_config.current.tenant_id
  sku_name                   = "standard"
  soft_delete_retention_days = 7
  purge_protection_enabled   = true  # Prevents permanent deletion
  enable_rbac_authorization  = true  # Use RBAC, not access policies
}

  key_vault_id = azurerm_key_vault.main.id
}
Common mistake: Enabling purge_protection in dev environments. Once enabled, you CANNOT delete the Key Vault for the retention period (7–90 days), even after deleting the resource group. This blocks Terraform destroy and makes dev cleanup painful. Only enable purge protection in production.
Cost: Standard tier: $0.03 per 10,000 operations. Secrets retrieval: ESO polls every 60s, so ~43,200 ops/month per secret = ~$0.13/month for 10 secrets. Total: ~$1–2/month. Premium tier (HSM-backed keys): $1 per key/month + $0.03 per 10,000 ops.

25 Managed Identities

What is it?

A Managed Identity is an Azure AD identity automatically managed by Azure. It gives your resources (VMs, AKS clusters, App Services) an identity they can use to authenticate to other Azure services without storing any credentials. Azure handles the credential lifecycle (creation, rotation, deletion) completely.

Analogy: A Managed Identity is like an employee ID badge that Azure issues and manages. You never see the badge number (credentials), Azure rotates it automatically, and if the employee (resource) leaves the company (is deleted), the badge is automatically deactivated. Compare this to a contractor who brings their own key (service principal with client secret) that you must remember to revoke.

Why do we need it?

Without Managed Identities, you would need service principals with client secrets — essentially passwords that must be stored somewhere, rotated periodically, and can be leaked. Managed Identities eliminate all of this.

26 Azure RBAC & Azure Policy

What is it?

Azure RBAC (Role-Based Access Control) controls who can do what on which resources. It is the authorization layer for all of Azure. Azure Policy enforces organizational rules across your resources (e.g., "all resources must have a cost-center tag," "no public IP addresses allowed in production").

Analogy: RBAC is like a key card system in an office building. Different employees (identities) have different access levels (roles) to different floors (scopes). The CEO has access everywhere; an intern only has access to their floor. Azure Policy is like building codes — rules that apply to everyone: "all doors must be fire-rated," "no smoking indoors." You cannot override building codes, even with a master key.

Why do we need it?

  • RBAC — Developers can deploy to the dev AKS cluster but cannot touch production. The CI/CD pipeline has Contributor access to deploy but cannot change networking. Security team has Reader access to audit but cannot modify anything.
  • Azure Policy — Enforces compliance automatically. A developer cannot accidentally create a storage account without encryption, or deploy a VM without NSGs. Violations are blocked (Deny effect) or flagged (Audit effect).

How does it work?

RBAC has three elements:

ScopeWhere the role applies/subscriptions/.../resourceGroups/rg-bookstore

Built-in roles: Owner (full access), Contributor (create/modify, no RBAC changes), Reader (view only). Plus hundreds of service-specific roles like AcrPull, Key Vault Secrets User, Azure Kubernetes Service Cluster User Role.

Azure Policy for AKS: The AKS Azure Policy addon installs a webhook that intercepts Kubernetes API requests and enforces policies. Examples:

  • Pods must not run as root
  • Containers must have resource limits
  • Only images from your ACR are allowed
  • No privileged containers

27 Istio Ambient Mesh

What is it?

Istio is a service mesh — a dedicated infrastructure layer for managing service-to-service communication. Ambient Mesh is Istio's latest mode that provides mutual TLS (mTLS) encryption and authorization between pods WITHOUT injecting sidecar proxy containers. Instead, it uses a per-node ztunnel daemon that transparently intercepts and encrypts traffic.

Analogy: Imagine an office building where every conversation between departments must be encrypted and authorized. The old approach (sidecar mode) is like giving every employee a personal bodyguard who encrypts their messages. Ambient Mesh is like installing building-wide encrypted phone lines (ztunnel) — every call is automatically encrypted without personal bodyguards, saving headcount (CPU/memory).

Why do we need it?

  • Zero-trust networking — Every pod-to-pod connection is encrypted with mTLS. Even if someone compromises the network, they cannot read traffic.
  • Authorization — AuthorizationPolicy defines which pods can talk to which. ecom-service can call inventory-service, but inventory-service cannot call ecom-service.
  • No code changes — mTLS and authorization are configured via Kubernetes CRDs. Your application code has zero awareness of the mesh.
  • Observability — Istio automatically collects metrics (request rate, error rate, latency) for every service-to-service call.

How does it work?

Three layers:

  1. ztunnel (per-node DaemonSet) — Handles L4 mTLS encryption/decryption. Transparent to pods. Low overhead (~10 MB per node).
  2. Waypoint proxies (optional, per-namespace) — Handle L7 policies (HTTP routing, header-based authorization). Only deployed if you need L7 features.
  3. Istiod (control plane) — Distributes certificates, pushes configuration to ztunnels and waypoints.
Terraform / Helm: Istio is deployed inside AKS via Helm:

}

resource "helm_release" "istiod" {

Common mistake: Enabling STRICT mTLS before all services have been enrolled in the mesh. If service A is in the mesh (expects mTLS) but service B is not (sends plain HTTP), connections fail with "connection reset by peer." Roll out PeerAuthentication: PERMISSIVE first, verify all traffic is mTLS, then switch to STRICT.
Cost: Istio is open-source and free. The only cost is the compute resources ztunnel consumes on each node (~10 MB RAM, negligible CPU). Istiod control plane: ~500 MB RAM, 0.5 CPU. Total additional cost: effectively $0 beyond the existing node pool.

28 Container Insights (Azure Monitor)

What is it?

Container Insights is Azure's built-in monitoring feature for AKS. It collects metrics (CPU, memory, disk, network) and logs (container stdout/stderr) from every node and pod in your cluster and stores them in Azure Monitor (a Log Analytics Workspace). You get pre-built dashboards, alerts, and KQL (Kusto Query Language) for custom queries.

Analogy: Container Insights is like the CCTV system and environmental sensors in a building. You can see which rooms (pods) are hot (high CPU), which hallways (networks) are congested, and review footage (logs) when something goes wrong. All without installing your own cameras.

Why do we need it?

  • Out-of-the-box — Enable it with one flag on the AKS cluster. No Prometheus setup, no Grafana installation. Instant metrics.

  • Container logs (stdout and stderr streams)
  • Kubernetes events (pod scheduling, node conditions, OOM kills)

30 Complete Data Flow

What is it?

This section traces a single user request through every Azure component, from the browser to the database and back. Understanding the full path helps you debug issues (where exactly is the request failing?) and optimize performance (where is the bottleneck?).

User Request Flow: Browser to UI to API to Database

  1. DNS Resolution — Browser asks Azure DNS: "what is the IP for myecom.net?" Response: 20.72.1.50 (App Gateway public IP)
  2. TLS Handshake — Browser connects to App Gateway on port 443. App Gateway presents the TLS certificate (from Key Vault). Encrypted HTTPS channel established.
  3. WAF Inspection — App Gateway WAF scans the request for SQL injection, XSS, and other OWASP attacks. Clean requests proceed; malicious ones are blocked (403).
  4. Routing — App Gateway matches the hostname (myecom.net) and routes to the UI backend pool.
  5. AKS Ingress — Traffic enters the AKS cluster via the internal load balancer. Istio ztunnel intercepts and applies mTLS.
  6. UI Pod — Nginx serves the React SPA (static HTML/JS/CSS).
  7. API Call — Browser JS calls api.service.net/ecom/books. Same path through DNS → App Gateway → AKS, but routes to the ecom-service backend pool.
  8. JWT Validation — ecom-service validates the JWT (from Keycloak) using the JWKS endpoint.
  9. Database Query — ecom-service queries PostgreSQL Flexible Server via the Private Endpoint (private IP in VNet).
  10. Response — Data flows back: PostgreSQL → ecom-service → App Gateway → Browser.

CDC (Change Data Capture) Flow

  1. User completes checkout → ecom-service writes order to PostgreSQL
  2. PostgreSQL WAL (write-ahead log) records the change (wal_level = logical)
  3. Debezium reads the WAL via logical replication slot
  4. Debezium publishes the change event to Azure Event Hubs (order.created topic)
  5. Flink SQL consumes from Event Hubs, transforms, and writes to the analytics PostgreSQL database
  6. Superset queries the analytics database and renders dashboards
Terraform: The complete data flow is the result of all 29 previous sections working together. There is no single Terraform resource for the "data flow" — it emerges from the correct configuration of DNS, App Gateway, AKS, Private Endpoints, Event Hubs, and service code.
Common mistake: Not tracing the full path during debugging. When "the API is slow," the issue could be at any layer: DNS resolution, TLS handshake, WAF inspection, pod scheduling, database query, or Event Hubs latency. Use Tempo distributed tracing to pinpoint the exact bottleneck.
Cost: The full data flow touches every paid service. See Section 31 for the complete cost breakdown.

  • Reserved Instances (1-year commit): 35% off VMs and PostgreSQL. Saves ~$500/month on production.
  • Spot Node Pools: For batch/analytics workloads, use spot instances (up to 90% cheaper, but can be evicted).
  • Right-size databases: Start with Burstable, monitor CPU credits, upgrade to GP only if needed.
  • Shut down dev at night: Scale dev AKS to 0 nodes and stop DB servers. Saves ~60% on dev costs.
  • Azure Advisor: Free tool that scans your resources and recommends cost savings.
Common mistake: Not setting Azure budgets and alerts. Without alerts, a runaway autoscaler or forgotten dev environment can generate a surprise $5,000 bill. Always set a monthly budget in Azure Cost Management with email alerts at 50%, 80%, and 100% thresholds.
