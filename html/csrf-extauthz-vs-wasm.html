1. Executive Summary

Verdict: ext_authz is the better choice for this use case

The BookStore platform's CSRF protection is best served by the current ext_authz Go microservice because it provides process isolation from the gateway, battle-tested fail-open semantics with a single Istio config line, and a maintenance model that any Go developer can contribute to without specialized Wasm toolchain knowledge. The marginal latency advantage of an in-process Wasm plugin (1-3ms savings per request) is irrelevant at the platform's scale and is largely negated by the fact that both approaches still require an async Redis call for token validation.

4. Security Deep Dive

Process Isolation (ext_authz)

The csrf-service runs in its own container with a minimal attack surface:

  • Distroless base image -- no shell, no package manager, no unnecessary binaries
  • Read-only root filesystem -- readOnlyRootFilesystem: true
  • Drop ALL capabilities -- capabilities: drop: ["ALL"]
  • Non-root user -- runAsNonRoot: true, runAsUser: 1000

A vulnerability in the CSRF validation code (buffer overflow in a dependency, logic bug in token comparison) is contained within the csrf-service container. It cannot affect the Envoy gateway process, other services, or the cluster.

Wasm Sandbox Limitations

Wasm runs inside Envoy's process in a sandboxed VM (V8 or Wasmtime). The sandbox provides memory isolation and controlled ABI access. However, it has full access to:

  • All request headers, body, and trailers passing through the filter
  • Envoy's internal state via the ABI (route metadata, cluster info)
  • The ability to modify responses, add headers, and reject requests

A Wasm panic or unrecoverable error causes the entire Envoy filter chain to fail. Depending on configuration, this can either crash the gateway pod or leave it in an inconsistent state where subsequent requests are also affected.

Memory Safety

  • Go (ext_authz): Garbage collected, memory safe. No buffer overflows, no use-after-free. Well-understood memory model.
  • Rust (Wasm): Also memory safe (ownership model). Equivalent guarantees. However, TinyGo (Go for Wasm) has known limitations: reduced stdlib, different garbage collector, less testing coverage.

Supply Chain

  • ext_authz: Standard Go modules (go-redis/redis/v9, prometheus/client_golang). Widely audited, reproducible builds, extensive CVE monitoring.
  • Wasm: Requires the proxy-wasm SDK with fewer contributors and less scrutiny. The Wasm build toolchain has more moving parts and fewer security-focused reviewers.

5. Resilience Deep Dive

ext_authz Fail-Open (Native Support)

Istio's extensionProvider configuration supports native fail-open behavior with a single config line:

extensionProviders:
  - name: csrf-ext-authz
    envoyExtAuthz:
      service: csrf-service.infra.svc.cluster.local
      port: 8080
      failOpen: true
      timeout: 2s

If the csrf-service is unreachable (pod crash, network partition, Redis outage), all requests pass through without CSRF validation. This is deliberate: availability takes priority over CSRF enforcement. The csrf-service also implements application-level fail-open (if Redis is unreachable, Validate() returns true).

Wasm Fail Behavior (Complex)

Wasm modules can be configured with failStrategy: FAIL_OPEN in the WasmPlugin CRD. However:

  • A panic in the Wasm VM may leave Envoy's filter chain in an inconsistent state
  • Envoy's Wasm runtime has a panic counter; after too many panics, it may disable the module entirely
  • FAIL_OPEN handles module loading failures but may not handle all runtime panics gracefully

Circuit Breaking and Independent Restarts

  • ext_authz: Benefits from Envoy's built-in circuit breaker on the outbound cluster. If csrf-service starts returning errors, Envoy circuit-breaks and fails open automatically.
  • Wasm: No equivalent circuit-breaking mechanism for in-process execution.
  • ext_authz restarts: kubectl rollout restart deploy/csrf-service -n infra has zero impact on the gateway.
  • Wasm updates: Require gateway pod restart or Envoy's experimental hot-reload (not production-stable for Wasm in Istio).

6. Performance / High Traffic Deep Dive

Latency Breakdown

Componentext_authzWasm
Authz execution~1ms (TCP + handler)~0.1ms (in-process)
Redis round-trip~1-2ms (go-redis direct)~1-2ms (dispatch_http_call via proxy)
Total overhead~2-5ms~1-3ms
Net difference~1-3ms (negligible at sub-1K RPS)

Scaling Characteristics

  • ext_authz: Scales independently via HPA (2-5 replicas). Each replica has its own Redis connection pool (10 connections). Under 10K+ RPS, additional replicas absorb load.
  • Wasm: Limited by the single gateway pod's CPU. Scaling Wasm means scaling the gateway itself -- a heavier operation that affects all traffic routing, not just CSRF.

Connection Pooling

The ext_authz csrf-service uses go-redis/redis/v9 with a dedicated connection pool: 10 connections per replica, 3 min idle, 5s connect timeout, 2s read/write timeout.

Wasm has no native Redis client. To reach Redis, the Wasm module must use one of three significantly more complex approaches:

  1. dispatch_http_call() to an HTTP-to-Redis proxy (e.g., Webdis) -- adds another service
  2. Implement Redis RESP protocol over Envoy's raw TCP support -- complex and fragile
  3. Sidecar container running a Redis proxy co-located with the gateway pod

At What Scale Does Wasm Win?

Only at extreme scale (>50,000 RPS) where the cumulative 1-3ms per-request savings becomes significant. At 100 RPS (BookStore's expected load), the total time saved per second is approximately 0.1-0.3 seconds -- well within noise. For most enterprise applications operating below 10K RPS, ext_authz is more than sufficient.

7. Maintenance / Developer Experience Deep Dive

ext_authz (Current)

  • Language: Go 1.25 -- large hiring pool, widely known
  • Codebase: 770 lines, 11 files, clean internal/ architecture
  • Testing: 25 unit tests (go test), miniredis mocks, httptest handlers
  • Debugging: Delve step debugger, printf debugging, structured slog
  • CI/CD: docker build + kind load + kubectl apply
  • IDE: Full support in GoLand, VS Code (gopls), Neovim

Wasm Plugin (Alternative)

  • Language: Rust recommended. TinyGo possible but no net/http, no encoding/json, no crypto/subtle
  • Build chain: cargo build --target wasm32-wasi -> OCI image -> WasmPlugin CRD
  • Testing: No standard test framework. Must mock entire proxy-wasm ABI or run real Envoy
  • Debugging: proxy_log() statements + Envoy debug logs only. No step debugger
  • CI/CD: Rust toolchain + wasm-pack + OCI build + CRD apply
  • Redis: No Redis client library for Wasm. Must use Envoy's async dispatch_http_call()

8. When Would Wasm Be the Better Choice?

Scenarios Where Wasm Wins

  • Ultra-low-latency requirements (sub-1ms total overhead) where every microsecond matters
  • No external state needed: validation is purely header-based (e.g., check a static header, validate an HMAC signature) with no Redis/database lookup
  • Simple logic: the filter performs a straightforward check without complex business rules
  • Rust/Wasm expertise already exists: the team has developers experienced with proxy-wasm
  • Extremely high RPS (>100K) where eliminating the 2-5ms network hop creates measurable capacity gains
  • Desire to eliminate a deployment: fewer pods, simpler operational footprint -- at the cost of coupling to Envoy's lifecycle

For the BookStore platform, none of these conditions apply. The CSRF logic requires Redis for stateful token storage, the team uses Go (not Rust), the RPS is well under 1K, and process isolation is valued for a security-critical component.

10. Verdict and Recommendation

Recommendation: Keep ext_authz (current approach)

The ext_authz approach wins on every dimension except raw latency and resource efficiency -- both of which are negligible for the BookStore platform's scale.

The Only Scenario Where Wasm Would Be Recommended

If the organization has strong Rust expertise, requires sub-millisecond CSRF overhead, AND the CSRF validation does not need Redis (e.g., uses HMAC-based stateless double-submit cookies where the token is cryptographically verified without a database lookup), then a Wasm plugin would be a reasonable choice. For the BookStore platform, none of these conditions hold.
