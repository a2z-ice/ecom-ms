1. What Is a Kubernetes Operator?

A Kubernetes operator is a custom controller that extends Kubernetes to manage application-specific resources. It follows the "human operator" metaphor: just like a database administrator manages a database, a Kubernetes operator manages a resource automatically.

Standard Kubernetes: You create a Deployment → K8s creates Pods → K8s restarts crashed Pods K8s knows HOW to manage Deployments, Services, ConfigMaps (built-in resources) But K8s doesn't know: How to manage TLS certificates How to display certificate status in a dashboard How to renew certificates by deleting secrets How to stream renewal progress via Server-Sent Events An Operator teaches K8s: You create a CertDashboard → Operator creates Deployment + Service + RBAC Operator watches cert-manager Certificates → shows status in web UI User clicks Renew → Operator deletes secret → cert-manager re-issues
The Operator Pattern in one sentence: Define your application as a Kubernetes resource (CRD), then write a controller that makes reality match the desired state.

2. What This Operator Does

The cert-dashboard-operator manages TLS certificate monitoring for the BookStore platform:

User creates this YAML:

apiVersion: certs.bookstore.io/v1alpha1
kind: CertDashboard
metadata:
  name: bookstore-certs
spec:
  nodePort: 32600
  yellowThresholdDays: 10
  redThresholdDays: 5

Operator automatically creates:

CertDashboard CR (what you wrote) ├── ServiceAccount (identity for the dashboard pod) ├── ClusterRole (permissions to read certificates + secrets) ├── ClusterRoleBinding (binds role to service account) ├── Deployment (runs the dashboard web server) │ └── Pod (Go binary serving HTML + API) └── Service (NodePort) (exposes dashboard on port 32600)

4. The Operator Pattern: Reconciliation Loop

The core of every Kubernetes operator is the reconciliation loop — a function that runs whenever the desired state (your CRD) or actual state (child resources) changes.

The Reconciliation Loop (runs forever): 1. Event triggers reconciliation: • User creates/updates/deletes a CertDashboard CR • A child Deployment/Service/ServiceAccount changes • Periodic resync (every 10 hours by default) 2. Reconciler runs: • Read the CertDashboard CR from Kubernetes API • Compare desired state (spec) with actual state (child resources) • Create/update/delete child resources to match • Update CertDashboard status 3. Return result: • ctrl.Result{} — done, wait for next event • ctrl.Result{RequeueAfter: 10s} — check again in 10 seconds • ctrl.Result{}, err — error, retry with backoff 4. Loop repeats on next event
// The reconciler function signature (controller-runtime convention)
func (r *CertDashboardReconciler) Reconcile(
    ctx context.Context,
    req ctrl.Request,         // Contains: {Namespace: "cert-dashboard", Name: "bookstore-certs"}
) (ctrl.Result, error) {

    // 1. Fetch the CertDashboard custom resource
    dashboard := &certsv1alpha1.CertDashboard{}
    if err := r.Get(ctx, req.NamespacedName, dashboard); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)  // Deleted? That's fine.
    }

    // 2. Create child resources if they don't exist
    r.reconcileServiceAccount(ctx, dashboard)
    r.reconcileClusterRole(ctx, dashboard)
    r.reconcileDeployment(ctx, dashboard)
    r.reconcileService(ctx, dashboard)

    // 3. Update status
    dashboard.Status.Ready = deploymentReady
    dashboard.Status.URL = fmt.Sprintf("http://localhost:%d", dashboard.Spec.NodePort)
    r.Status().Update(ctx, dashboard)

    return ctrl.Result{}, nil
}
Idempotent by design. The reconciler can run 100 times and produce the same result. If the Deployment already exists, it checks if it matches the desired spec and updates only if needed. This is the key principle of Kubernetes controllers.

5. Custom Resource Definition (CRD)

A CRD extends the Kubernetes API with a new resource type. In Go, you define it as a struct with special marker comments:

// +kubebuilder:object:root=true
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:name="Ready",type=boolean,JSONPath=`.status.ready`
// +kubebuilder:printcolumn:name="URL",type=string,JSONPath=`.status.url`
type CertDashboard struct {
    metav1.TypeMeta   `json:",inline"`      // apiVersion + kind
    metav1.ObjectMeta `json:"metadata"`     // name, namespace, labels, etc.

    Spec   CertDashboardSpec   `json:"spec,omitempty"`    // DESIRED state (user writes this)
    Status CertDashboardStatus `json:"status,omitempty"`  // ACTUAL state (operator writes this)
}

type CertDashboardSpec struct {
    // +kubebuilder:default:=32600
    // +kubebuilder:validation:Minimum=30000
    // +kubebuilder:validation:Maximum=32767
    NodePort           int32    `json:"nodePort,omitempty"`
    YellowThresholdDays int     `json:"yellowThresholdDays,omitempty"`  // Days to yellow
    RedThresholdDays    int     `json:"redThresholdDays,omitempty"`    // Days to red
    Replicas           int32   `json:"replicas,omitempty"`
    Image              string  `json:"image,omitempty"`
    Namespaces         []string `json:"namespaces,omitempty"`         // Empty = all
}

type CertDashboardStatus struct {
    Ready      bool                `json:"ready"`
    URL        string              `json:"url,omitempty"`
    Conditions []metav1.Condition  `json:"conditions,omitempty"`
}
// +kubebuilder:... comments are code generators. Kubebuilder reads these comments and generates CRD YAML, validation rules, and printer columns. They're not regular comments — they're compile-time directives. Run make manifests to regenerate.

6. The Reconciler: Creating Child Resources

How the operator creates a Deployment:

func (r *CertDashboardReconciler) reconcileDeployment(
    ctx context.Context, cd *certsv1alpha1.CertDashboard,
) error {
    deploy := &appsv1.Deployment{
        ObjectMeta: metav1.ObjectMeta{
            Name:      cd.Name,
            Namespace: cd.Namespace,
        },
    }

    // CreateOrUpdate: fetch existing, or create new
    _, err := ctrl.CreateOrUpdate(ctx, r.Client, deploy, func() error {
        // This function is called INSIDE CreateOrUpdate.
        // If Deployment exists: it's fetched into `deploy`, then we mutate it.
        // If it doesn't exist: `deploy` is empty, we fill it in.

        deploy.Spec.Replicas = &cd.Spec.Replicas
        deploy.Spec.Template.Spec.Containers = []corev1.Container{{
            Name:    "dashboard",
            Image:   cd.Spec.Image,
            Command: []string{"/dashboard"},
            Env: []corev1.EnvVar{
                {Name: "DASHBOARD_PORT", Value: "8080"},
                {Name: "YELLOW_THRESHOLD_DAYS",
                 Value: strconv.Itoa(cd.Spec.YellowThresholdDays)},
            },
        }}

        // Set owner reference: when CertDashboard is deleted, this Deployment is garbage-collected
        return ctrl.SetControllerReference(cd, deploy, r.Scheme)
    })
    return err
}
ctrl.SetControllerReference is the magic that ties child resources to the parent CRD. When you delete the CertDashboard CR, Kubernetes automatically deletes all child resources (Deployment, Service, ServiceAccount) through garbage collection.

7. Certificate Watcher: Polling Kubernetes API

// The watcher polls Kubernetes every 15 seconds for cert-manager Certificates
func (w *CertWatcher) Start(ctx context.Context) {
    ticker := time.NewTicker(15 * time.Second)
    defer ticker.Stop()
    w.Refresh(ctx)                     // Initial fetch
    for {
        select {
        case <-ctx.Done():
            return                     // Context cancelled (shutdown)
        case <-ticker.C:
            w.Refresh(ctx)             // Poll every 15s
        }
    }
}
Go's select statement waits on multiple channels simultaneously. It blocks until one of them is ready. Here, it either receives a tick (every 15s) or a cancellation signal (shutdown). This is how Go handles concurrent event loops without callbacks or async/await.

How it reads certificates from the Kubernetes API:

// Uses the dynamic client (unstructured) because cert-manager types
// aren't in the standard K8s API — they're CRDs themselves
certResource := schema.GroupVersionResource{
    Group:    "cert-manager.io",

    Resource: "certificates",
}

    Namespace(ns).
    List(ctx, metav1.ListOptions{})
// Returns an unstructured.UnstructuredList — raw JSON-like map

8. Dashboard Server: Embedded Web UI

// Go 1.16+ can embed files directly into the binary
//go:embed templates/*
var templateFS embed.FS

// The HTML, CSS, and JavaScript are compiled INTO the Go binary.
// No external files needed at runtime. The entire dashboard is one binary.
mux.Handle("/", http.FileServer(http.FS(templateFS)))
embed.FS is how Go bundles static files into the binary. The //go:embed templates/* directive (note: no space after //) tells the compiler to include all files in templates/. At runtime, they're served from memory — no disk access.

9. Certificate Renewal: SSE Streaming

Renewal Flow (user clicks "Renew" button): 1. Browser: POST /api/renew {name: "bookstore-gateway-cert", namespace: "infra"} Authorization: Bearer <serviceaccount-token> 2. Server: validates token via Kubernetes TokenReview API → creates UUID streamId → starts goroutine for async renewal → returns {"streamId": "abc-123"} 3. Browser: GET /api/sse/abc-123 (Server-Sent Events — long-lived HTTP connection) 4. Goroutine: performs renewal steps, sends events through channel: event: status\ndata: deleting-secret → delete TLS secret event: status\ndata: waiting-issuing → cert-manager re-issues event: status\ndata: issued → new cert issued event: status\ndata: ready → cert mounted on gateway event: complete\ndata: {"success": true} → done 5. Browser: receives events in real-time, updates UI
// SSE handler: streams events to the browser
func (s *Server) handleSSE(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "text/event-stream")    // SSE content type
    w.Header().Set("Cache-Control", "no-cache")             // Don't cache events
    w.Header().Set("Connection", "keep-alive")              // Keep TCP open

    flusher, ok := w.(http.Flusher)  // Type assertion: can we flush?

    for {
        select {
        case event := <-eventChan:
            fmt.Fprintf(w, "event: %s\ndata: %s\n\n", event.Type, event.Data)
            flusher.Flush()                    // Push bytes to browser NOW
        case <-time.After(15 * time.Second):
            fmt.Fprint(w, ": keepalive\n\n")   // Prevent connection timeout
            flusher.Flush()
        case <-r.Context().Done():
            return                             // Client disconnected
        }
    }
}

10. Authentication: TokenReview API

// Validate a ServiceAccount token by asking the Kubernetes API server
func validateToken(ctx context.Context, token string) (bool, error) {
    review := &authv1.TokenReview{
        Spec: authv1.TokenReviewSpec{
            Token: token,   // The ServiceAccount JWT from the Authorization header
        },
    }

    // POST to /apis/authentication.k8s.io/v1/tokenreviews
    result, err := clientset.AuthenticationV1().TokenReviews().Create(ctx, review, metav1.CreateOptions{})

    return result.Status.Authenticated, err
    // Authenticated=true means the token is valid and belongs to a real ServiceAccount
}
TokenReview is Kubernetes' built-in token validation. Instead of implementing JWT verification ourselves, we ask the API server to validate the token. This is the official way to authenticate users/services in Kubernetes.

12. Admission Webhook Validation

// Runs BEFORE the CertDashboard CR is stored in etcd
func (v *CertDashboardValidator) validate(cd *CertDashboard) error {
    if cd.Spec.RedThresholdDays >= cd.Spec.YellowThresholdDays {
        return fmt.Errorf("redThresholdDays (%d) must be < yellowThresholdDays (%d)",
            cd.Spec.RedThresholdDays, cd.Spec.YellowThresholdDays)
    }
    if cd.Spec.NodePort < 30000 || cd.Spec.NodePort > 32767 {
        return fmt.Errorf("nodePort must be 30000-32767")
    }
    return nil  // Valid
}

13. Deployment: OLM & Lifecycle

Deployment order (scripts/cert-dashboard-up.sh): 1. Install OLM (Operator Lifecycle Manager) if not present 2. Build operator + dashboard Docker images 3. Load images into kind cluster 4. Install CRD: kubectl apply -f config/crd/ 5. Deploy operator: kubectl apply -f config/manager/ 6. Create CertDashboard CR: kubectl apply -f config/samples/ 7. 14. How to Build, Test & Run

Prerequisites

# Install Go, Docker, kind, kubectl
brew install go docker kind kubectl

Run Tests

cd cert-dashboard-operator

# Run all unit tests
make test

# Or run directly with go test
go test -v ./...

# Run specific controller tests
go test -v ./internal/controller/

# Run with coverage
make test-coverage

Build

# Build both binaries
make build

# Build Docker image
docker build -t bookstore/cert-dashboard-operator:latest .

# Generate CRD YAML from Go types (run after changing types.go)
make manifests

# Generate deepcopy methods (run after changing types.go)
make generate

Deploy to Cluster

# One command: build, load, deploy, verify
bash scripts/cert-dashboard-up.sh

# Or step by step:
make docker-build
kind load docker-image bookstore/cert-dashboard-operator:latest --name bookstore
make install        # Install CRDs
make deploy         # Deploy operator
kubectl apply -f config/samples/certs_v1alpha1_certdashboard.yaml

# Verify
kubectl get certdashboard               # Shows Ready=true, URL
kubectl get pods -n cert-dashboard      # Operator + dashboard pods
curl http://localhost:32600/api/certs   # Certificate list API
open http://localhost:32600             # Web dashboard

Run E2E Tests

# Playwright E2E tests (29 tests)
cd e2e
npx playwright test cert-dashboard.spec.ts

# Run specific test
npx playwright test cert-dashboard.spec.ts -g "Health endpoint returns ok"

Development Workflow

# 1. Edit CRD types (api/v1alpha1/certdashboard_types.go)
# 2. Regenerate manifests
make manifests generate

# 3. Edit controller (internal/controller/certdashboard_controller.go)
# 4. Run tests
go test -v ./internal/controller/

# 5. Build and redeploy
make docker-build
kind load docker-image bookstore/cert-dashboard-operator:latest --name bookstore
kubectl rollout restart deploy/cert-dashboard-operator -n cert-dashboard
