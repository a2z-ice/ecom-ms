1 AKS Subnet NSG: Default Allow vs Explicit Deny-All

Before -- Default Azure NSG

Default rule allows all VNet traffic in. Any compromised VM in any subnet can reach AKS pods.

# Azure default -- no custom NSG
# AllowVNetInBound (priority 65000)
# Allows ALL traffic within VNet
# Cannot be deleted
After -- Explicit Allow + Deny-All

Only App Gateway, AKS control plane, and intra-subnet traffic allowed. Everything else denied at priority 4000.

security_rule {
  name     = "DenyAllInbound"
  priority = 4000
  direction = "Inbound"
  access   = "Deny"
  protocol = "*"
  source_port_range       = "*"
  destination_port_range  = "*"
  source_address_prefix   = "*"
  destination_address_prefix = "*"
}

What Changed

Added a custom NSG with explicit allow rules at priorities 100-130 for App Gateway, AKS control plane, intra-subnet, and Azure LB probes. Added DenyAllInbound at priority 4000 to override the default AllowVNetInBound (65000).

Attack Prevented: Lateral movement from any compromised VM in any other subnet. Without this, a compromised Redis or jump box can port-scan and exploit AKS node services.

2 Database Subnet NSG: VNet-Wide vs AKS-Only

Before -- Default AllowVNetInBound

Any resource in the VNet (10.0.0.0/16) can connect to PostgreSQL on port 5432.

# No custom NSG on DB subnet
# Default AllowVNetInBound (65000):
# source = VirtualNetwork
# destination = VirtualNetwork
# action = Allow
After -- AKS-Only Access on 5432

Only AKS subnet (10.0.1.0/22) on port 5432. All other sources denied.

security_rule {
  name     = "AllowAKStoPostgres"
  priority = 100
  direction = "Inbound"
  access   = "Allow"
  protocol = "Tcp"
  source_port_range       = "*"
  destination_port_ranges = ["5432"]
  source_address_prefix   = "10.0.1.0/22"
  destination_address_prefix = "10.0.8.0/24"
}

security_rule {
  name     = "DenyAllInbound"
  priority = 4000
  access   = "Deny"
  protocol = "*"
  # ...
}

What Changed

Created a dedicated bookstore-db-nsg with a single allow rule: AKS subnet on port 5432 only. Added DenyAllInbound at priority 4000 to block everything else before the default VNet rule at 65000.

Attack Prevented: Any compromised VM, jump box, or other service in the VNet reaching the database. This is the single most impactful NSG rule -- databases are the #1 target in lateral movement attacks.

3 App Gateway NSG: Missing GatewayManager Rule

Before -- No GatewayManager Ports

Without GatewayManager rule, Azure cannot send health probes to App Gateway on ports 65200-65535. Gateway enters unhealthy state.

# Only allows 443/80 from internet
# MISSING: GatewayManager ports!
security_rule {
  name     = "AllowInternet"
  priority = 100
  source_address_prefix = "Internet"
  destination_port_ranges = ["80","443"]
  # ...
}
After -- GatewayManager at P100

GatewayManager at priority 100 allows Azure health probes. Internet HTTPS at 110. Deny-all at 4000.

security_rule {
  name     = "AllowGatewayManager"
  priority = 100
  direction = "Inbound"
  access   = "Allow"
  protocol = "Tcp"
  source_address_prefix      = "GatewayManager"
  destination_port_range     = "65200-65535"
  destination_address_prefix = "10.0.30.0/24"
}

security_rule {
  name     = "AllowInternetHTTPS"
  priority = 110
  source_address_prefix = "Internet"
  destination_port_ranges = ["80","443"]
  # ...
}

What Changed

Added the GatewayManager service tag rule at priority 100, allowing Azure infrastructure health probes on ports 65200-65535. Without this rule, App Gateway v2 cannot start and all traffic routing stops.

Attack Prevented: Complete service outage. This is not a security attack but an availability failure -- without the GatewayManager rule, the App Gateway enters a perpetual unhealthy state and cannot route any user traffic.

4 No Private Endpoints vs All PaaS Private

Before -- Public Endpoints

All PaaS services have public endpoints. Traffic routes over the internet backbone.

# Default for all Azure PaaS
public_network_access_enabled = true
# No private endpoints defined
After -- All Private Endpoints in VNet

Every PaaS service uses a Private Endpoint. Public access disabled. Traffic never leaves the VNet.

public_network_access_enabled = false

resource "azurerm_private_endpoint" "pg" {
  name     = "pe-bookstore-pg"
  subnet_id = azurerm_subnet.pe.id

  private_service_connection {
    name = "pg-connection"
    private_connection_resource_id =
      azurerm_postgresql_flexible_server
        .main.id
    subresource_names = ["postgresqlServer"]
    is_manual_connection = false
  }
}

What Changed

Created Private Endpoints for all 6 PaaS services (PostgreSQL, Redis, Event Hubs, Key Vault, ACR, Blob Storage). Set public_network_access_enabled = false on each. Added Private DNS Zones for name resolution within the VNet.

Attack Prevented: Data exfiltration via public endpoints, man-in-the-middle on Azure backbone, and exposure to internet-wide vulnerability scans. With private endpoints, services are invisible to the internet.

5 Redis Public vs Private Endpoint + TLS

Before -- Public Redis, No TLS 
resource "azurerm_redis_cache" "main" {
  sku_name = "Premium"
  family   = "P"
  capacity = 1
  # public_network_access_enabled
  # defaults to true
  minimum_tls_version = "1.0"  # weak!
  enable_non_ssl_port = true   # port 6379
}
After -- Private Endpoint + TLS 1.2 
resource "azurerm_redis_cache" "main" {
  sku_name = "Standard"
  family   = "C"
  capacity = 1

  public_network_access_enabled = false
  minimum_tls_version = "1.2"
  enable_non_ssl_port = false # TLS only
}

resource "azurerm_private_endpoint" "redis" {
  subnet_id = azurerm_subnet.pe.id
  private_service_connection {
    subresource_names = ["redisCache"]
    # ...
  }
}

What Changed

Disabled the non-SSL port (6379), enforced TLS 1.2 minimum, disabled public network access, and added a Private Endpoint. Redis is now accessible only from within the VNet on port 6380 with encryption.

Attack Prevented: Redis authentication credentials sent in plaintext over the network (AUTH command). Without TLS, any network sniffer between AKS and Redis captures the password and gains full access to session data, CSRF tokens, and rate-limit state.

6 Key Vault Access Policies vs RBAC

Before -- Legacy Access Policies 
resource "azurerm_key_vault" "main" {
  # Legacy access policy model
  enable_rbac_authorization = false

  access_policy {
    tenant_id = data.azurerm_client_config
                  .current.tenant_id
    object_id = azurerm_user_assigned_identity
                  .aks.principal_id
    secret_permissions = [
      "Get","List","Set","Delete"
    ]  # All-or-nothing!
  }
}
After -- Azure RBAC (Granular) 
resource "azurerm_key_vault" "main" {
  enable_rbac_authorization = true
  # No access_policy blocks needed
}

# AKS can only READ secrets
resource "azurerm_role_assignment" "aks_kv" {
  scope        = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets User"
  principal_id = azurerm_user_assigned_identity
                   .aks.principal_id
}

# DevOps can read + write secrets
resource "azurerm_role_assignment" "devops_kv" {
  scope        = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets Officer"
  principal_id = var.devops_sp_object_id
}

What Changed

Switched from legacy access policies (enable_rbac_authorization = false) to Azure RBAC (= true). Assigned Key Vault Secrets User (read-only) to AKS and Key Vault Secrets Officer (read/write) to the DevOps pipeline. Removed all access_policy blocks.

Attack Prevented: Privilege escalation via over-permissioned access policies. With access policies, AKS could delete or overwrite secrets. With RBAC, AKS can only read them -- even if AKS is compromised, the attacker cannot modify secrets.

7 No Azure Policy vs 5 Gatekeeper Policies

Before -- Unguarded Cluster

Any container image, any privilege level, root user, writable filesystem -- all allowed by default.

# No Azure Policy addon enabled
# No OPA/Gatekeeper constraints
# Any kubectl apply succeeds
After -- 5 Azure Policy Constraints

Azure Policy addon with 5 built-in policies in Deny mode. Bad deployments rejected at admission.

resource "azurerm_kubernetes_cluster" "main" {
  azure_policy_enabled = true
}

# 5 policies assigned to AKS RG:
# 1. No privileged containers
# 2. Container images from ACR only
# 3. No privilege escalation
# 4. No root user (must runAsNonRoot)
# 5. Read-only root filesystem

resource "azurerm_resource_group_policy_assignment" "no_privileged" {
  name                 = "no-privileged-containers"
  resource_group_id    = azurerm_resource_group.main.id
  policy_definition_id = "/providers/Microsoft.Authorization/policyDefinitions/95edb821-ddaf-4404-9732-666045e056b4"
  parameters = jsonencode({
    effect = { value = "Deny" }
  })
}

What Changed

Enabled the Azure Policy addon on AKS and assigned 5 built-in policies in Deny mode. Any deployment that violates these policies is rejected at the Kubernetes admission webhook level -- it never starts running.

Attack Prevented: Container escape via privileged mode, supply chain attacks via untrusted container images (e.g., cryptominer from Docker Hub), and persistence via writable root filesystem modifications.

8 No VNet Flow Logs vs NSG Flow Logs

Before -- No Flow Logs

NSGs are active but no logging. Cannot answer "who connected to the database last night?" or detect anomalous traffic patterns.

# No azurerm_network_watcher_flow_log
# No storage account for flow logs
# No Log Analytics integration
# Blind to all network activity
After -- NSG Flow Logs v2

NSG Flow Logs v2 enabled on all 4 subnet NSGs. Data sent to Log Analytics workspace with 30-day retention.

resource "azurerm_network_watcher_flow_log" "aks" {
  network_watcher_name = azurerm_network_watcher
                          .main.name
  resource_group_name  = azurerm_resource_group
                          .main.name
  network_security_group_id =
    azurerm_network_security_group.aks.id
  storage_account_id =
    azurerm_storage_account.flow_logs.id
  enabled = true
  version = 2

  retention_policy {
    enabled = true
    days    = 30
  }

  traffic_analytics {
    enabled               = true
    workspace_id          = azurerm_log_analytics_workspace
                             .main.workspace_id
    workspace_region      = var.location
    workspace_resource_id = azurerm_log_analytics_workspace
                             .main.id
    interval_in_minutes   = 10
  }
}

What Changed

Enabled NSG Flow Logs v2 with Traffic Analytics on all 4 subnet NSGs. Logs stored in a dedicated storage account and forwarded to Log Analytics workspace with 30-day retention and 10-minute analytics intervals.

Attack Prevented: Without flow logs, a breach goes undetected. Flow logs enable: forensic investigation of lateral movement, detection of data exfiltration patterns, compliance auditing (SOC2, PCI-DSS), and alerting on anomalous traffic volumes.

9 No Disk Encryption vs Customer-Managed Keys

Before -- Azure-Managed Keys (Default)

All disks encrypted with Azure platform-managed keys. You cannot rotate, audit, or revoke the key.

# Default -- no explicit encryption config
# Azure auto-encrypts with SSE
# Key managed by Microsoft
# No audit trail for key usage
After -- Customer-Managed Key in Key Vault 
resource "azurerm_key_vault_key" "disk_cmk" {
  name         = "bookstore-disk-cmk"
  key_vault_id = azurerm_key_vault.main.id
  key_type     = "RSA"
  key_size     = 2048
  key_opts     = ["wrapKey","unwrapKey"]

  rotation_policy {
    automatic {
      time_before_expiry = "P30D"
    }
    expire_after = "P365D"
  }
}

resource "azurerm_disk_encryption_set" "main" {
  name     = "bookstore-des"
  key_vault_key_id = azurerm_key_vault_key
                      .disk_cmk.id
  identity { type = "SystemAssigned" }
}

# AKS: attach to node pools
resource "azurerm_kubernetes_cluster" "main" {
  disk_encryption_set_id =
    azurerm_disk_encryption_set.main.id
}

What Changed

Created an RSA-2048 key in Key Vault with automatic 365-day rotation. Created a Disk Encryption Set referencing that key. Assigned it to AKS node pools and PostgreSQL servers. All disk encryption now uses your key, with full audit trail in Key Vault logs.

Attack Prevented: Regulatory non-compliance (HIPAA, PCI-DSS require CMK for data-at-rest). CMK also enables "crypto-shredding" -- revoking the key makes all data permanently inaccessible, useful for data destruction on contract termination.

10 Intel On-Demand vs AMD Spot Instances

Before -- 5x D4s_v3 On-Demand 
default_node_pool {
  name       = "default"
  vm_size    = "Standard_D4s_v3"
  node_count = 5
  # priority defaults to Regular
  # Intel, 4 vCPU, 16 GB each
  # $144/node/mo x 5 = $720/mo
}
After -- 2x System + 3x AMD Spot 
default_node_pool {
  name       = "system"
  vm_size    = "Standard_D2s_v3"
  node_count = 2
  # $72/node/mo x 2 = $144/mo
}

resource "azurerm_kubernetes_cluster_node_pool" "app" {
  name       = "appspot"
  vm_size    = "Standard_D2as_v5" # AMD
  priority   = "Spot"
  eviction_policy = "Delete"
  spot_max_price  = -1
  node_count = 3
  node_taints = ["kubernetes.azure.com/scalesetpriority=spot:NoSchedule"]
  # ~$8.70/node/mo x 3 = $26/mo
}

What Changed

Split the single node pool into two: a 2-node system pool (Standard_D2s_v3, on-demand) for kube-system, and a 3-node app pool (Standard_D2as_v5 AMD, Spot) for application workloads. Spot nodes use the Delete eviction policy and tolerations.

Cost Saved: $720/mo reduced to $170/mo = 76% savings ($6,600/year). Spot instances are up to 90% cheaper. AMD D2as_v5 offers identical performance to Intel D2s_v3 at a lower base price.

11 PostgreSQL GP vs Burstable (Dev)

Before -- General Purpose D4s_v3 x4 
resource "azurerm_postgresql_flexible_server" "ecom" {
  sku_name = "GP_Standard_D4s_v3"
  # 4 vCPU, 16 GB RAM
  # $272/mo each x 4 = $1,088/mo
}
After -- Burstable B1ms (Dev) / GP D2s (Prod) 
# DEV environment
resource "azurerm_postgresql_flexible_server" "ecom" {
  sku_name = "B_Standard_B1ms"
  # 1 vCPU, 2 GB RAM -- burstable
  # $13/mo each x 4 = $52/mo
}

# PROD environment (right-sized)
resource "azurerm_postgresql_flexible_server" "ecom" {
  sku_name = "GP_Standard_D2s_v3"
  # 2 vCPU, 8 GB RAM
  # $136/mo each x 4 = $544/mo
}

What Changed

Dev environments use Burstable B1ms ($13/mo) instead of General Purpose D4s_v3 ($272/mo). Production is right-sized to D2s_v3 ($136/mo) -- half the original cost with 2 vCPU and 8 GB RAM, which is sufficient for this workload.

Cost Saved: Dev: $1,088 to $52/mo (95% savings). Prod: $1,088 to $544/mo (50% savings). Burstable SKUs are ideal for dev/test with sporadic usage. GP D2s is right-sized for production based on actual CPU/memory utilization.

12 Redis Premium vs Standard

Before -- Premium P1 ($224/mo) 
resource "azurerm_redis_cache" "main" {
  sku_name = "Premium"
  family   = "P"
  capacity = 1  # 6 GB
  # Zone-redundant, clustering
  # Overkill: we use <100 MB
}
After -- Standard C0/C1 
# DEV
resource "azurerm_redis_cache" "main" {
  sku_name = "Standard"
  family   = "C"
  capacity = 0  # 250 MB, $32/mo
}

# PROD
resource "azurerm_redis_cache" "main" {
  sku_name = "Standard"
  family   = "C"
  capacity = 1  # 1 GB, $64/mo
}

What Changed

Downgraded from Premium P1 (6 GB, zone-redundant, clustering) to Standard C0/C1. The bookstore uses Redis only for sessions, CSRF tokens, and rate limiting -- under 100 MB total. Standard tier provides replication without the Premium overhead.

Cost Saved: Dev: $224 to $32/mo (86% savings). Prod: $224 to $64/mo (71% savings). Premium features (zone-redundancy, clustering, VNet injection) are unnecessary for a cache holding less than 100 MB of session data.

13 Event Hubs Premium vs Standard

Before -- Premium 2 PU 
resource "azurerm_eventhub_namespace" "main" {
  sku      = "Premium"
  capacity = 2  # 2 Processing Units
  # $731/PU/mo x 2 = $1,462/mo
  # We produce ~10 events/min
}
After -- Standard 1 TU + Auto-Inflate 
resource "azurerm_eventhub_namespace" "main" {
  sku      = "Standard"
  capacity = 1  # 1 Throughput Unit

  auto_inflate_enabled     = true
  maximum_throughput_units = 5
  # $22/TU/mo base
  # Auto-scales up to 5 TU if needed
  # Scales back down automatically
}

What Changed

Downgraded from Premium (2 Processing Units at $731/PU) to Standard (1 Throughput Unit at $22/TU) with auto-inflate enabled up to 5 TU. The bookstore produces approximately 10 events per minute -- far below the 1 MB/s capacity of a single Standard TU.

Cost Saved: $1,462 to $22/mo = 98% reduction ($17,280/year). Premium Processing Units are designed for workloads requiring dedicated compute, zone-redundancy, and 100+ MB/s throughput -- massive overkill for CDC event streaming.

14 No Stop/Start vs Automated Scheduling

Before -- Dev Runs 24/7 
# Dev environment runs 24/7/365
# Nobody uses it nights/weekends
# Paying for 168 hrs/week
# Only used ~40 hrs/week
After -- Automated Start/Stop Schedule 
# Azure Logic App triggers daily
# 8:50 AM: Start AKS + PostgreSQL
az aks start \
  --resource-group bookstore-dev \
  --name bookstore-aks-dev

az postgres flexible-server start \
  --resource-group bookstore-dev \
  --name bookstore-pg-dev

# 5:10 PM: Stop AKS + PostgreSQL
az aks stop ...
az postgres flexible-server stop ...

What Changed

Created an Azure Logic App (or Automation Runbook) that stops AKS and PostgreSQL at 5:10 PM and starts them at 8:50 AM, Monday through Friday. Weekends are fully stopped. Dev environment runs 40 hours/week instead of 168.

Cost Saved: ~60% reduction on dev environment compute costs. At 24% uptime (40h/168h), you pay only for the hours the environment is running. AKS stop deallocates all nodes; PostgreSQL stop deallocates the compute (storage still charged).

15 No Resource Tags vs Full Cost Attribution

Before -- Untagged Resources 
# No tags on any resource
resource "azurerm_kubernetes_cluster" "main" {
  name     = "bookstore-aks"
  location = var.location
  # tags = {} -- missing!
}

resource "azurerm_postgresql_flexible_server" "ecom" {
  # tags = {} -- missing!
}
After -- Full Tag Taxonomy 
locals {
  common_tags = {
    Environment = var.environment
    Project     = "bookstore"
    Team        = "platform"
    CostCenter  = "eng-42"
    ManagedBy   = "terraform"
  }
}

resource "azurerm_kubernetes_cluster" "main" {
  tags = merge(local.common_tags, {
    Service = "aks"
  })
}

resource "azurerm_postgresql_flexible_server" "ecom" {
  tags = merge(local.common_tags, {
    Service = "ecom-db"
  })
}

# Azure Policy: deny untagged resources
resource "azurerm_resource_group_policy_assignment" "require_tags" {
  policy_definition_id = "/providers/Microsoft.Authorization/policyDefinitions/871b6d14-10aa-478d-b466-ef6e8766b9a0"
  parameters = jsonencode({
    tagName = { value = "CostCenter" }
  })
}

What Changed

Added a common_tags local with 5 mandatory tags (Environment, Project, Team, CostCenter, ManagedBy) applied to every resource. Added a per-resource Service tag. Enforced tagging via Azure Policy that denies resource creation without a CostCenter tag.

Cost Saved: Tags themselves cost nothing. But without them, you cannot: identify orphan resources ($200+/mo wasted), do per-service cost breakdown, implement team chargebacks, or set per-service budget alerts. Typical savings from tag-driven cleanup: 10-15% of total spend.
