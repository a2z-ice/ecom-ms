2. Host Configuration

The platform uses custom hostnames for gateway routing. Add these entries to your /etc/hosts file:

* Edit /etc/hosts

sudo sh -c 'cat >> /etc/hosts <<EOF
127.0.0.1  idp.keycloak.net
127.0.0.1  myecom.net
127.0.0.1  api.service.net
EOF'

This maps all three hostnames to localhost. The Istio Gateway uses SNI to route requests to the correct backend service.

Verify: Run ping -c1 myecom.net — it should resolve to 127.0.0.1.

3. One-Command Bootstrap

1 Clone and start

# Clone the repository
git clone https://github.com/a2z-ice/ecom-ms.git
cd ecom-ms

# Bootstrap everything (auto-detects scenario)
bash scripts/up.sh
~25 minutes on first run (builds images, deploys infra, imports realm)

That's it. The up.sh script is the single entry point that handles every scenario:

ScenarioWhat happensTime
No clusterFull bootstrap from scratch~25 min
Healthy clusterVerify connectors + smoke test only~2 min
Degraded clusterRecovery (restart ztunnel, pods, Flink)~5 min
Non-interactive mode (CI/automation): bash scripts/up.sh --yes skips all confirmation prompts.
Force fresh rebuild: bash scripts/up.sh --fresh deletes the existing cluster and rebuilds from scratch. Add --data to also wipe persisted data.

4. What Happens Under the Hood

The bootstrap pipeline executes these stages in order. Docker image builds run in parallel with infrastructure deployment for speed.

Create kind cluster~2 min
3-node kind cluster (1 control-plane + 2 workers) with NodePort mappings for all services. Creates ./data/ directories for persistence.
Install Istio Ambient Mesh 1.29.1~1 min
Sidecar-free mTLS mesh via ztunnel. All namespaces labeled for ambient mode. Kubernetes Gateway API installed.
Deploy databases (CloudNativePG)~3 min
4 PostgreSQL HA clusters (ecom-db, inventory-db, analytics-db, keycloak-db) with 2 instances each (primary + standby). Waited in parallel.
Deploy infrastructure services~4 min
Redis, Kafka (KRaft), Schema Registry, Debezium Server (2 pods), PgAdmin, cert-manager with self-signed CA, and TLS certificates.
Build & load Docker imagesparallel
5 images built concurrently: ecom-service (Spring Boot), inventory-service (FastAPI), ui-service (React+Nginx), csrf-service (Go), flink (CDC pipeline).
Import Keycloak realm~1 min
BookStore realm with OIDC clients, PKCE-enabled flows, users (user1, admin1), and role mappings (customer, admin).
Deploy application services~2 min
ecom-service + inventory-service + ui-service + csrf-service. Init containers run DB migrations (Liquibase, Alembic) before app starts.
Apply security policies~30 sec
NetworkPolicies, PeerAuthentication (STRICT mTLS), RequestAuthentication (JWT), AuthorizationPolicies, CSRF ext_authz registration.
Deploy analytics & observability~3 min
Flink SQL pipeline, Superset dashboards, Prometheus, Grafana, Loki, Tempo, OTel Collector, Kiali.
Smoke test & CA trust~2 min
38 verification checks (pods, HTTP routes, Kafka lag, Debezium health, TLS certs). Optional CA install to macOS Keychain for browser trust.

10. Cluster Lifecycle

Start / Resume

bash scripts/up.sh           # smart start (auto-detects scenario)
bash scripts/up.sh --fresh   # force full rebuild (keeps data)
bash scripts/up.sh --fresh --data  # full rebuild + wipe data

Stop / Teardown

bash scripts/down.sh          # delete cluster (keeps ./data/)
bash scripts/down.sh --data   # delete cluster + wipe all data
bash scripts/down.sh --images # delete cluster + remove Docker images
bash scripts/down.sh --all    # delete everything

Health Check

bash scripts/smoke-test.sh    # 38-point full stack verification
bash scripts/verify-routes.sh # quick HTTP route check
bash scripts/sanity-test.sh   # comprehensive cluster health

Backup & Restore

bash scripts/backup.sh                # timestamped backup of all DBs + Kafka + Keycloak
bash scripts/restore.sh <timestamp>  # restore from backup

11. Recovery After Docker Restart

When Docker Desktop restarts, the kind cluster survives but Istio's HBONE mesh and Kafka offsets break. Recovery is automatic:

! After Docker restart

bash scripts/up.sh

The script auto-detects the degraded state and runs recovery: ztunnel restart → pod rolling restart in dependency order → Flink resubmit → smoke test.

~5-8 minutes
What breaks and why:
1. ztunnel HBONE — ztunnel daemonset loses L4 tunnel state; must restart first
2. Pod registration — existing pods lose HBONE registration; must restart after ztunnel
3. Kafka offsets — Debezium offset topics may be lost; re-snapshots if needed
4. Flink jobs — SQL streaming jobs lost; automatically resubmitted

12. Troubleshooting

Pod stuck in Pending

Cause: Insufficient CPU/memory in Docker.
Fix: Increase Docker Desktop memory to 8+ GB (Settings → Resources). Then: kubectl delete pod -n <ns> <pod>

HTTP endpoints return 502/503

Cause: Service crashed or migration failed.
Diagnose: kubectl logs -n ecom deploy/ecom-service --tail=50
Fix: kubectl rollout restart deploy/ecom-service -n ecom

Browser shows "connection not private"

Cause: Self-signed CA not trusted.
Fix: bash scripts/trust-ca.sh --install — then hard-refresh the browser (Cmd+Shift+R).

Keycloak not reachable at idp.keycloak.net:30000

Cause: Missing /etc/hosts entry.
Verify: grep keycloak /etc/hosts
Fix: Add 127.0.0.1 idp.keycloak.net to /etc/hosts.

Kafka consumer lag stuck

Cause: inventory-service not consuming.
Diagnose: kubectl logs -n inventory deploy/inventory-service | tail -20
Fix: kubectl rollout restart deploy/inventory-service -n inventory

Flink SQL runner job not completing

Cause: SQL Gateway not ready or analytics DB not initialized.
Fix:
kubectl delete job flink-sql-runner -n analytics
kubectl apply -f infra/flink/flink-sql-runner.yaml
