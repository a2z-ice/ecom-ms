What is CloudNativePG?

CloudNativePG (CNPG) is a Kubernetes operator designed to manage the full lifecycle of PostgreSQL database clusters inside Kubernetes. It is the first open-source PostgreSQL operator to join the CNCF Sandbox, and it is the industry-standard approach for running production PostgreSQL on Kubernetes.

Why CloudNativePG Over Alternatives?

CloudNativePG (Chosen)

  • Kubernetes-native operator with CRD-based management
  • Automatic failover via built-in controller (no external tools)
  • Auto-creates -rw, -ro, -r Services for read/write splitting
  • Supports wal_level=logical (Debezium CDC compatible)
  • Single YAML CRD per database cluster
  • PVC-based storage with dynamic provisioning
  • Built-in backup/restore with Barman
  • CNCF Sandbox project (community-backed)

Alternatives Considered

  • Patroni — Requires etcd/Consul for leader election, more operational overhead
  • Manual replication — No auto-failover, requires custom scripts
  • CrunchyData PGO — Heavier footprint, different CRD model
  • Zalando Postgres Operator — Patroni-based internally, needs sidecars

CloudNativePG was chosen for its simplicity, native Kubernetes integration, and alignment with our existing operator pattern (cert-manager, cert-dashboard-operator).

How CNPG Works

The CNPG operator watches for Cluster custom resources and reconciles the desired state:

  1. Cluster CR Applied — You declare a Cluster resource specifying instances, storage, PostgreSQL parameters, and bootstrap configuration
  2. Operator Creates Pods — CNPG creates the specified number of PostgreSQL pods. The first pod becomes the primary (read-write), and additional pods become standbys (read-only replicas)
  3. Services Auto-Created — Three Kubernetes Services are automatically created:
    • <cluster>-rw — Routes to the current primary (for writes)
    • <cluster>-ro — Routes to standby replicas (for reads)
    • <cluster>-r — Routes to all instances (for any read)
  4. Streaming Replication — CNPG configures PostgreSQL native streaming replication between primary and standbys. WAL records are shipped continuously
  5. Health Monitoring — The instance manager (sidecar process) on each pod reports health to the operator every ~10 seconds
  6. Automatic Failover — When the primary pod fails, CNPG promotes the most up-to-date standby to primary and updates the -rw Service endpoint
# Example CNPG Cluster CR
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: ecom-db
  namespace: ecom
spec:
  instances: 2                    # 1 primary + 1 standby

How Debezium CDC Works with PostgreSQL WAL

Write-Ahead Log (WAL) Fundamentals

PostgreSQL uses a Write-Ahead Log (WAL) to ensure data durability. Before any change is written to the actual data files (tables, indexes), it is first recorded in the WAL. This guarantees that even if PostgreSQL crashes mid-transaction, the database can recover to a consistent state by replaying the WAL.

WAL is the Source of Truth

Every INSERT, UPDATE, and DELETE operation generates WAL records. These records contain the exact byte-level changes to the data pages. PostgreSQL uses WAL for three critical purposes: crash recovery, streaming replication (to standbys), and logical replication (to external consumers like Debezium).

Physical vs Logical WAL

AspectPhysical WAL (default)Logical WAL (wal_level=logical)
ContentRaw byte-level page diffsLogical row changes (INSERT/UPDATE/DELETE with column values)
Use caseStreaming replication, crash recoveryChange Data Capture, logical replication
ConsumersStandby PostgreSQL instancesDebezium, pg_logical, custom consumers
OverheadMinimalSlightly larger WAL volume (~10-20% more)
PostgreSQL settingwal_level=replicawal_level=logical

wal_level=logical is a superset of replica — it includes all physical WAL data PLUS decoded logical change records. This means a single WAL stream supports both streaming replication (to standbys) and logical replication (to Debezium) simultaneously.

Debezium's CDC Pipeline from WAL

Debezium captures changes from PostgreSQL through the following mechanism:

  1. Logical Replication Slot — Debezium creates a replication slot (debezium_ecom_slot) on the primary. This slot tells PostgreSQL to retain WAL segments that haven't been consumed yet, preventing them from being recycled
  2. Publication — A PostgreSQL PUBLICATION defines which tables are included in the change stream. We pre-create debezium_ecom_pub FOR ALL TABLES as the superuser during database bootstrap
  3. pgoutput Plugin — Debezium uses PostgreSQL's built-in pgoutput logical decoding plugin to decode WAL records into a structured format (table name, operation type, old/new row values)
  4. Streaming Protocol — Debezium connects to PostgreSQL via the streaming replication protocol (port 5432, same as regular connections) and receives a continuous stream of decoded changes
  5. Kafka Sink — Each change is transformed into a Kafka message and published to the appropriate topic (ecom-connector.public.orders, etc.)
  6. Offset Storage — Debezium records the current WAL position (LSN — Log Sequence Number) in a Kafka topic (debezium.ecom.offsets) so it can resume from the exact position after a restart
# Debezium WAL reading flow (simplified)
PostgreSQL Primary
  │
  ├── Transaction: INSERT INTO orders (id, user_id, total) VALUES (1, 'u1', 39.98)
  │
  ├── WAL Record Generated:
  │     LSN: 0/1A3B4C0
  │     Type: INSERT
  │     Table: public.orders
  │     New Row: {id: 1, user_id: 'u1', total: 39.98}
  │
  ├── Logical Replication Slot: debezium_ecom_slot
  │     Confirmed flush LSN: 0/1A3B4B0  (previous position)
  │     Slot retains WAL from this LSN onward
  │
  ├── pgoutput Plugin decodes → structured change event
  │
  └── Debezium Server receives:
        {
          "source": {"lsn": 27229376, "table": "orders"},
          "op": "c",  // c=create, u=update, d=delete
          "after": {"id": 1, "user_id": "u1", "total": 39.98}
        }
        │
        └── Published to Kafka topic: ecom-connector.public.orders
            Offset stored in: debezium.ecom.offsets (Kafka, compacted)

ExternalName Service Pattern

The key to achieving zero application code changes is the ExternalName Service pattern. Applications continue connecting to ecom-db — the same hostname they always used — which now resolves through a chain:

Application JDBC URL: jdbc:postgresql://ecom-db:5432/ecomdb
                              │
                              ▼
            ExternalName Service: ecom-db
            (type: ExternalName)
            externalName: ecom-db-rw.ecom.svc.cluster.local
                              │
                              ▼
            CNPG -rw Service: ecom-db-rw
            (Endpoints: current primary pod IP)
                              │
                              ▼

Zero Configuration Changes

No application code, Debezium config, Flink SQL, Keycloak, or any other component needed modification to their database connection strings. The ExternalName → -rw Service chain handles all routing transparently, including during failover.

Why 5 Kubernetes Services Per Database?

When you run kubectl get svc -n inventory, you'll see 5 services for a single database cluster like inventory-db. This can be confusing at first — here's exactly what each one is, who creates it, and where it's used.

When CNPG processes a Cluster CR named inventory-db, it automatically creates 4 services: -rw, -ro, -r, and -any. The 5th service (inventory-db itself, the ExternalName alias) is manually defined in our CNPG cluster YAML files to maintain backward compatibility with existing app connection strings.

Automatic Failover Sequence

This animated diagram shows the complete 8-phase failover sequence, from primary failure to full recovery.

Animated: 8-phase automatic failover — failure detection, promotion, service update, Debezium reconnect, offset resume

Failover Timeline

1

Normal Operation

App connects via ecom-db (ExternalName) → ecom-db-rw → Primary pod. Debezium reads WAL via logical replication slot. Streaming replication ships WAL to standby.

2

Primary Pod Fails

Primary pod crashes (OOMKill, node failure, etc.). App JDBC connections break immediately (TCP RST). Debezium loses its TCP connection. Standby continues running with latest replicated WAL data.

3

CNPG Operator Detects Failure

Instance manager reports unhealthy. Reconcile loop (~10s interval) detects pod not ready. CNPG evaluates cluster topology: 1 standby available for promotion. Decision: promote most up-to-date standby based on WAL position.

4

Promote Standby to Primary

pg_ctl promote on standby pod. Standby exits recovery mode, begins accepting writes. Timeline ID increments. CNPG updates instance labels: cnpg.io/instanceRole=primary.

5

Update -rw Service Endpoint

CNPG patches ecom-db-rw Service: Endpoints now point to new primary pod IP. ExternalName alias (ecom-db) resolves to ecom-db-rw → new primary. Zero config change needed anywhere.

6

Application Reconnects

HikariCP / connection pool detects broken connections, creates new ones. DNS resolves ecom-db → ecom-db-rw → new primary pod IP. Application resumes normal operation. Users experience ~5-15 second interruption.

7

Debezium Resumes CDC

Debezium reconnects to same hostname → reaches new primary via ExternalName. Creates new logical replication slot on new primary (old slot was on old primary). Reads last committed offset from Kafka topic debezium.ecom.offsets. Resumes from that LSN position — if WAL is available (wal_keep_size=256MB ensures this).

8

Fully Recovered

CNPG recreates old primary pod as new standby. Streaming replication resumes. Cluster returns to 2 healthy instances (1 primary + 1 standby). Total downtime: ~5-15 seconds. Zero data loss (synchronous replication).

KafkaOffsetBackingStore: The Key to Debezium Resilience

Previously, Debezium used FileOffsetBackingStore with emptyDir storage — meaning offsets were lost on every pod restart, triggering a full re-snapshot. By switching to KafkaOffsetBackingStore, offsets are stored in a compacted Kafka topic (debezium.ecom.offsets) that survives both Debezium restarts AND database failovers. After failover, Debezium reads the last committed LSN from Kafka and resumes exactly where it left off.

Why CNPG is Excluded from Istio Ambient Mesh

This was one of the most critical architectural decisions in the HA implementation. CloudNativePG manages its own TLS for streaming replication between primary and standby pods, and this conflicts with Istio Ambient Mesh's ztunnel HBONE tunneling.

The Conflict: Double TLS Encryption

Istio Ambient Mesh uses ztunnel to transparently intercept all TCP traffic and wrap it in HBONE (HTTP/2-based) tunnels with mTLS encryption. When a pod sends traffic to another pod, ztunnel on the source node encrypts it, and ztunnel on the destination node decrypts it — all transparently to the application.

CNPG, however, manages its own TLS certificates for PostgreSQL streaming replication. The primary and standby pods communicate using PostgreSQL's native TLS, which CNPG configures automatically (it generates its own CA and certificates).

When both are active, the traffic path becomes:

# With Istio Ambient + CNPG TLS (BROKEN):
Primary Pod
  └── PostgreSQL starts TLS handshake (CNPG-managed cert)
        └── ztunnel intercepts → wraps in HBONE mTLS tunnel
              └── ztunnel on destination decrypts HBONE
                    └── Delivers to Standby Pod
                          └── PostgreSQL tries to complete TLS handshake
                                └── FAILS: TLS state corrupted by double encryption

# Without Istio Ambient (WORKING):
Primary Pod
  └── PostgreSQL starts TLS handshake (CNPG-managed cert)
        └── Direct TCP to Standby Pod
              └── PostgreSQL completes TLS handshake
                    └── Streaming replication active
Symptom: TLS Handshake Timeout

When CNPG pods are enrolled in the Istio Ambient mesh, standby pods fail to connect to the primary with SSL error: certificate verify failed or TLS handshake timeouts. The streaming replication never establishes, and the standby pod continuously restarts.

The Solution: Mesh Exclusion

All CNPG database pods are excluded from the Istio Ambient mesh using the ambient.istio.io/redirection: disabled annotation. This tells ztunnel to not intercept traffic for these specific pods:

# In every CNPG Cluster CR:
spec:
  inheritedMetadata:
    annotations:
      ambient.istio.io/redirection: disabled

This annotation is set at the Cluster level via inheritedMetadata, which ensures it is automatically applied to all pods created by CNPG (both primary and standby).

Why cnpg-system Namespace is NOT in the Mesh

The CNPG operator itself (running in cnpg-system namespace) also cannot be enrolled in the Istio Ambient mesh. The reason is different from the database pods:

Webhook Accessibility

CNPG registers a ValidatingWebhookConfiguration with the Kubernetes API server. When you apply a Cluster CR, the kube-apiserver calls the CNPG webhook to validate it. The kube-apiserver is outside the mesh — it does not have an ztunnel identity. If the CNPG webhook pod is enrolled in STRICT mTLS mode, the kube-apiserver's plain TCP connection is rejected, and all CNPG Cluster operations fail with EOF errors.

# cnpg-system PeerAuthentication: PERMISSIVE (not STRICT)
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: cnpg-mtls
  namespace: cnpg-system
spec:
  mtls:
    mode: PERMISSIVE  # Must allow kube-apiserver plaintext webhook calls

Security Impact Assessment

AspectImpactMitigation
Inter-pod replication trafficNot encrypted by IstioCNPG manages its own TLS — traffic IS encrypted, just not by Istio
App → Database trafficNot encrypted by IstioCNPG enables TLS by default on all connections (server cert auto-generated)

Defense in Depth

Even though CNPG pods are excluded from the Istio mesh, they are NOT unprotected. CNPG manages its own TLS for all PostgreSQL connections (both client and replication), and Kubernetes NetworkPolicies restrict which pods can connect. The only difference is that traffic encryption is managed by CNPG instead of Istio ztunnel.

KafkaOffsetBackingStore Configuration

The critical change for HA support is switching Debezium's offset storage from file-based to Kafka-based:

# Debezium Server application.properties (HA-ready)
# ─── Offset Storage (Kafka) ───

# KafkaOffsetBackingStore requires these in embedded worker config:
debezium.source.bootstrap.servers=kafka.infra.svc.cluster.local:9092
debezium.source.offset.storage.partitions=1
debezium.source.offset.storage.replication.factor=1

# ─── Publication (pre-created as superuser) ───
debezium.source.publication.name=debezium_ecom_pub
debezium.source.publication.autocreate.mode=disabled

# ─── Source Database ───
debezium.source.connector.class=io.debezium.connector.postgresql.PostgresConnector

debezium.source.database.port=5432
debezium.source.database.dbname=ecomdb
debezium.source.plugin.name=pgoutput
debezium.source.slot.name=debezium_ecom_slot

Kafka Offset Topics

Offset topics must be created with cleanup.policy=compact to retain the latest offset for each key (partition). These are pre-created in the Kafka topic initialization job:

# Offset topics (compacted — retains last value per key)
recreate_compacted_topic "debezium.ecom.offsets"
recreate_compacted_topic "debezium.inventory.offsets"

# Why "recreate"? Stale offsets from a previous database instance cause
# "no longer available on the server" errors. Fresh bootstrap = fresh offsets.

Issues Encountered & Fixes

Implementing HA PostgreSQL with CNPG + Debezium + Istio Ambient Mesh involved resolving several complex interactions. Here are the issues we encountered and how they were fixed.

Issue 1: CNPG Streaming Replication TLS Timeout

Symptom: Standby pods failed to connect to primary with TLS handshake errors. Streaming replication never established.

Root Cause: Istio ztunnel double-encrypted CNPG's own TLS traffic, corrupting the handshake.

Fix: Added inheritedMetadata.annotations: ambient.istio.io/redirection: disabled to all CNPG Cluster CRs.

spec:
  inheritedMetadata:
    annotations:
      ambient.istio.io/redirection: disabled

Issue 2: CNPG Webhook Unreachable (EOF Error)

Symptom: kubectl apply of Cluster CR failed with Internal error: failed calling webhook "cluster.cnpg.io": EOF.

Root Cause: cnpg-system namespace was enrolled in Istio Ambient mesh with STRICT mTLS. kube-apiserver (outside mesh) couldn't reach the webhook.

Fix: Removed cnpg-system from ambient mesh enrollment. Changed PeerAuthentication to PERMISSIVE.

# infra/cnpg/peer-auth.yaml
spec:
  mtls:
    mode: PERMISSIVE  # kube-apiserver needs plain TCP access to webhook

Issue 3: KafkaOffsetBackingStore Missing Properties

Symptom: Debezium Server failed to start with Missing required config: bootstrap.servers.

Root Cause: KafkaOffsetBackingStore requires multiple properties beyond the offset topic name. The embedded Kafka Connect worker needs bootstrap.servers, offset.storage.partitions, and offset.storage.replication.factor — all under the debezium.source prefix.

Fix: Added all required properties:

debezium.source.offset.storage.bootstrap.servers=kafka.infra.svc.cluster.local:9092
debezium.source.bootstrap.servers=kafka.infra.svc.cluster.local:9092
debezium.source.offset.storage.partitions=1
debezium.source.offset.storage.replication.factor=1

Issue 4: REPLICATION Role Not Granted

Symptom: Debezium couldn't create a replication slot: ERROR: must be superuser or replication role.

Root Cause: CNPG creates the database owner user but does NOT automatically grant the REPLICATION attribute.

Fix: Added postInitApplicationSQL to the Cluster CR bootstrap:

bootstrap:
  initdb:
    postInitApplicationSQL:
      - ALTER USER ecomuser WITH REPLICATION;

Issue 5: Publication Must Be Created as Superuser

Symptom: ERROR: must be superuser to create FOR ALL TABLES publication when Debezium tried to auto-create the publication.

Root Cause: PostgreSQL requires superuser privileges for CREATE PUBLICATION FOR ALL TABLES. The application user (ecomuser) doesn't have superuser.

Fix: Pre-create the publication in postInitApplicationSQL (runs as postgres superuser) and disable Debezium's auto-creation:

# In Cluster CR (runs as superuser):
postInitApplicationSQL:
  - CREATE PUBLICATION debezium_ecom_pub FOR ALL TABLES;

# In Debezium config:
debezium.source.publication.name=debezium_ecom_pub
debezium.source.publication.autocreate.mode=disabled

Issue 6: Peer Authentication Failed for Application User

Symptom: psql -U ecomuser from kubectl exec failed with Peer authentication failed.

Root Cause: CNPG uses peer authentication for local socket connections. Only the postgres OS user maps to the postgres database superuser. Local kubectl exec commands must use -U postgres.

Fix: Updated all E2E tests and scripts to use psql -U postgres for kubectl exec commands.

Issue 7: Stale Kafka Offsets from Previous Run

Symptom: After a fresh cluster bootstrap, Debezium failed with Replication slot "debezium_ecom_slot" is no longer available on the server.

Root Cause: Kafka data persists on PVC across cluster recreations. Old offset data referenced LSNs from a previous database instance that no longer exists.

Fix: Updated kafka-topics-init.yaml to delete and recreate offset topics on every fresh bootstrap:

recreate_compacted_topic() {
  local topic=$1
  # Delete stale topic if exists
  kafka-topics --delete --topic "$topic" || true
  sleep 2
  # Recreate with compact policy
  kafka-topics --create --topic "$topic" \
    --partitions 1 --replication-factor 1 \
    --config cleanup.policy=compact
}

Issue 8: NetworkPolicy Blocking K8s API Server Access

Symptom: CNPG pods couldn't communicate with the Kubernetes API server, preventing the instance manager from reporting status.

Root Cause: Namespace NetworkPolicies had restrictive egress rules that didn't include the K8s API server.

Fix: Added egress rules for K8s API server (ports 443, 6443) to all database namespace NetworkPolicies:

# CNPG instance manager → K8s API server
- to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
  ports:
    - port: 443
# Direct ClusterIP access (10.96.0.1)
- ports:
    - port: 443
    - port: 6443

Issue 9: Offset Topic cleanup.policy Must Be Compact

Symptom: Debezium started but lost offsets after Kafka log retention kicked in.

Root Cause: Offset topics were created with default cleanup.policy=delete, which eventually purges old offset records. KafkaOffsetBackingStore requires compact policy to retain the latest offset indefinitely.

Fix: Create offset topics with explicit --config cleanup.policy=compact.

If migrating this architecture to AWS, the CNPG operator is replaced by managed database services. Here are the specific configuration differences:

AWS RDS PostgreSQL

RDS Multi-AZ vs Read Replicas

AWS RDS offers two HA mechanisms: Multi-AZ (synchronous standby in a different AZ, automatic failover) and Read Replicas (asynchronous replicas, manual promotion). For HA equivalent to CNPG, use Multi-AZ. For read scaling, add Read Replicas on top.

AWS Aurora PostgreSQL

Aurora vs RDS for CDC

Aurora PostgreSQL supports logical replication but with some differences. Aurora's storage is shared across all instances (no traditional standby replication). Failover is faster (~30 seconds vs ~60 seconds for RDS Multi-AZ). Aurora also supports Aurora Replicas for read scaling with automatic failover.

CNPG (Kubernetes)AWS Aurora PostgreSQLNotes
instances: 2 1 Writer + 1+ Reader in Aurora cluster Aurora cluster provides automatic failover between writer and readers
wal_level: logical Cluster Parameter Group: rds.logical_replication = 1 Same as RDS but uses cluster parameter group
ExternalName Service Aurora Cluster Endpoint (writer) + Reader Endpoint Writer endpoint auto-follows failover; reader endpoint for read queries
storageClass: standard Aurora storage (auto-scaling, 6-way replicated) No storage configuration needed; Aurora auto-scales to 128 TiB
wal_keep_size: 256MB Not applicable Aurora manages WAL differently (distributed storage layer)
CNPG streaming replication Aurora shared storage Aurora readers share the same storage volume — no WAL shipping needed

debezium.source.offset.storage.bootstrap.servers=kafka-broker:9092

# ─── Publication (unchanged) ───
# Pre-create on AWS same as CNPG:
#   ALTER USER ecomuser WITH REPLICATION;
#   CREATE PUBLICATION debezium_ecom_pub FOR ALL TABLES;
debezium.source.publication.name=debezium_ecom_pub
debezium.source.publication.autocreate.mode=disabled
AWS-Specific Considerations
  • No superuser access: RDS/Aurora don't provide postgres superuser. Use the rds_superuser role for CREATE PUBLICATION FOR ALL TABLES — the master user has this role automatically
  • Reboot required: Changing rds.logical_replication requires a database reboot (can be deferred to maintenance window)
  • Aurora logical replication slots: Replication slots on Aurora are tied to the writer instance. On failover, the promoted reader creates new slots — Debezium must reconnect and create a new slot
  • WAL retention: Aurora doesn't use wal_keep_size. Instead, logical replication slots retain WAL automatically. Monitor ReplicationSlotDiskUsage CloudWatch metric
  • MSK/Kafka: If using Amazon MSK (Managed Kafka), update bootstrap.servers to the MSK broker endpoints

Failover Test

# 1. Identify current primary
kubectl get pods -n ecom -l cnpg.io/cluster=ecom-db,cnpg.io/instanceRole=primary

# 2. Delete primary pod (simulates failure)
kubectl delete pod -n ecom -l cnpg.io/instanceRole=primary,cnpg.io/cluster=ecom-db

# 3. Watch CNPG promote standby (~5-15 seconds)
kubectl get pods -n ecom -l cnpg.io/cluster=ecom-db -w

# 4. Verify app still works
curl -sk https://api.service.net:30000/ecom/books
# Expected: 200 OK (may take 5-15 seconds to reconnect)

# 5. Verify cluster recovers to 2 instances
kubectl wait --for=jsonpath='{.status.readyInstances}'=2 \
  cluster/ecom-db -n ecom --timeout=120s
