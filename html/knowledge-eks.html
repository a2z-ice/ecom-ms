Part 1: Foundation — The Network Layer

1 Region & Availability Zones

What is it?

An AWS Region is a physical geographic area — a cluster of data centers in a specific part of the world. Examples: us-east-1 (Northern Virginia), eu-west-1 (Ireland), ap-southeast-1 (Singapore).

An Availability Zone (AZ) is an isolated data center within a region. Each AZ has independent power, cooling, and networking. A region typically has 3 AZs: us-east-1a, us-east-1b, us-east-1c.

Real-world analogy: A region is a city. Availability Zones are separate power plants within that city. If one power plant fails, the others keep the lights on. Your house is wired to all three so it never goes dark.

Why do we need it?

Without multiple AZs, a single hardware failure could take your entire application offline. By spreading resources across 3 AZs, you get fault isolation — no single point of failure. AZs within a region are connected by high-bandwidth, sub-millisecond private fiber.

How does it work?

When you create resources, you specify which AZ(s) to place them in. For HA, place copies in multiple AZs. If AZ-a fails, traffic shifts to AZ-b and AZ-c automatically.

Terraform resource: provider "aws" { region = "us-east-1" } — AZs discovered via data "aws_availability_zones" "available" {}.
Common mistake: Deploying everything in a single AZ. Multi-hour AZ outages happen. Always use at least 2 AZs, preferably 3.
Cost implication: Regions are free but cross-AZ data transfer costs $0.01/GB each way. Keep chatty services co-located where possible.

2 VPC (Virtual Private Cloud)

What is it?

A VPC is your own private, isolated network within AWS. Every VPC has a CIDR block — a range of private IP addresses. For example, 10.0.0.0/16 gives you 65,536 IP addresses. The /16 means the first 16 bits are the network prefix; the remaining 16 bits are available for hosts.

Real-world analogy: A VPC is your private office floor in a skyscraper. You control who enters (firewall rules), how rooms are arranged (subnets), and which rooms face the street (public subnets). Other tenants cannot walk into your space.

Why do we need it?

Network isolation — your traffic is private, you define the address space, and you control all connectivity to and from the internet. No other AWS customer can access your VPC.

How does it work?

You create a VPC with a CIDR block, then subdivide it into subnets. All resources live inside subnets. Traffic within the VPC flows freely; traffic to the internet requires explicit gateways and routes.

Terraform resource: module "vpc" (terraform-aws-modules/vpc/aws) with cidr = "10.0.0.0/16".
Common mistake: Choosing /24 (256 IPs). EKS pods and nodes consume IPs rapidly. Always use /16 for production.
Cost implication: VPC itself is free. Costs come from NAT Gateways, VPC endpoints, and cross-AZ data transfer inside the VPC.

3 Subnets

What is it?

A subnet is a partition of your VPC. Three tiers:

  • Public (10.0.1-3.0/24) — Load balancers live here, direct internet access via IGW
  • Private (10.0.11-13.0/24) — EKS worker nodes live here, internet via NAT only
  • Database (10.0.21-23.0/24) — Databases live here, no internet at all

Each subnet exists in exactly one AZ. For HA: 3 tiers x 3 AZs = 9 subnets.

Real-world analogy: Public subnets are the reception area — visitors can enter. Private subnets are workrooms — employees work here but visitors cannot walk in. Database subnets are the vault — sealed off, accessible only from workrooms.

Why do we need it?

Defense in depth. A compromised resource in one tier cannot directly access resources in a deeper tier without crossing network boundaries.

How does it work?

Each subnet gets a CIDR and a route table. Public subnets route 0.0.0.0/0 to IGW. Private subnets route 0.0.0.0/0 to NAT. Database subnets have no internet route.

Terraform resource: VPC module: private_subnets, public_subnets, database_subnets lists. EKS-required tags: kubernetes.io/role/elb = 1 on public, kubernetes.io/role/internal-elb = 1 on private.
Common mistake: Forgetting EKS subnet tags. Without kubernetes.io/role/elb, the AWS Load Balancer Controller cannot create ALBs.
Cost implication: Subnets are free. Cost comes from NAT Gateways attached to private subnets. Database subnets avoid NAT cost entirely.

4 Internet Gateway (IGW)

What is it?

The Internet Gateway is the front door of your VPC — the single connection point between your VPC and the public internet. Without it, nothing inside your VPC can reach the internet.

Real-world analogy: The IGW is the main entrance to your building. It does not decide who gets in — that is the job of security guards (Security Groups). It simply provides the doorway.

Why do we need it?

Your load balancer sits in a public subnet and must receive traffic from users on the internet. The IGW enables that. It also allows public resources to send responses back.

How does it work?

Attach one IGW to the VPC. Add route 0.0.0.0/0 -> igw-xxxxx in public subnet route tables. The IGW performs 1:1 NAT between private and public IPs.

Terraform resource: VPC module auto-creates: aws_internet_gateway (default create_igw = true).
Common mistake: Creating the IGW but not adding the route in the public route table. The subnet appears public but has no internet connectivity.
Cost implication: IGW is completely free. You only pay for data transfer through it ($0.09/GB outbound).

5 NAT Gateway

What is it?

A NAT Gateway allows private subnet resources to reach the internet without being reachable from the internet. EKS nodes need to pull container images but should not have public IPs.

Real-world analogy: A NAT Gateway is a one-way mail room. You can send letters out (requests) and receive replies. But nobody outside can open the door to come in. Your return address is the mail room, not your private office number.

Why do we need it?

EKS nodes pull images from ECR/Docker Hub, communicate with AWS APIs, and download patches. NAT provides outbound access without exposing nodes to inbound internet traffic.

How does it work?

NAT sits in a public subnet with an Elastic IP. Private route tables point 0.0.0.0/0 -> nat-xxxxx. It replaces the source IP with its own public IP before forwarding. One per AZ for HA.

Terraform resource: VPC module: enable_nat_gateway = true, one_nat_gateway_per_az = true. Creates aws_nat_gateway + aws_eip per AZ.
Common mistake: Single NAT Gateway for all AZs. If that AZ fails, all private subnets lose internet. Use one per AZ in production.
Cost implication: ~$45/month per gateway + $0.045/GB processed. With 3 AZs = $135/month for NAT alone. Use 1 NAT for dev to save costs.

6 Route Tables

What is it?

A route table is a set of rules (routes) that determine where network traffic is directed. Every subnet is associated with exactly one route table. Think of it as a set of road signs telling packets which direction to go.

Real-world analogy: Route tables are like road signs at intersections. "To reach the internet, go through this door (IGW/NAT). To reach another subnet, go down this hallway (local). No sign for your destination? Traffic is dropped."

Why do we need it?

Without route tables, traffic would have nowhere to go. They define the network topology: which subnets can reach the internet, which go through NAT, and which are fully isolated.

How does it work?

Three route tables in our architecture:

  • Public: 10.0.0.0/16 -> local + 0.0.0.0/0 -> igw-xxx
  • Private: 10.0.0.0/16 -> local + 0.0.0.0/0 -> nat-xxx
  • Database: 10.0.0.0/16 -> local only (no default route = no internet)
Terraform resource: Auto-created by VPC module: aws_route_table, aws_route, aws_route_table_association.
Common mistake: Associating a private subnet with the public route table. Nodes suddenly become internet-accessible without Security Groups properly configured.
Cost implication: Route tables are free. No cost to create or maintain them.

7 Security Groups

What is it?

A Security Group is a virtual firewall attached to individual resources (EC2 instances, load balancers, databases). It controls inbound and outbound traffic at the instance level. Security Groups are stateful — if you allow an inbound request, the response is automatically allowed out.

Real-world analogy: A Security Group is a personal bodyguard for each resource. The bodyguard has a clipboard with a list of who is allowed in and who the resource can talk to. When someone is let in, their reply is automatically allowed out (stateful).

Why do we need it?

Network-level access control. Even within the same subnet, Security Groups ensure that only authorized traffic reaches each resource. The ALB SG allows port 443 from the internet; the EKS node SG only allows traffic from the ALB SG.

How does it work?

You define inbound rules (source IP/SG + port) and outbound rules (destination IP/SG + port). A powerful feature: rules can reference other Security Groups. "Allow inbound on port 8080 from SG sg-alb" means only the ALB can talk to your nodes on that port.

Terraform resource: aws_security_group + aws_security_group_rule. EKS module creates cluster and node SGs automatically.
Common mistake: Opening 0.0.0.0/0 on all ports for debugging and forgetting to close it. Use SG-to-SG references instead of IP ranges wherever possible.
Cost implication: Security Groups are free. No cost, no limits that matter in practice.

8 Network ACLs (NACLs)

What is it?

A Network ACL is a firewall at the subnet level (vs Security Groups at the instance level). NACLs are stateless — you must explicitly allow both inbound AND outbound traffic, including return traffic on ephemeral ports.

Real-world analogy: If Security Groups are personal bodyguards, NACLs are the security desk in the lobby of each floor. Everyone entering or leaving the floor is checked, regardless of which office they visit. Unlike bodyguards, the desk does not remember who came in — they check both entry and exit independently.

Why do we need it?

Defense in depth. NACLs provide a second layer of network security. Even if a Security Group is misconfigured, the NACL can block unwanted traffic. In practice, most teams rely primarily on Security Groups and keep NACLs at default (allow all). NACLs become important when you need to block specific IP ranges at the subnet level.

How does it work?

Rules are evaluated in order by rule number (lowest first). First matching rule wins. Must have both allow inbound AND allow outbound (including ephemeral ports 1024-65535 for return traffic).

Terraform resource: aws_network_acl + aws_network_acl_rule. Default NACL allows all — most teams leave it as-is.
Common mistake: Adding a NACL deny rule but forgetting to allow ephemeral ports (1024-65535) for return traffic. Because NACLs are stateless, blocking ephemeral ports breaks ALL responses.
Cost implication: NACLs are free.

Part 2: Compute — Kubernetes on AWS

9 EKS (Elastic Kubernetes Service)

What is it?

Kubernetes is a system that manages containers at scale — it decides which servers run which containers, restarts crashed ones, scales up when traffic increases, and rolls out new versions without downtime. Think of it as an orchestra conductor for containers.

EKS is AWS's managed Kubernetes. AWS runs the control plane (the brain) — you just run worker nodes (the muscles) and deploy your applications.

Real-world analogy: Running Kubernetes yourself is like owning a restaurant AND being the manager, head chef, and accountant. EKS is like hiring AWS as your manager — they handle scheduling, health checks, and keeping the kitchen running. You just provide the cooks (worker nodes) and the recipes (container images).

Why do we need it?

Without Kubernetes, you would manually SSH into servers, start containers, monitor them, and restart failures. With 10+ microservices, this becomes impossible. Kubernetes automates all of it. EKS specifically removes the burden of managing the control plane — etcd backups, API server patching, certificate rotation.

How does it work?

EKS has two planes: the Control Plane (AWS-managed: API server, etcd, scheduler, controller manager) and the Data Plane (your worker nodes that run pods). They communicate via ENIs (Elastic Network Interfaces) injected into your VPC.

Terraform resource: module "eks" (terraform-aws-modules/eks/aws) — creates cluster, OIDC provider, managed node groups, IAM roles.
Common mistake: Making the API server endpoint fully public without IP restrictions. Always restrict to known CIDRs or use private endpoint.
Cost implication: EKS control plane: $0.10/hour = ~$73/month fixed. Worker node costs are separate (EC2 pricing).

10 EKS Control Plane (Master Nodes)

What is it?

The control plane is the brain of your Kubernetes cluster. It lives in an AWS-managed VPC (not your VPC) and consists of: the API Server (front door for all cluster operations), etcd (distributed key-value database storing all cluster state), the Scheduler (decides which node runs each pod), and Controller Manager (ensures desired state matches actual state).

Real-world analogy: The control plane is like a hospital's administration office. The API Server is the reception desk (all requests go through it). etcd is the patient records system. The Scheduler is the triage nurse deciding which ward gets each patient. Controllers are the department heads ensuring every ward has the right number of beds.

Why do we need it?

Without the control plane, there is no coordination. Nodes would not know which pods to run, failed pods would not be restarted, and new deployments could not be rolled out. AWS manages this entirely — you never SSH into master nodes, patch etcd, or rotate control plane certificates.

How does it work?

The control plane runs in AWS's VPC. It connects to your VPC through ENIs (Elastic Network Interfaces) placed in your private subnets. Your worker nodes talk to the API Server through these ENIs. You choose public or private endpoint access — private means only resources within your VPC can reach the API Server.

Terraform resource: module "eks" with cluster_endpoint_private_access = true and cluster_endpoint_public_access = true (restrict via cluster_endpoint_public_access_cidrs).
Common mistake: Disabling both public and private endpoints. You must have at least one enabled or you cannot reach the API server at all.
Cost implication: Included in the $73/month EKS fee. No additional cost for ENIs created by the control plane.

11 Worker Nodes (Managed Node Groups)

What is it?

Worker nodes are EC2 instances that actually run your containers (pods). A Managed Node Group means AWS handles the lifecycle — AMI updates, instance replacement, graceful draining during upgrades.

Real-world analogy: Worker nodes are the chefs in your restaurant. The control plane (manager) tells each chef what dishes to prepare (pods to run). Managed Node Groups mean AWS is the staffing agency — they hire, train, and replace chefs for you. You just say "I need 3-6 chefs."

Why do we need it?

Containers need servers to run on. Managed Node Groups remove the operational burden of patching OS images, handling instance failures, and rolling upgrades. You define the instance type, min/max count, and AWS does the rest.

How does it work?

Two node groups are common: System (runs infrastructure pods like CoreDNS, metrics-server, Istio) and App (runs your application pods). This separation ensures infrastructure pods always have resources, even when app pods are under heavy load. Each group has its own instance type and autoscaling range.

Terraform resource: eks_managed_node_groups block inside module "eks". Key settings: instance_types, min_size, max_size, desired_size.
Common mistake: Using a single node group for everything. A spike in application traffic can starve system pods (CoreDNS, metrics-server), causing cascading failures.
Cost implication: t3.medium: ~$30/month. t3.large: ~$60/month. With 4-9 nodes total: ~$180-540/month for compute. Use Spot instances for non-critical workloads to save 60-90%.

12 Pods, Deployments, Services

What is it?

Three fundamental Kubernetes objects:

  • Pod — The smallest deployable unit. Contains one or more containers sharing network and storage. Ephemeral — pods come and go.
  • Deployment — Manages a set of identical pods. You declare "I want 3 replicas" and the Deployment ensures exactly 3 are always running. Handles rolling updates and rollbacks.
  • Service — A stable network endpoint for a set of pods. Pods have random IPs that change; a Service provides a fixed DNS name and IP that routes to healthy pods.
Real-world analogy: Pods are individual cashiers. A Deployment is the staffing policy: "always have 3 cashiers working; if one calls in sick, hire a replacement immediately." A Service is the "Cashier" sign above the counter — customers go to the sign, not a specific person. If cashier #2 goes on break, the sign still works because #1 and #3 are there.

Why do we need it?

Containers crash, nodes fail, traffic fluctuates. Deployments ensure your app stays running with the right number of replicas. Services ensure other apps can always find yours, even as individual pods are replaced.

How does it work?

You create a Deployment YAML specifying the image, replicas, and resources. Kubernetes creates pods across nodes. A Service selects pods by label and load-balances traffic to them. For external access, an Ingress or Gateway routes outside traffic to Services.

Terraform resource: Kubernetes resources are typically deployed via Helm charts or kubectl apply, not Terraform. Terraform manages the cluster; Helm/ArgoCD manages workloads inside it.
Common mistake: Not setting resource requests/limits. Without them, a single pod can consume all node resources and crash everything else on that node.
Cost implication: Pods run on EC2 nodes. More replicas = more node capacity needed = more EC2 cost. Right-size your resource requests.

13 Namespaces

What is it?

A Namespace is a virtual cluster within a Kubernetes cluster. It provides logical isolation — resources in different namespaces can have the same names without conflict. RBAC policies, resource quotas, and network policies are scoped to namespaces.

Real-world analogy: Namespaces are departments in a company. Engineering and Marketing can both have a "dashboard" project without conflict. Each department has its own budget (resource quotas) and access controls (RBAC). People in Engineering cannot access Marketing's confidential files unless explicitly granted.

Why do we need it?

Organization and security. Without namespaces, all resources are in the default namespace — a flat mess. Namespaces let you apply per-team resource quotas, restrict access with RBAC, and enforce network policies that prevent cross-service communication by default.

How does it work?

BookStore namespaces: ecom, inventory, analytics, identity, infra, observability, cert-manager, istio-system. Each service lives in its own namespace with NetworkPolicies controlling what can communicate.

Terraform resource: kubernetes_namespace or created by Helm charts. Istio labels (istio.io/dataplane-mode: ambient) are applied per namespace.
Common mistake: Putting everything in default namespace. No isolation, no resource control, no RBAC separation.
Cost implication: Namespaces are free. No cost.

14 OIDC Provider & IRSA

What is it?

IRSA (IAM Roles for Service Accounts) lets individual Kubernetes pods assume specific AWS IAM roles. Instead of giving ALL pods on a node the same AWS permissions, each pod gets only what it needs.

Real-world analogy: Without IRSA, everyone in the office uses the same master keycard (node IAM role). The janitor, the CEO, and the intern all have access to everything. With IRSA, each person gets their own keycard that only opens the doors they need. The janitor can enter supply closets but not the vault.

Why do we need it?

Principle of least privilege. The ecom-service needs access to its Aurora database and MSK. The inventory-service needs access to different Aurora and MSK. Without IRSA, both share the node's IAM role — which must have ALL permissions. One compromised pod gets access to everything.

How does it work?

EKS creates an OIDC provider (identity verifier). When a pod uses a Kubernetes ServiceAccount annotated with an IAM role ARN, the pod receives a JWT token. It exchanges this token via AWS STS (Security Token Service) for temporary AWS credentials scoped to that specific IAM role.

Terraform resource: OIDC provider created by EKS module. IRSA roles via module "irsa" or aws_iam_role with OIDC trust policy + kubernetes_service_account with annotation.
Common mistake: Using the node IAM role for everything. If any pod on the node is compromised, the attacker gets access to all AWS services the node role permits.
Cost implication: IRSA is free. STS calls are free. No additional cost.

Part 3: Storage & Data

15 ECR (Elastic Container Registry)

What is it?

ECR is AWS's private Docker image registry. When you build a container image, you push it to ECR. When Kubernetes needs to run that container, it pulls the image from ECR.

Real-world analogy: ECR is your company's private recipe book. You write recipes (build images) and store them securely. When the kitchen needs to make a dish, the chef (Kubernetes) pulls the recipe from the book. Nobody outside your company can access these recipes.

Why do we need it?

Security (images are private, scanned for vulnerabilities), speed (same-region pull is fast), and IAM integration (no Docker Hub credentials to manage). Lifecycle policies auto-delete old images.

How does it work?

CI/CD builds images, pushes to ECR. EKS nodes pull images via IRSA credentials or node IAM role. ECR can scan images on push for known CVEs.

Terraform resource: aws_ecr_repository per service. aws_ecr_lifecycle_policy to auto-delete images older than N days.
Common mistake: Not setting lifecycle policies. Old images accumulate and storage costs grow silently. Keep only the last 10-20 images.
Cost implication: $0.10/GB/month storage + $0.09/GB data transfer (cross-region). Same-region pull is free. Typical cost: $1-5/month.

16 Aurora PostgreSQL (Serverless v2)

What is it?

Aurora is AWS's cloud-native relational database, compatible with PostgreSQL. It stores 6 copies of your data across 3 AZs automatically. Serverless v2 means compute scales up and down automatically based on demand — measured in ACUs (Aurora Capacity Units).

Real-world analogy: Regular PostgreSQL on EC2 is like buying a generator for your home — you maintain it, fuel it, fix it. Aurora is like city power — it just works, auto-scales, and you pay for what you use. The 6 copies across 3 AZs is like having 6 backup generators in 3 different buildings.

Why do we need it?

Each microservice needs its own database (strict isolation). Aurora provides automatic failover (~30 seconds), 6-way replication, auto-scaling, point-in-time recovery, and wal_level=logical support for Debezium CDC.

How does it work?

Writer instance handles all writes. Reader instance handles read queries (offloading the writer). Storage is shared across AZs — a write in AZ-a is replicated to AZ-b and AZ-c within milliseconds. On failure, the reader is promoted to writer automatically.

Terraform resource: aws_rds_cluster with engine_mode = "provisioned", serverlessv2_scaling_configuration. aws_rds_cluster_instance per instance.
Common mistake: Not enabling wal_level=logical via rds.logical_replication = 1 parameter. Debezium CDC requires this and it needs a reboot.
Cost implication: Minimum ~$43/month per cluster (0.5 ACU). With 4 clusters (ecom, inventory, analytics, keycloak) = ~$175/month minimum. Scales up with traffic.

17 S3 (Simple Storage Service)

What is it?

S3 is object storage — think of it as an infinite hard drive in the cloud. You store files (objects) in buckets. S3 has 99.999999999% (eleven 9s) durability — your data is essentially indestructible.

Real-world analogy: S3 is a warehouse with infinite shelves. Each bucket is a section, each object is a box on a shelf. The warehouse makes 11 copies of each box across multiple buildings. You would need to simultaneously lose all 11 copies to lose data — virtually impossible.

Why do we need it?

In our EKS setup, S3 stores Terraform state — the file that tracks what infrastructure exists. Without remote state, two engineers running Terraform simultaneously could create duplicate resources or destroy each other's work. S3 + versioning means every state change is tracked and recoverable.

How does it work?

Create a bucket with versioning enabled. Terraform writes its .tfstate file to S3. Combined with DynamoDB locking, this prevents concurrent modifications. Server-side encryption ensures the state (which contains secrets) is encrypted at rest.

Terraform resource: aws_s3_bucket + aws_s3_bucket_versioning + aws_s3_bucket_server_side_encryption_configuration.
Common mistake: Not enabling versioning. A corrupted state file without versioning means you lose track of all your infrastructure.
Cost implication: $0.023/GB/month. Terraform state files are tiny (KB). Cost is effectively $0.

18 DynamoDB

What is it?

DynamoDB is a fully managed key-value NoSQL database. In our architecture, it serves exactly one purpose: Terraform state locking.

Real-world analogy: DynamoDB for state locking is like a "Do Not Disturb" sign on a hotel room door. When Terraform starts working, it puts up the sign (acquires the lock). If another Terraform process tries to enter, it sees the sign and waits. Without the sign, two people could rearrange the same room simultaneously and create chaos.

Why do we need it?

S3 alone does not prevent two people from running terraform apply at the same time. DynamoDB provides a lock — only one Terraform process can modify infrastructure at a time. This prevents race conditions and state corruption.

How does it work?

Terraform writes a lock record to DynamoDB before modifying state. The record contains who holds the lock and when it was acquired. When done, Terraform deletes the lock. If Terraform crashes, the lock has a timeout.

Terraform resource: aws_dynamodb_table with hash_key = "LockID", billing_mode = "PAY_PER_REQUEST".
Common mistake: Forgetting to configure the DynamoDB table in the Terraform backend. State locking silently does not work, and concurrent applies corrupt state.
Cost implication: Pay-per-request pricing. Terraform locks are infrequent — cost is effectively $0.

Part 4: Messaging & Caching

19 Amazon MSK (Managed Streaming for Kafka)

What is it?

Kafka is a distributed event streaming platform — a high-throughput message bus. Producers publish messages to topics, and consumers read from them. Messages are stored durably and can be replayed. MSK is AWS-managed Kafka — no broker management, patching, or ZooKeeper headaches.

Real-world analogy: Kafka is a newspaper printing press. Writers (producers) submit articles to sections (topics). Readers (consumers) subscribe to sections. The press keeps all back issues (message retention). MSK means AWS runs the printing press — you just write articles and read the paper.

Why do we need it?

Two use cases: (1) CDC events — Debezium captures database changes and publishes to Kafka topics. Flink reads these to populate the analytics database. (2) Application events — ecom-service publishes order.created, inventory-service consumes it to deduct stock.

How does it work?

3 brokers across 3 AZs. Topics are partitioned for parallelism. Replication factor of 3 means every message exists on 3 brokers. min.insync.replicas = 2 means a write is only acknowledged when at least 2 brokers have it — balancing durability with availability.

Terraform resource: aws_msk_cluster with broker_node_group_info, number_of_broker_nodes = 3.
Common mistake: Not sizing broker storage. Kafka stores messages for the retention period. If disk fills up, the cluster stops accepting writes.
Cost implication: Starts at ~$150/month (kafka.t3.small x 3). Storage: $0.10/GB/month. Production with kafka.m5.large: ~$450/month.

20 ElastiCache Redis

What is it?

Redis is an in-memory key-value store — extremely fast (sub-millisecond). ElastiCache is AWS-managed Redis with automatic failover, patching, and backups.

Real-world analogy: Redis is a sticky note board right next to your desk. Need to look something up? Glance at the board (microseconds) instead of walking to the filing cabinet (database query, milliseconds). ElastiCache means AWS maintains the board — replacing worn-out notes and keeping a backup.

Why do we need it?

Three use cases: CSRF tokens (stored per session, validated on every mutating request), rate limiting (token buckets per user/endpoint), and session cache (fast access to session data).

How does it work?

Primary node handles reads and writes. Replica in another AZ handles reads and serves as failover target. If primary fails, ElastiCache promotes replica to primary in seconds. DNS endpoint auto-updates.

Terraform resource: aws_elasticache_replication_group with num_cache_clusters = 2, automatic_failover_enabled = true.
Common mistake: Single-node Redis with no replica. Node failure = total CSRF/rate-limiting loss. Always have at least 1 replica.
Cost implication: cache.t3.micro: ~$13/month. cache.r6g.large for production: ~$130/month. Primary + replica doubles the cost.

Part 5: Networking & Traffic

21 Application Load Balancer (ALB)

What is it?

An ALB is a Layer 7 (HTTP/HTTPS) load balancer. It distributes incoming traffic across multiple targets (pods). It understands HTTP, so it can route based on host name, URL path, headers, and query strings.

Real-world analogy: An ALB is a receptionist at a large building with many offices. The receptionist reads the visitor's purpose (HTTP request) and directs them: "Looking for bookstore.com? Floor 3 (UI pods). Looking for api.bookstore.com/ecom? Floor 5 (ecom pods)." If an office is closed (unhealthy), the receptionist redirects to another one.

Why do we need it?

Without a load balancer, users would need to know the IP address of a specific pod — which changes constantly. ALB provides a stable DNS name, distributes load evenly, terminates TLS, and routes requests to the correct service.

How does it work?

The AWS Load Balancer Controller (runs in EKS) watches for Kubernetes Ingress/Gateway objects. When you create one, it automatically provisions an ALB with the right rules. ALB performs health checks every 15 seconds and removes unhealthy targets.

Terraform resource: ALB is created dynamically by the AWS Load Balancer Controller (Helm chart), not by Terraform directly. Terraform installs the controller: helm_release "aws-load-balancer-controller".
Common mistake: Not enabling cross-zone load balancing. Without it, an AZ with fewer pods gets the same traffic as others — overloading those pods.
Cost implication: ~$22/month base + $0.008 per LCU-hour (load-dependent). Typical small app: $25-40/month.

22 AWS WAF (Web Application Firewall)

What is it?

WAF filters malicious HTTP/HTTPS traffic before it reaches your application. It inspects requests for common attack patterns like SQL injection, cross-site scripting (XSS), and excessive request rates.

Real-world analogy: WAF is a bouncer at a nightclub. The bouncer checks IDs (request inspection), turns away troublemakers (malicious requests), and enforces capacity limits (rate limiting). The bouncer stands in front of the receptionist (ALB), so threats are stopped before they even reach the building.

Why do we need it?

Defense against OWASP Top 10 attacks (SQL injection, XSS), bot traffic, DDoS (layer 7), and geo-blocking. Even with application-level validation, WAF provides an additional layer that catches attacks before they consume application resources.

How does it work?

WAF attaches to the ALB. Every request is evaluated against rule groups. AWS provides managed rule groups (free or paid) for common threats. You can add custom rules for rate limiting per IP, geo-blocking, or pattern matching.

Terraform resource: aws_wafv2_web_acl with aws_wafv2_web_acl_association to attach to ALB ARN.
Common mistake: Enabling WAF in "count" mode instead of "block" mode during testing and forgetting to switch. Attacks are logged but not stopped.
Cost implication: $5/month per web ACL + $1/month per rule + $0.60 per million requests. Managed rule groups: free (AWS Core) to $5/month each.

23 Route 53 (DNS)

What is it?

Route 53 is AWS's DNS service. DNS translates human-readable domain names (myecom.net) into IP addresses (54.23.1.100) that computers use to route traffic. Route 53 is also a domain registrar and health checker.

Real-world analogy: DNS is the phone book of the internet. When you want to call "myecom.net," you look it up in Route 53's phone book to find the actual phone number (IP address). Route 53 can also redirect calls to a backup number if the primary line is down (health checks + failover routing).

Why do we need it?

Users type myecom.net, not an ALB's auto-generated DNS name (k8s-default-alb-a1b2c3.us-east-1.elb.amazonaws.com). Route 53 maps your domain to the ALB. It also enables health-check-based routing for disaster recovery.

How does it work?

Create a hosted zone for your domain. Add an ALIAS record pointing myecom.net to the ALB. ALIAS records are AWS-specific — they resolve to the ALB's changing IP addresses without a CNAME hop, and they are free to query.

Terraform resource: aws_route53_zone + aws_route53_record with alias block pointing to ALB.
Common mistake: Using CNAME instead of ALIAS for zone apex (myecom.net). CNAME records are not allowed at zone apex per DNS specification. Use ALIAS (A record).
Cost implication: $0.50/month per hosted zone + $0.40 per million queries. ALIAS queries to AWS resources are free. Typical cost: $1-2/month.

24 ACM (AWS Certificate Manager)

What is it?

ACM provides free, auto-renewing TLS/SSL certificates. A TLS certificate does two things: proves your site's identity (you own myecom.net) and encrypts traffic between browser and server (HTTPS).

Real-world analogy: ACM is like a notary service that verifies your identity and gives you an official stamp (certificate). When visitors see the stamp on your door (HTTPS lock icon), they know they are talking to the real you, not an imposter. ACM renews the stamp automatically before it expires — you never have to visit the notary office again.

Why do we need it?

HTTPS is mandatory for production websites. Without TLS, all traffic (including passwords) flows in plain text. Browsers show "Not Secure" warnings. ACM eliminates the pain of certificate management — no expiration surprises, no manual renewal.

How does it work?

Request a certificate for your domain(s). ACM creates a DNS validation record — you add it to Route 53 to prove domain ownership. Once validated, attach the certificate to the ALB. ACM auto-renews 60 days before expiration.

Terraform resource: aws_acm_certificate with validation_method = "DNS". aws_route53_record for DNS validation. aws_acm_certificate_validation to wait for it.
Common mistake: Creating a certificate in the wrong region. For ALB, ACM certs must be in the same region. For CloudFront, they must be in us-east-1.
Cost implication: Completely free for public certificates. Zero cost, zero renewal headaches.

Part 6: Security & Identity

25 IAM (Identity & Access Management)

What is it?

IAM controls who can do what in your AWS account. It has four building blocks: Users (people), Groups (collections of users), Roles (temporary identities assumed by services), and Policies (JSON documents defining permissions).

Real-world analogy: IAM is the HR department of your AWS account. HR decides who gets hired (Users), which department they join (Groups), what temporary contractor badges look like (Roles), and what each badge allows access to (Policies). The principle of least privilege means: give everyone the minimum access needed to do their job.

Why do we need it?

Without IAM, any process with AWS credentials could do anything — delete databases, read all secrets, launch expensive instances. IAM enforces least privilege: the EKS cluster role can only manage EKS, the node role can only pull images and write logs, and each service's IRSA role can only access its own database.

How does it work?

EKS uses several IAM roles: Cluster Role (allows EKS to manage ENIs, security groups, logging), Node Role (allows EC2 instances to join the cluster, pull from ECR, write to CloudWatch), and IRSA Roles (per-pod permissions for specific AWS services).

Terraform resource: aws_iam_role, aws_iam_role_policy_attachment, aws_iam_policy. EKS module creates cluster and node roles automatically.
Common mistake: Attaching AdministratorAccess to roles because "it works." This violates least privilege and means a compromised pod can do anything in your account.
Cost implication: IAM is completely free. Unlimited users, roles, and policies at no cost.

26 KMS (Key Management Service)

What is it?

KMS manages encryption keys. Instead of handling raw encryption keys yourself (risky — they can be leaked, lost, or mismanaged), KMS stores keys in tamper-resistant hardware (HSMs) and performs encrypt/decrypt operations on your behalf.

Real-world analogy: KMS is a bank vault for your keys. You never take the key home — when you need to lock or unlock something, you visit the vault. The vault keeps an audit log of every time you used the key. You can revoke the key without touching the locked items.

Why do we need it?

Encryption at rest for: EKS secrets (etcd encryption), Aurora storage, S3 state files, ElastiCache data, MSK data. Without KMS, data at rest is stored in plain text on disk. A stolen hard drive exposes everything.

How does it work?

Create a KMS key. Configure each service to use it. AWS encrypts data before writing to disk and decrypts when reading. You can use AWS-managed keys (free, AWS controls rotation) or customer-managed keys (you control policy, rotation schedule, and who can use the key).

Terraform resource: aws_kms_key + aws_kms_alias. Reference in EKS: cluster_encryption_config. Aurora: kms_key_id.
Common mistake: Deleting a KMS key. All data encrypted with that key becomes permanently unrecoverable. KMS has a mandatory 7-30 day waiting period before deletion for this reason.
Cost implication: Customer-managed key: $1/month + $0.03 per 10,000 API calls. AWS-managed keys are free. Total cost is negligible.

27 Secrets Manager

What is it?

Secrets Manager securely stores passwords, API keys, database credentials, and other sensitive data. It provides automatic rotation, audit logging, and fine-grained access control.

Real-world analogy: Secrets Manager is a high-security safe deposit box at a bank. Each secret has its own box. You control who can open each box (IAM policies). The bank can automatically change the combination (rotate secrets) on a schedule. Every access is logged in a security camera (CloudTrail).

Why do we need it?

Kubernetes Secrets are base64-encoded (not encrypted by default). They live in etcd and can be read by anyone with kubectl access. Secrets Manager provides: encryption, automatic rotation, cross-service access, and audit trails. The External Secrets Operator (ESO) syncs Secrets Manager values into Kubernetes Secrets automatically.

How does it work?

Store secrets in AWS Secrets Manager. Install ESO in the cluster. ESO watches for ExternalSecret CRDs, fetches values from Secrets Manager using IRSA, and creates corresponding Kubernetes Secrets that pods consume as usual.

Terraform resource: aws_secretsmanager_secret + aws_secretsmanager_secret_version. ESO installed via Helm: helm_release "external-secrets".
Common mistake: Hardcoding secrets in Terraform code or Kubernetes manifests. Use aws_secretsmanager_secret_version with values from a secure source, never from plain text files.
Cost implication: $0.40/secret/month + $0.05 per 10,000 API calls. With ~10 secrets: $4/month.

Part 7: Service Mesh & Security

28 Istio Ambient Mesh

What is it?

A service mesh is an infrastructure layer that handles service-to-service communication. Istio adds mTLS encryption, traffic management, and access control without changing application code. Ambient mode uses a shared node-level proxy (ztunnel) instead of per-pod sidecar proxies — lower resource usage, simpler operations.

Real-world analogy: Without a service mesh, each department in a company handles its own security: checking IDs, encrypting letters, logging visitors. With Istio, a building-wide security system handles all of this. Every conversation between departments is automatically encrypted (mTLS), logged, and access-controlled. Ambient mode means one security guard per floor (ztunnel per node) instead of one per office (sidecar per pod).

Why do we need it?

Zero-trust networking. Even inside the cluster, traffic between pods is encrypted (mTLS). AuthorizationPolicies ensure only ecom-service can talk to inventory-service. No pod can eavesdrop or impersonate another. This is defense in depth beyond Kubernetes NetworkPolicies.

How does it work?

Label a namespace with istio.io/dataplane-mode: ambient. Ztunnel (one per node) intercepts all traffic from labeled namespaces, encrypts it with mTLS, and enforces AuthorizationPolicies. No sidecar injection, no pod restarts.

Terraform resource: Istio installed via Helm: helm_release "istio-base", helm_release "istiod", helm_release "ztunnel".
Common mistake: Setting PeerAuthentication to STRICT before all pods are enrolled. Non-mesh pods cannot communicate with STRICT mTLS pods — causes connection resets.
Cost implication: Istio is open-source (free). Cost is in ztunnel CPU/memory on each node — approximately 100m CPU and 128Mi per node.

29 Cert-Manager

What is it?

Cert-Manager is a Kubernetes-native certificate management controller. It automates TLS certificate issuance and renewal inside the cluster. While ACM handles external certificates (ALB), cert-manager handles internal ones (service-to-service, Istio gateway).

Real-world analogy: ACM is the notary for your front door (public). Cert-Manager is the internal badge office — it issues ID badges (certificates) to every employee (service) inside the building, verifies them automatically, and replaces them before they expire.

Why do we need it?

Internal services need TLS certificates for the Istio gateway (TLS termination), webhook endpoints, and any service that needs to prove its identity. Cert-Manager watches Certificate custom resources and automatically creates, renews, and rotates certificates.

How does it work?

Define an Issuer (CA, Let's Encrypt, self-signed). Create Certificate CRDs specifying the domain and issuer. Cert-Manager creates the certificate, stores it in a Kubernetes Secret, and renews it before expiration (default: 2/3 of lifetime).

Terraform resource: Installed via Helm: helm_release "cert-manager". CRDs applied via kubectl_manifest or Helm.
Common mistake: Not installing CRDs before deploying cert-manager. Without CRDs, Certificate resources are rejected by the API server.
Cost implication: Open-source (free). Let's Encrypt certificates are free. Resource usage: ~50m CPU, 64Mi memory.

Part 8: Observability

30 Prometheus + Grafana

What is it?

Prometheus collects and stores metrics — numerical measurements over time (CPU usage, request count, error rate, response latency). It scrapes /metrics endpoints from every pod every 15 seconds. Grafana visualizes these metrics as dashboards with charts, gauges, and alerts.

Real-world analogy: Prometheus is a diligent inspector who walks through every office every 15 seconds, writing down measurements: "Server A: 45% CPU, 12 requests/second, 0 errors." Grafana is the dashboard on the CEO's wall that turns those notes into real-time graphs. When a graph turns red, Grafana sends an alert to your phone.

Why do we need it?

You cannot improve what you cannot measure. Without metrics, you are flying blind — you will not know if a service is slow, overloaded, or about to run out of memory until users complain. Prometheus + Grafana provide proactive monitoring.

How does it work?

Prometheus discovers pods via Kubernetes service discovery (label-based). It pulls metrics from each pod's /metrics endpoint (pull model). Data is stored in a time-series database (TSDB). Grafana queries Prometheus using PromQL and renders dashboards.

Terraform resource: Deployed via Helm: helm_release "kube-prometheus-stack" — installs Prometheus, Grafana, AlertManager, and default dashboards in one chart.
Common mistake: Not setting retention or storage limits. Prometheus TSDB grows indefinitely and fills the node's disk, crashing the monitoring system itself.
Cost implication: Open-source (free). Main cost is EBS storage for metrics retention. 15-day retention with moderate traffic: ~10-50GB = $1-5/month.

31 Loki + Tempo

What is it?

Loki is a log aggregation system — it collects, stores, and queries logs from all pods. Unlike traditional log systems (ELK), Loki only indexes labels (service name, namespace), not full text, making it much cheaper. Tempo is a distributed tracing backend — it tracks individual requests as they flow across multiple services.

Real-world analogy: Loki is a filing cabinet that stores every conversation from every office. You can search by department (namespace) or person (service), but you do not need a full-text index of every word. Tempo is a package tracking system — when a package (request) moves from warehouse to sorting to delivery, Tempo records every stop (span) so you can see exactly where delays happen.

Why do we need it?

When something goes wrong, you need logs to understand what happened. When a request is slow, you need traces to find the bottleneck. Loki centralizes logs from all pods (otherwise you would kubectl into each one). Tempo shows the full journey of a request across ecom -> inventory -> database.

How does it work?

The OpenTelemetry Collector runs on each node, collects logs and traces from pods, and ships them to Loki (logs) and Tempo (traces). Grafana provides a unified UI to query both — correlating logs with traces for a specific request ID.

Terraform resource: Deployed via Helm: helm_release "loki", helm_release "tempo", helm_release "opentelemetry-collector".
Common mistake: Not setting log retention limits. Loki stores logs indefinitely by default. Without limits, storage costs escalate rapidly.
Cost implication: Open-source (free). Storage cost for S3-backed Loki/Tempo: $5-20/month depending on log volume.

32 Kiali

What is it?

Kiali is the service mesh visualization console for Istio. It shows real-time traffic flow between services as an interactive graph — which service talks to which, how much traffic, error rates, and response times.

Real-world analogy: Kiali is a live map of your office building showing all the inter-department communication happening right now. You can see: "Marketing is sending 100 messages/second to Engineering (green = healthy), but Engineering to Database has turned red (errors)." You instantly know where the problem is without checking each department individually.

Why do we need it?

Microservices create a complex web of communication. Without visualization, debugging "which service is causing the slowdown?" requires checking logs of every service. Kiali shows the answer in one glance — the service with red edges or high latency is the problem.

How does it work?

Kiali reads telemetry data from Prometheus (metrics collected by Istio proxies) and displays it as an interactive service graph. It also validates Istio configuration (AuthorizationPolicies, VirtualServices) and flags misconfigurations.

Terraform resource: Deployed via Helm: helm_release "kiali-server". Requires Prometheus as a data source.
Common mistake: Not configuring Prometheus URL. Kiali shows empty graphs without Prometheus. Set external_services.prometheus.url in Kiali config.
Cost implication: Open-source (free). Minimal resource usage: ~100m CPU, 128Mi memory.

Part 9: Putting It All Together

33 Complete Data Flow

How a user request flows through every component

1. User visits myecom.net:

  1. Browser queries Route 53 for myecom.net -> gets ALB IP address
  2. HTTPS request hits WAF -> passes rule checks -> forwarded to ALB
  3. ACM certificate on ALB terminates TLS
  4. ALB routes to Istio Gateway pod in the cluster (via target group)
  5. Gateway routes / to UI pod (serves React SPA)

2. User browses books (API call):

  1. Browser sends GET /ecom/books with JWT in Authorization header
  2. ALB routes /ecom/* to ecom-service pods
  3. Istio ztunnel encrypts with mTLS
  4. ecom-service validates JWT via Keycloak JWKS
  5. ecom-service queries Aurora ecom-db (via IRSA credentials)
  6. ecom-service calls inventory-service for stock (mTLS)
  7. Response flows back through the chain

3. User checks out (creates order):

  1. ecom-service creates order in Aurora
  2. ecom-service publishes order.created to MSK Kafka
  3. inventory-service consumes the event, deducts stock in its Aurora
  4. Debezium CDC captures both DB changes
  5. CDC events flow through MSK to Flink SQL
  6. Flink writes aggregated data to analytics Aurora
  7. Superset queries analytics DB for dashboards
Key insight: Every hop is secured. TLS at the ALB, mTLS between pods, IRSA for AWS API access, JWT for user identity, CSRF for state-changing requests, and encryption at rest for all data stores.
