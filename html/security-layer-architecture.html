LAYER 1

Edge — HTTPS Redirect + HSTS

Port 30080 → 30000 (HTTP 301) · HSTS enforced in nginx responses

Control (from architecture.html)

  • HTTP → HTTPS 301 redirect (port 30080 → 30000)
  • HSTS headers sent on HTTPS responses to prevent SSL stripping

Implementation

ControlFileEvidence
HTTP 301 redirect infra/kgateway/routes/https-redirect.yaml Kubernetes Gateway API HTTPRoute with RequestRedirect filter, scheme: https, port: 30000, statusCode: 301, attached to the http listener on port 30080
Gateway HTTP listener (port 30080) infra/kgateway/gateway.yaml Gateway resource defines http listener on port 8080 (NodePort 30080); scripts/up.sh patches the service to bind NodePort 30080
HSTS header ui/nginx/default.conf add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

Proof / Verification

# 1. HTTP → HTTPS redirect returns 301 with Location header
curl -sI http://myecom.net:30080 | head -5
# Expected: HTTP/1.1 301 Moved Permanently
#           location: https://myecom.net:30000

# 2. HSTS header present on HTTPS responses
curl -skI https://myecom.net:30000 | grep -i "strict-transport-security"
# Expected: strict-transport-security: max-age=31536000; includeSubDomains

Gotchas

  • The HSTS header is added by nginx (UI pod), not the gateway — so it appears only on UI responses, not on api.service.net API calls. API clients are expected to be HTTPS-aware.
  • HSTS max-age=31536000 (1 year) is the recommended minimum for production. preload is NOT set because this is a POC on localhost.
LAYER 2

Gateway — TLS Termination with cert-manager

Self-signed CA (10yr) → Leaf cert (30d, 7d renewBefore) · ECDSA P-256 · Multi-SAN

Control (from architecture.html)

  • TLS 1.2/1.3 terminated at the Istio Gateway on port 30000 (HTTPS 8443)
  • cert-manager v1.17.2 automates the full certificate lifecycle
  • ECDSA P-256 keys across the chain (CA + leaf)
  • Multi-SAN certificate covers all gateway hostnames
  • 30-day leaf, 7-day renewBefore (auto-rotation)

Implementation

10-year CA Certificate infra/cert-manager/ca-issuer.yaml isCA: true · duration: 87600h (10yr) · renewBefore: 8760h (1yr) · algorithm: ECDSA · size: 256
CA ClusterIssuer infra/cert-manager/ca-issuer.yaml kind: ClusterIssuer / name: bookstore-ca-issuer / ca.secretName: bookstore-ca-secret
30-day leaf gateway cert infra/cert-manager/gateway-certificate.yaml duration: 720h (30d) · renewBefore: 168h (7d) · algorithm: ECDSA · size: 256
Multi-SAN coverage infra/cert-manager/gateway-certificate.yaml dnsNames: [myecom.net, api.service.net, idp.keycloak.net, localhost] · ipAddresses: ["127.0.0.1"]

Proof / Verification

# 1. CA certificate issued and Ready
kubectl get certificate bookstore-ca -n cert-manager \
  -o jsonpath='{.status.conditions[0].status}'
# Expected: True

# 2. Leaf cert is ECDSA P-256 with multi-SAN and 30d duration
kubectl get secret bookstore-gateway-tls -n infra -o jsonpath='{.data.tls\.crt}' \
  | base64 -d | openssl x509 -noout -text \
  | grep -E "Signature Algorithm|DNS:|Public Key Algorithm|Not Before|Not After"
# Expected: Public Key Algorithm: id-ecPublicKey (P-256)
#           Signature Algorithm: ecdsa-with-SHA256
#           DNS:myecom.net, DNS:api.service.net, DNS:idp.keycloak.net, DNS:localhost
#           ~30 days between Not Before and Not After

# 3. TLS negotiated at gateway
curl -sv --max-time 5 https://api.service.net:30000/ecom/books 2>&1 \
  | grep -E "SSL connection using|issuer:|subject:" | head -5

Gotchas

  • Browsers distrust the self-signed CA until scripts/trust-ca.sh --install installs it into macOS Keychain. curl requires -k or --cacert certs/bookstore-ca.crt.
  • All E2E tests pass ignoreHTTPSErrors: true in Playwright config — needed because the CA is self-signed.
  • Tool NodePorts (PgAdmin 31111, Superset 32000, Kiali 32100, Grafana 32500, etc.) stay plain HTTP. Only the main gateway on 30000 is HTTPS.
LAYER 3

Mesh — Istio Ambient mTLS (STRICT)

SPIFFE identity · HBONE tunneling · No sidecars · ztunnel DaemonSet

Control (from architecture.html)

  • All pod-to-pod traffic encrypted via Istio Ambient mTLS
  • Every workload gets a SPIFFE identity (spiffe://cluster.local/ns/<ns>/sa/<sa>)
  • HBONE (HTTP/2-based tunneling) on port 15008 replaces sidecars
  • STRICT mTLS by default in all app namespaces

Implementation

ControlFileEvidence
Istio Ambient installed infra/istio/install.sh Installs Istio 1.29.1 with ambient profile (istiod + ztunnel DaemonSet + istio-cni)
Namespace enrolled in ambient infra/namespaces.yaml Each app namespace (ecom, inventory, infra, identity, analytics, observability) has label istio.io/dataplane-mode: ambient
STRICT mTLS enforcement infra/istio/security/peer-auth.yaml PeerAuthentication strict-mtls with mtls.mode: STRICT applied in 6 namespaces: ecom, inventory, analytics, identity, infra, observability
Port-level PERMISSIVE exceptions infra/istio/security/peer-auth.yaml NodePort-exposed workloads get portLevelMtls PERMISSIVE on specific ports: Superset(8088), Flink(8081), Keycloak(8080), Debezium(8080), Grafana(3000), PgAdmin(80), Prometheus(9090 for Kiali)
Service Account identities infra/istio/security/serviceaccounts.yaml Dedicated ServiceAccounts for each workload — ztunnel uses these for SPIFFE identity attestation

Proof / Verification

# 1. ztunnel DaemonSet is running on every node
kubectl get ds -n istio-system ztunnel
# Expected: 3/3 DESIRED/READY

# 2. STRICT mTLS is the namespace default
kubectl get peerauthentication strict-mtls -n ecom -o jsonpath='{.spec.mtls.mode}'
# Expected: STRICT

# 3. Inspect a real HBONE connection in ztunnel logs
kubectl logs -n istio-system -l app=ztunnel --tail=30 \
  | grep "hbone_addr" | head -3
# Expected: lines showing src.identity="spiffe://cluster.local/ns/.../sa/..."
#           dst.addr=POD_IP:15008 (the HBONE port)

# 4. Verify no sidecars injected
kubectl get pods -n ecom -o jsonpath='{.items[*].spec.containers[*].name}'
# Expected: just "ecom-service" — NO "istio-proxy"

Gotchas

  • Istio Ambient + non-mesh callers: AuthorizationPolicy rules using namespaces: [X] silently FAIL for callers whose namespace is not ambient-enrolled. Use rules: [{}] (ALLOW-ALL) with NetworkPolicy defense-in-depth. See up.sh Stability Fixes Issue #3 for the Kiali→Prometheus case.
  • NodePort traffic has no SPIFFE identity: host→NodePort traffic arrives plaintext. Every NodePort-exposed port needs a portLevelMtls: PERMISSIVE exception on its specific port.
  • OTel namespace PERMISSIVE: the otel namespace uses namespace-wide PERMISSIVE because ambient.istio.io/redirection: disabled on OTel/Loki/Tempo pods prevents ztunnel from establishing proper HBONE listeners. Security is enforced at NetworkPolicy layer.

Proof / Verification

# 1. Verify default-deny in every app namespace
for ns in ecom inventory analytics identity infra observability otel; do
  kubectl get netpol -n $ns | grep -q default-deny && echo "$ns: default-deny OK" || echo "$ns: MISSING"
done
# Expected: all 7 show "default-deny OK"

# 2. Count total NetworkPolicies
kubectl get netpol -A --no-headers | wc -l
# Expected: 37

# 3. Count total AuthorizationPolicies
kubectl get authorizationpolicy -A --no-headers | wc -l
# Expected: 12+

# 4. Verify ecom-db only accepts traffic from ecom + infra + cnpg-system + admin-tools
kubectl get authorizationpolicy ecom-db-policy -n ecom \
  -o jsonpath='{.spec.rules[*].from[*].source.namespaces}' | tr ' ' '\n'
# Expected: ["ecom"] ["infra"] ["cnpg-system"] ["admin-tools"]

# 5. Negative test — pod in analytics cannot reach ecom-db
kubectl run -n analytics test --rm -it --restart=Never --image=curlimages/curl --timeout=10s \
  -- curl -s --max-time 5 http://ecom-db.ecom.svc.cluster.local:5432 2>&1 | head
# Expected: connection refused / timeout (policy rejection)

Gotchas

  • Both NetworkPolicy AND AuthorizationPolicy must allow a connection — ztunnel enforces AuthZ at L4 before ALLOW; kindnet enforces NetworkPolicy at L3.
  • The PgAdmin case (in admin-tools namespace) required patching BOTH policy types for all 4 databases. scripts/up.sh lines ~510-520 do this patching via inline Python.
  • Non-mesh callers lack SPIFFE identity — namespaces: [X] rules silently fail. Use ALLOW-ALL + NetworkPolicy (see Prometheus case).
LAYER 5

Identity — Keycloak OIDC + JWT Validation

OIDC Authorization Code Flow with PKCE S256 · JWT validated at every service · In-memory token storage

Control (from architecture.html)

  • Keycloak 26.5.4 as OIDC issuer at https://idp.keycloak.net:30000/realms/bookstore
  • Authorization Code Flow with PKCE S256 (SHA-256 challenge)
  • JWT validated at every backend service via Istio RequestAuthentication and Spring Security / FastAPI middleware
  • Customer / admin roles via Keycloak realm roles, enforced by AuthorizationPolicy
  • In-memory token storage in the UI (never localStorage)

Implementation

ControlFileEvidence
Keycloak realm + clients infra/keycloak/realm-export.json Defines bookstore realm, ui-client (public), customer/admin realm roles, users user1/admin1
UI OIDC config (PKCE) ui/src/auth/oidcConfig.ts response_type: 'code' — oidc-client-ts defaults to PKCE S256 when response_type is code. Uses crypto.subtle (requires HTTPS)
In-memory token storage ui/src/auth/oidcConfig.ts userStore: new WebStorageStateStore({ store: new InMemoryWebStorage() }) — never touches localStorage
PKCE S256 requirement check ui/src/auth/AuthContext.tsx:67 Verifies crypto.subtle availability (requires secure context / HTTPS)
JWT validation @ mesh (ecom) infra/istio/security/request-auth.yaml RequestAuthentication keycloak-jwt in ecom namespace with jwtRules.issuer = Keycloak realm URL, jwksUri = internal Keycloak service
JWT validation @ mesh (inventory) infra/istio/security/request-auth.yaml Same pattern for inventory namespace
JWT validation @ ecom-service ecom-service/src/main/java/com/bookstore/ecom/config/SecurityConfig.java Spring Security OIDC Resource Server with oauth2ResourceServer().jwt() — validates JWT signature against Keycloak JWKS
JWT validation @ inventory-service inventory-service/app/middleware/auth.py python-jose + httpx middleware fetches JWKS from KEYCLOAK_JWKS_URI env var, verifies on every request
RBAC — admin role enforcement ecom-service/src/main/java/com/bookstore/ecom/config/SecurityConfig.java /admin/** endpoints require hasRole('admin'); customer tokens return 403

Proof / Verification

# 1. Obtain a customer JWT from Keycloak
TOKEN=$(curl -sk -X POST \
  "https://idp.keycloak.net:30000/realms/bookstore/protocol/openid-connect/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password&client_id=ui-client&username=user1&password=CHANGE_ME" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")
echo "Token length: ${#TOKEN}"
# Expected: ~900-1000 characters

# 2. Protected endpoint WITHOUT JWT → 401
curl -sk -o /dev/null -w "%{http_code}" https://api.service.net:30000/ecom/cart
# Expected: 401

# 3. Protected endpoint WITH customer JWT → 200
curl -sk -H "Authorization: Bearer $TOKEN" \
  -o /dev/null -w "%{http_code}" https://api.service.net:30000/ecom/cart
# Expected: 200

# 4. Admin endpoint with CUSTOMER token → 403 (role enforcement)
curl -sk -H "Authorization: Bearer $TOKEN" \
  -o /dev/null -w "%{http_code}" https://api.service.net:30000/ecom/admin/books
# Expected: 403

# 5. JWT claims include 'sub', 'realm_access.roles', 'preferred_username'
echo "$TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool \
  | grep -E "realm_access|preferred_username|sub"
# Expected: customer role present, not admin

# 6. localStorage has no tokens (browser DevTools)
# Open https://localhost:30000 → F12 → Application → LocalStorage → should be empty for tokens

Gotchas

  • PKCE requires HTTPS: crypto.subtle is only available in secure contexts. On http://myecom.net:30080 login would fail — hence the HTTPS-only edge (Layer 1).
  • Istio JWT validation is L7: RequestAuthentication only validates the JWT signature and claims; AuthorizationPolicy with requestPrincipals is needed to MANDATE a token for specific paths. In this project, backend services also validate at the application layer for defense-in-depth.
  • Refresh tokens: never exposed to JavaScript. Stored in HTTP-only cookies (Keycloak handles this).
LAYER 6

Application — CSRF, Rate Limiting, Container Hardening

Gateway ext_authz CSRF · Bucket4j rate limits · runAsNonRoot · readOnlyRootFilesystem · drop ALL capabilities

Control (from architecture.html)

  • CSRF tokens (Redis-backed, UUID v4, 10min sliding TTL)
  • Rate limiting via Bucket4j (token bucket, per-user)
  • Containers run as non-root with read-only filesystem
  • Drop ALL Linux capabilities; allowPrivilegeEscalation: false

Implementation

ControlFileEvidence
CSRF service (Go) csrf-service/internal/handler/csrf.go Gateway-level ext_authz check (Istio CUSTOM AuthorizationPolicy) — validates X-CSRF-Token header against Redis on mutating methods
CSRF Istio integration infra/istio/csrf-envoy-filter.yaml AuthorizationPolicy action: CUSTOM with provider.name: csrf-ext-authz — applied to the gateway pod
CSRF sliding TTL + auto-regeneration csrf-service/internal/handler/csrf.go TTL refreshed on every authenticated safe method via Redis EXPIRE (fire-and-forget goroutine). Expired tokens trigger auto-regeneration in 403 response body
Rate limiting (ecom-service) ecom-service/src/main/java/com/bookstore/ecom/config/RateLimitConfig.java Bucket4j OncePerRequestFilter: in-memory ConcurrentHashMap<userId, Bucket>. Limits: CART 60/min, CHECKOUT 10/min, ADMIN 30/min, BOOKS 200/min
Non-root containers ecom-service/k8s/ecom-service.yaml:60 securityContext.runAsNonRoot: true, runAsUser: 1000

Proof / Verification

# 1. CSRF enforcement — POST without X-CSRF-Token → 403
TOKEN=$(curl -sk -X POST "https://idp.keycloak.net:30000/realms/bookstore/protocol/openid-connect/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password&client_id=ui-client&username=user1&password=CHANGE_ME" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

curl -sk -X POST "https://api.service.net:30000/ecom/cart" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"bookId":"00000000-0000-0000-0000-000000000002","quantity":1}' \
  -w "\nHTTP_CODE:%{http_code}\n"
# Expected: HTTP_CODE:403

# 2. CSRF enforcement — POST WITH valid X-CSRF-Token → 200
CSRF=$(curl -sk -H "Authorization: Bearer $TOKEN" \
  https://api.service.net:30000/csrf/token \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['token'])")
curl -sk -X POST "https://api.service.net:30000/ecom/cart" \
  -H "Authorization: Bearer $TOKEN" -H "X-CSRF-Token: $CSRF" \
  -H "Content-Type: application/json" \
  -d '{"bookId":"00000000-0000-0000-0000-000000000002","quantity":1}' \
  -w "\nHTTP_CODE:%{http_code}\n"
# Expected: HTTP_CODE:200

# 3. Verify container security context on ecom-service
kubectl get pod -n ecom -l app=ecom-service -o jsonpath='{.items[0].spec.containers[0].securityContext}' \
  | python3 -m json.tool
# Expected:
#   "runAsNonRoot": true
#   "runAsUser": 1000
#   "readOnlyRootFilesystem": true
#   "allowPrivilegeEscalation": false
#   "capabilities": { "drop": ["ALL"] }

# 4. Rate limit — flood /ecom/cart and count 429s
for i in {1..80}; do
  curl -sk -o /dev/null -w "%{http_code}\n" \
    -H "Authorization: Bearer $TOKEN" -H "X-CSRF-Token: $CSRF" \
    -X POST "https://api.service.net:30000/ecom/cart" \
    -H "Content-Type: application/json" \
    -d '{"bookId":"00000000-0000-0000-0000-000000000002","quantity":1}'
done | sort | uniq -c
# Expected: some 429 after the cart bucket (60/min) drains

# 5. Verify process is non-root inside ecom-service pod
kubectl exec -n ecom deploy/ecom-service -- id
# Expected: uid=1000 gid=1000 (NOT root)

Gotchas

  • CSRF validation runs at the gateway, not in the service: the Istio CUSTOM AuthorizationPolicy calls csrf-service as an ext_authz provider BEFORE the request reaches the backend. Backend services trust the header was validated upstream.
  • Rate limits reset on pod restart because the ConcurrentHashMap is in-memory. For distributed rate limiting, move to Redis-backed Bucket4j.
  • Read-only root filesystem needs emptyDir volumes for /tmp — Spring Boot's default temp dir must be writable for multipart uploads.
