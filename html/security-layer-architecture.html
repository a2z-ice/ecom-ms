LAYER 1

Edge — HTTPS Redirect + HSTS

Port 30080 → 30000 (HTTP 301) · HSTS enforced in nginx responses

What This Layer Means

The Edge layer is the platform's outermost boundary — the first place an incoming request hits. It has ONE job: make sure traffic is HTTPS-encrypted before any authentication, session, or application data touches the wire. It does this by:

  • Listening on a plaintext HTTP port (30080) and bouncing every request to the HTTPS port (30000) with a 301 Moved Permanently redirect.
  • On every HTTPS response, adding the HSTS header which tells browsers "for the next year, never come back to me over HTTP — always upgrade to HTTPS automatically, even if the user types http://".

Why This Layer Matters for Security

  • Eliminates plaintext credential exposure. Without this layer, a user who types http://myecom.net would send the login form, JWT, and session cookies in cleartext — anyone sniffing the network (coffee-shop Wi-Fi, compromised router, malicious ISP) would steal them.
  • Defeats SSL-stripping attacks. Attackers running a MITM proxy can remove the Location: https:// header and serve plaintext back to the user. HSTS makes the browser refuse plaintext without even asking the server, so the attacker can't downgrade the connection.
  • Protects the layer immediately above (Gateway TLS). Layer 2 only terminates TLS if the request arrives over TLS — Layer 1 guarantees that every request does.
  • Closes cookie-hijack vectors. With HSTS active, cookies marked Secure are never sent over HTTP. Without HSTS, a single HTTP request can leak them.

Control (from architecture.html)

  • HTTP → HTTPS 301 redirect (port 30080 → 30000)
  • HSTS headers sent on HTTPS responses to prevent SSL stripping

LAYER 2

Gateway — TLS Termination with cert-manager

Self-signed CA (10yr) → Leaf cert (30d, 7d renewBefore) · ECDSA P-256 · Multi-SAN

What This Layer Means

The Gateway layer is the single HTTPS front door for the entire platform. It does three things:

  • Terminates TLS. The encrypted bytes from the user's browser are decrypted here so the platform can route them to the correct internal service. The gateway holds the private key.
  • Presents a valid certificate. Every TLS handshake requires the server to prove its identity with a certificate that the client trusts (signed by a CA the client knows).
  • Auto-rotates certificates. A stale or expired certificate is a hard outage — cert-manager renews leaf certificates 7 days before they expire, with zero human intervention.

Why This Layer Matters for Security

  • Confidentiality + integrity of all external traffic. TLS 1.2/1.3 guarantees that an attacker cannot read or modify requests in transit. Without this layer, login credentials, JWTs, cart contents, and admin actions all travel in cleartext.
  • Short-lived leaf certs (30d) limit blast radius. If a leaf cert's private key is somehow exfiltrated, it is automatically replaced within 7 days of expiry — an attacker has a hard deadline on stolen keys. Long-lived (1yr+) certs are a common source of silent outages and credential abuse.
  • ECDSA P-256 over RSA. ECDSA provides equivalent security to RSA-3072 with far smaller keys and signatures (64 bytes vs ~380 bytes), which means faster handshakes and lower latency on every first-connect.
  • Multi-SAN single certificate. One certificate covers all 4 gateway hostnames (myecom.net, api.service.net, idp.keycloak.net, localhost). Each hostname presents identical trust — users cannot be tricked by a look-alike name with a different CA.
  • Automation eliminates human error. Manual cert rotation is the #1 cause of avoidable TLS outages. cert-manager's declarative lifecycle means renewals happen silently, on schedule, every 23 days.

  • Multi-SAN certificate covers all gateway hostnames
  • 30-day leaf, 7-day renewBefore (auto-rotation)

LAYER 3

Mesh — Istio Ambient mTLS (STRICT)

SPIFFE identity · HBONE tunneling · No sidecars · ztunnel DaemonSet

What This Layer Means

The Mesh layer encrypts and authenticates traffic inside the cluster — between pods, services, and namespaces. It is invisible to application code. The Istio Ambient mesh has three pillars:

  • Automatic mTLS (mutual TLS). Every pod-to-pod TCP connection is wrapped in TLS in both directions: the caller authenticates to the callee AND the callee authenticates to the caller. Neither side can talk to the other without presenting a valid cluster-signed certificate.
  • SPIFFE identities. Every workload gets a cryptographic identity of the form spiffe://cluster.local/ns/<namespace>/sa/<serviceaccount>. This is the pod's passport — the one thing you can't spoof.
  • ztunnel + HBONE. A per-node ztunnel DaemonSet transparently intercepts all pod traffic and tunnels it over HTTP/2 CONNECT (HBONE) on port 15008. No per-pod sidecars, no application awareness needed.

Why This Layer Matters for Security

  • Defeats the "if an attacker is on the network, you lose" threat model. Traditional networks trust anything inside the perimeter. Zero-trust mTLS means an attacker who breaches one pod cannot freely read traffic destined for any other pod — every connection requires mutual proof of identity.
  • Stops lateral movement. A compromised pod in the observability namespace cannot impersonate ecom-service to reach ecom-db — its SPIFFE identity is baked into its workload cert, and the destination ztunnel checks AuthorizationPolicies against that exact identity.
  • Replaces IP-based trust with identity-based trust. Pod IPs churn on every restart. Identities (ServiceAccount + namespace) are stable. Security policies written against identities survive pod churn; policies written against IPs rot immediately.
  • Per-workload attestation. SPIFFE identities are issued by istiod after it verifies the pod's ServiceAccount via the Kubernetes API. This gives cryptographic proof that the pod is running under a specific SA — not just claiming to be.
  • No sidecars = lower operational cost. Ambient mode puts ztunnel on each node instead of injecting a sidecar into every pod. Memory overhead is ~50MB per node instead of ~50MB per pod. Application containers remain minimal and unaware of the mesh.
  • STRICT mode closes plaintext backdoors. PERMISSIVE mode accepts both mTLS AND plaintext — useful for migrations but a security hole. STRICT refuses any connection that doesn't present a valid peer cert.

Control (from architecture.html)

  • All pod-to-pod traffic encrypted via Istio Ambient mTLS
  • Every workload gets a SPIFFE identity (spiffe://cluster.local/ns/<ns>/sa/<sa>)
  • HBONE (HTTP/2-based tunneling) on port 15008 replaces sidecars
  • STRICT mTLS by default in all app namespaces

LAYER 4

Access — NetworkPolicy + AuthorizationPolicy

37 NetworkPolicies · 12 AuthorizationPolicies · 7 default-deny-ingress policies · namespace isolation

What This Layer Means

The Access layer decides who can talk to whom inside the cluster. It operates at two Kubernetes/Istio enforcement points:

  • NetworkPolicy (kindnet, L3/L4). Kubernetes-native firewall rules declared per-namespace. Each namespace starts with a default-deny-ingress policy that blocks ALL incoming traffic, then explicit policies allow only required flows (e.g. "gateway pods can reach ecom-service on port 8080").
  • AuthorizationPolicy (Istio, L4/L7). Istio rules enforced by ztunnel. They operate on SPIFFE identities and namespaces, not raw IPs. They decide whether the identity on the client side of the mTLS connection is allowed to talk to the workload on the server side.

Together these give "zero trust" internal networking: every connection must be explicitly allowed by BOTH layers to succeed.

Why This Layer Matters for Security

  • Enforces least-privilege network access. ecom-service needs the ecom database and Kafka — nothing else. Without this layer, a compromised ecom-service pod could reach the inventory database, Keycloak's internal API, or Kafka consumer groups it has no business touching.
  • Default-deny is the security-critical default. Without a default-deny-ingress policy, Kubernetes allows ALL pod-to-pod traffic by default. Adding allow-policies on top is useless if the implicit default is allow. Default-deny inverts the model: if it's not explicitly allowed, it's blocked.
  • Defense in depth across L3, L4, and L7. NetworkPolicy and AuthorizationPolicy use different enforcement mechanisms (kindnet vs ztunnel). An attacker needs to bypass BOTH. If a bug in one lets traffic through, the other still blocks it.
  • Namespace isolation contains blast radius. Databases live in the namespace of their owner service. If an attacker pops inventory-service, the ecom-db-policy blocks them from the ecom database. Compromise of one bounded context stays bounded.
  • Identity-based rules survive IP churn. AuthorizationPolicy rules like from.source.namespaces: [ecom] continue to work after pod restarts, scaling events, or rolling updates. IP-based firewall rules rot minutes after deployment.
  • Observable and auditable. Policies are YAML in Git. Access decisions can be code-reviewed, diffed, and rolled back — unlike ad-hoc iptables rules added via SSH.

LAYER 5

Identity — Keycloak OIDC + JWT Validation

OIDC Authorization Code Flow with PKCE S256 · JWT validated at every service · In-memory token storage

What This Layer Means

The Identity layer answers two questions on every request:

  • Who is the user? (authentication) — verified via a signed JWT issued by Keycloak after the user logs in.
  • What can they do? (authorization) — determined by the realm_access.roles claims embedded in the JWT (customer, admin).

The UI obtains a JWT via the OIDC Authorization Code Flow with PKCE (Proof Key for Code Exchange) using the S256 (SHA-256) challenge method. The JWT is attached to every API call as Authorization: Bearer <jwt>. Every backend service independently verifies its signature against Keycloak's JWKS (JSON Web Key Set).

Why This Layer Matters for Security

  • Single source of truth for identity. Passwords live in exactly one place (Keycloak). Backend services never see a password — only signed JWTs. Rotating credentials, disabling accounts, or forcing re-auth is a one-click action in Keycloak and instantly affects every service.
  • PKCE S256 defeats authorization code interception. In public clients (SPAs, mobile apps) the client secret cannot be kept secret. PKCE replaces the secret with a dynamically generated code_verifier that is hashed to code_challenge with SHA-256. An attacker who intercepts the authorization code cannot exchange it for a token without the verifier — which never leaves the legitimate browser.
  • JWT validation at every hop — zero-trust tokens. Tokens are validated at two layers: (1) Istio RequestAuthentication checks the JWT signature at the mesh before traffic reaches the pod, (2) Spring Security / FastAPI middleware re-validates inside the app. Defense-in-depth: even if the mesh is misconfigured, the app still enforces auth.
  • Role-based access control at the edge. Admin endpoints require hasRole('admin'). A customer's JWT lacks the admin role — their request to /admin/** returns 403 before any business logic runs. No privilege escalation via URL guessing.
  • In-memory token storage blocks XSS token theft. localStorage is readable by ANY JavaScript on the page (including injected scripts from a successful XSS). Tokens stored in an in-memory JS closure cannot be exfiltrated via localStorage.getItem. Paired with CSP (Content Security Policy), this makes XSS-based account takeover much harder.
  • HTTP-only refresh tokens. Refresh tokens (long-lived) live in HttpOnly cookies — JavaScript cannot read them at all. Only the browser, talking to Keycloak, can use them. XSS cannot steal long-lived credentials.
  • Short-lived access tokens limit stolen-token damage. Keycloak-issued access tokens expire in minutes. Even if an attacker captures one, they have a small window before it's useless.

  • Keycloak 26.5.4 as OIDC issuer at https://idp.keycloak.net:30000/realms/bookstore
  • Authorization Code Flow with PKCE S256 (SHA-256 challenge)
  • JWT validated at every backend service via Istio RequestAuthentication and Spring Security / FastAPI middleware
  • Customer / admin roles via Keycloak realm roles, enforced by AuthorizationPolicy
  • In-memory token storage in the UI (never localStorage)

Gotchas

  • PKCE requires HTTPS: crypto.subtle is only available in secure contexts. On http://myecom.net:30080 login would fail — hence the HTTPS-only edge (Layer 1).
  • Istio JWT validation is L7: RequestAuthentication only validates the JWT signature and claims; AuthorizationPolicy with requestPrincipals is needed to MANDATE a token for specific paths. In this project, backend services also validate at the application layer for defense-in-depth.
  • Refresh tokens: never exposed to JavaScript. Stored in HTTP-only cookies (Keycloak handles this).
LAYER 6

Application — CSRF, Rate Limiting, Container Hardening

Gateway ext_authz CSRF · Bucket4j rate limits · runAsNonRoot · readOnlyRootFilesystem · drop ALL capabilities

What This Layer Means

The Application layer protects against attacks that a valid JWT can't stop — where the user IS authenticated but is being tricked, overloaded, or the container itself is compromised. Three categories:

  • CSRF protection. The gateway runs csrf-service as an Istio ext_authz provider. Every mutating request (POST/PUT/DELETE/PATCH) must carry a valid X-CSRF-Token header that matches a UUID stored in Redis for that JWT.
  • Rate limiting. Bucket4j token-bucket filters at ecom-service cap mutations per user: CART 60/min, CHECKOUT 10/min, ADMIN 30/min, BOOKS 200/min. Excess requests return HTTP 429.
  • Container hardening. Every pod runs as non-root (UID 1000), on a read-only root filesystem, with ALL Linux capabilities dropped, allowPrivilegeEscalation: false, and seccompProfile: RuntimeDefault.

Why This Layer Matters for Security

  • CSRF stops cross-origin trickery. If a user is logged in and visits evil.com, that site can't easily call api.service.net/ecom/checkout on the user's behalf — even with cookies, because the attacker doesn't know the per-session CSRF token stored in Redis. Prevents silent fund transfers, profile changes, and cart manipulation attacks.
  • CSRF enforced at the gateway, not the app. Putting CSRF checking in csrf-service (Istio ext_authz) means every downstream service benefits automatically. A developer cannot accidentally skip CSRF validation by forgetting a decorator.
  • Rate limiting defeats brute-force and resource exhaustion. Without limits, an attacker with a valid customer JWT could fire thousands of /checkout requests per second, locking inventory, flooding Kafka, or draining credit card processor quota. Per-user buckets isolate abusers from honest users.
  • Rate limiting throttles credential-stuffing. Even with weak passwords, automated login attempts slow down at the rate limit — reducing success rate of stolen-credentials attacks.
  • runAsNonRoot prevents root escalation. Many container escapes require root in the container to begin with. Running as UID 1000 (non-root) closes most kernel-exploit paths that rely on root capabilities.
  • readOnlyRootFilesystem blocks in-place persistence. Malware inside a pod cannot modify the binary or drop persistence files. On the next restart, the pod comes back clean. Forensics also benefit: the filesystem IS the signed image.
  • drop ALL capabilities eliminates the kernel surface area. Linux capabilities grant root-adjacent powers (CAP_NET_ADMIN, CAP_SYS_PTRACE, etc.). Dropping ALL and then re-adding ONLY what the app needs (usually: nothing) means a kernel CVE affecting a specific capability doesn't affect this workload.
  • allowPrivilegeEscalation: false blocks setuid binaries. Prevents tools like sudo, ping, or any setuid binary from elevating privileges inside the container — commonly exploited in multi-tenant scenarios.
  • seccompProfile.RuntimeDefault blocks ~55 syscalls known to be dangerous. The default seccomp profile blocks syscalls used by rootkits and kernel-exploit payloads (e.g. keyctl, add_key, kexec_load). A huge reduction in kernel attack surface.

Control (from architecture.html)

  • CSRF tokens (Redis-backed, UUID v4, 10min sliding TTL)
  • Rate limiting via Bucket4j (token bucket, per-user)
  • Containers run as non-root with read-only filesystem
  • Drop ALL Linux capabilities; allowPrivilegeEscalation: false

Gotchas

  • CSRF validation runs at the gateway, not in the service: the Istio CUSTOM AuthorizationPolicy calls csrf-service as an ext_authz provider BEFORE the request reaches the backend. Backend services trust the header was validated upstream.
  • Rate limits reset on pod restart because the ConcurrentHashMap is in-memory. For distributed rate limiting, move to Redis-backed Bucket4j.
  • Read-only root filesystem needs emptyDir volumes for /tmp — Spring Boot's default temp dir must be writable for multipart uploads.
