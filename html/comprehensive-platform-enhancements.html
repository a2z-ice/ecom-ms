Foundation (Sessions 1-7)

The initial sessions established the core platform: a kind cluster with Istio Ambient Mesh, Kubernetes Gateway API routing, PostgreSQL databases, and the first two application services.

Why this approach produces the best outcome: Starting with infrastructure-first (cluster, mesh, gateway) before application code ensures that every service deployed from day one operates within the security and networking model used in production. Retrofitting a mesh or gateway is far more error-prone than building on top of one.

Identity & Security (Sessions 8-12, 23, 30)

Security was layered incrementally: Keycloak OIDC for identity, Istio mTLS for transport, NetworkPolicies for network isolation, container hardening for defense-in-depth, and rate limiting for abuse prevention.

Why this approach produces the best outcome: Defense-in-depth means no single layer failure compromises the system. JWT validation at every service (not just the gateway) prevents token relay attacks. PKCE with in-memory token storage eliminates XSS token theft. Default-deny NetworkPolicies ensure that even if a pod is compromised, lateral movement is blocked.

Event-Driven Architecture (Sessions 13-19, 22)

The CDC pipeline captures every database mutation via Debezium Server, streams through Kafka, transforms with Flink SQL into a star schema, and visualizes in Superset dashboards.

Why this approach produces the best outcome: Debezium Server (not Kafka Connect) eliminates the need for a Connect cluster, reducing operational overhead. Flink SQL with exactly-once semantics ensures analytical data is never duplicated or lost. The star schema in analytics-db means services never need cross-database queries — strict isolation is preserved while analytics gets a unified view.

Data Reliability (Sessions 14, 27)

Data persistence evolved from ephemeral containers to CloudNativePG-managed HA clusters with automatic failover, replication, and PVC-backed storage for all stateful services.

Why this approach produces the best outcome: CloudNativePG manages primary/standby replication natively in Kubernetes, providing automatic failover without external tooling. ExternalName Service aliases mean applications reference a stable DNS name while CNPG handles instance promotion transparently. PVC-backed storage survives pod restarts and node failures.

Observability (Sessions 14, 22-23)

A full observability stack was built progressively: Prometheus for metrics, Grafana for dashboards, Loki for centralized logs, Tempo for distributed traces, OTel Collector as the telemetry hub, and Kiali for service mesh visualization.

Why this approach produces the best outcome: The OTel Collector as a single telemetry pipeline means services emit once and the collector routes to multiple backends (Loki, Tempo, Prometheus). Resource attributes (service.name, service.namespace) enable precise filtering. Kiali provides real-time mesh topology that raw metrics cannot — it shows actual traffic patterns and mTLS status at a glance.

TLS & Certificates (Sessions 24-25)

cert-manager automates the entire certificate lifecycle: a self-signed CA issues leaf certificates with 30-day rotation. The Cert Dashboard Operator (a custom Go operator) provides a web UI for monitoring and one-click renewal.

Why this approach produces the best outcome: Automated rotation eliminates certificate expiry outages — the number one cause of unplanned TLS downtime. The self-signed CA with ECDSA P-256 provides strong cryptography without external CA dependency. The Cert Dashboard Operator turns certificate operations from kubectl commands into visual, auditable actions accessible to any team member.

Application Resilience (Sessions 30-33)

Resilience patterns were applied systematically: circuit breakers prevent cascade failures, DLQ handles poison messages, HPA scales under load, and PDB prevents disruption during rolling updates.

Why this approach produces the best outcome: Resilience4j circuit breakers with half-open probing allow automatic recovery without manual intervention. DLQ with admin retry ensures no message is silently lost. HPA based on CPU+memory (not just one metric) prevents both CPU saturation and OOM kills. PDB guarantees at least one replica serves traffic during voluntary disruptions like node drains.
AspectBefore (Session 29)After (Session 33)
Circuit breakingNone (cascading failures possible)Resilience4j: 50% threshold, 60s wait, half-open probe
Message failure handlingSilent drop or infinite retryRetry 3x with backoff then route to DLQ
DLQ managementN/AAdmin API for inspect, retry, purge
Auto-scalingFixed replica countHPA: CPU+memory targets, 2-4 replicas
Disruption safetyNonePDB: minAvailable 1 per service
IdempotencyNone (duplicate processing risk)Idempotency-Key header support
Rate limitingBasicTiered: CART 60/min, CHECKOUT 10/min, ADMIN 30/min

Infrastructure Hardening (Session 34)

Session 34 focused on production hardening of core infrastructure: Kafka compression and probes, Redis memory management, consumer commit safety, resource quotas, and API documentation security.

Why this approach produces the best outcome: LZ4 compression reduces Kafka network bandwidth 10-30% with minimal CPU cost. Manual consumer commits with error handling prevent offset advancement on failed processing (the root cause of silent data loss). ResourceQuota + LimitRange prevent any single namespace from starving others. Disabling Swagger in production eliminates a reconnaissance vector.

Session 34 — Detailed Configuration & Code Changes

34.1 Kafka Production Configs (infra/kafka/kafka.yaml)

Added 6 environment variables to the Kafka container for production hardening:

- name: KAFKA_COMPRESSION_TYPE
  value: "lz4"                    # Reduces disk/network I/O 10-30%
- name: KAFKA_LOG_RETENTION_HOURS
  value: "168"                    # 7-day explicit retention
- name: KAFKA_DELETE_RETENTION_MS
  value: "86400000"               # 24h tombstone retention for compacted topics
- name: KAFKA_MIN_INSYNC_REPLICAS
  value: "1"                      # Documents single-broker intent
- name: KAFKA_UNCLEAN_LEADER_ELECTION_ENABLE
  value: "false"                  # Prevents data loss on failover
- name: KAFKA_NUM_PARTITIONS
  value: "3"                      # Default for auto-created topics

34.2 Kafka Exec Liveness Probe (infra/kafka/kafka.yaml)

Replaced TCP socket probe with exec probe that validates actual broker API availability:

# BEFORE (only checks if port is open):
livenessProbe:
  tcpSocket:
    port: 9092

# AFTER (validates broker API responds correctly):
livenessProbe:
  exec:
    command: [sh, -c, "kafka-broker-api-versions --bootstrap-server localhost:9092 | grep -q ApiVersion"]
  initialDelaySeconds: 60
  periodSeconds: 30
  timeoutSeconds: 10

34.3 Redis Production Configs (infra/redis/redis.yaml)

Added 5 flags to the redis-server command array:

command:
  - redis-server
  - --requirepass $(REDIS_PASSWORD)
  - --appendonly yes
  - --save "60 1"
  - --maxmemory 200mb              # Prevents OOMKill (container limit 256Mi)
  - --maxmemory-policy allkeys-lru # Evicts LRU keys when full
  - --tcp-backlog "511"            # Matches Linux default
  - --timeout "300"                # Close idle connections after 5 min
  - --tcp-keepalive "60"           # Detect dead connections in 60s

34.4 Spring Redis Connection Pool (ecom-service/src/main/resources/application.yml + pom.xml)

Added Lettuce pool config under spring.data.redis:

spring:

      timeout: 5s

34.6 Kafka Consumer Commit Safety (inventory-service/app/kafka/consumer.py)

Wrapped consumer.commit() in try/except — reprocessing is safe because _deduct_stock uses SELECT FOR UPDATE:

# BEFORE:
await consumer.commit()

# AFTER:
try:
    await consumer.commit()
except Exception as exc:
    logger.error(
        "Failed to commit offset for orderId=%s: %s — may be reprocessed on restart",
        order_event.get("orderId"), exc,
    )

34.7 DLQ Consumer Manual Commit (inventory-service/app/kafka/dlq_consumer.py)

Changed from auto-commit to manual commit with error handling:

# BEFORE:
enable_auto_commit=True   # Messages can be lost if consumer crashes between auto-commit intervals

# AFTER:
enable_auto_commit=False
# After processing each message:
try:
    await consumer.commit()
except Exception as exc:
    logger.error("Failed to commit DLQ offset: %s", exc)

34.8 Checkout Idempotency Key

Liquibase migration (006-add-idempotency-key.yaml):

- addColumn:
    tableName: orders
    columns:
      - column:
          name: idempotency_key
          type: VARCHAR(64)
- addUniqueConstraint:
    tableName: orders
    columnNames: idempotency_key
    constraintName: uq_orders_idempotency_key

Order.java — new field:

@Column(name = "idempotency_key", unique = true)
private String idempotencyKey;

OrderRepository.java — new query:

@EntityGraph(attributePaths = {"items", "items.book"})
Optional<Order> findByIdempotencyKey(String idempotencyKey);

OrderController.java — accepts header:

@PostMapping
public ResponseEntity<OrderResponse> checkout(
        @AuthenticationPrincipal Jwt jwt,
        @RequestHeader(value = "Idempotency-Key", required = false) String idempotencyKey) {
    Order order = orderService.checkout(jwt.getSubject(), idempotencyKey);
    return ResponseEntity.ok(OrderResponse.from(order));
}

OrderService.java — idempotency logic:

// At start of checkout method:
if (idempotencyKey != null && !idempotencyKey.isBlank()) {
    Optional<Order> existing = orderRepository.findByIdempotencyKey(idempotencyKey);
    if (existing.isPresent()) {
        log.info("Idempotent checkout: returning existing order for key={}", idempotencyKey);
        return existing.get();
    }
}
// ... after creating order:
if (idempotencyKey != null && !idempotencyKey.isBlank()) {
    order.setIdempotencyKey(idempotencyKey);
}
