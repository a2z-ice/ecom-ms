1. What Is a Cuckoo Filter?

A Cuckoo filter is a space-efficient probabilistic data structure that answers the question: "Has this item been seen before?" It is named after the cuckoo bird's brood parasitism strategy — when a new item needs to be inserted and its target bucket is full, it kicks out an existing item and rehashes it to an alternate bucket, like a cuckoo pushing eggs out of another bird's nest.

Core Operations

OperationTimeDescription
Insert(item)O(1) amortizedStore the item's fingerprint in one of two candidate buckets
Lookup(item)O(1)Check if the fingerprint exists in either candidate bucket
Delete(item)O(1)Remove the fingerprint from whichever bucket contains it

How It Works Internally

A Cuckoo filter consists of a table of buckets, each holding a fixed number of fingerprints (partial hashes of inserted items). For any item, two candidate buckets are computed:

bucket_1 = hash(item) bucket_2 = bucket_1 XOR hash(fingerprint) // partial-key cuckoo hashing Insert: 1. Compute fingerprint = hash(item) truncated to f bits 2. If bucket_1 or bucket_2 has space → store fingerprint 3. If both full → kick random entry from one bucket 4. Kicked entry re-inserted at its alternate bucket 5. Repeat up to 500 times (then filter is "full") Lookup: 1. Compute fingerprint 2. Check bucket_1 and bucket_2 3. Return true if fingerprint found in either Delete: 1. Compute fingerprint 2. Find and remove from bucket_1 or bucket_2 3. This is impossible with Bloom filters!
Key property: Cuckoo filters can have false positives (saying "yes, seen" when the item was never inserted) but never false negatives (if an item was inserted, Lookup always returns true). The false positive rate for this project's configuration is approximately 0.01% (1 in 10,000).

2. Why Cuckoo Over Bloom?

The critical requirement for CSRF single-use enforcement is deletion. When a rolling window rotates, old tokens must be purged. Bloom filters fundamentally cannot delete entries — they can only be rebuilt from scratch.

FeatureBloom FilterCuckoo FilterWinner
LookupO(k) hash opsO(1) — 2 bucket checksCuckoo
InsertO(k) hash opsO(1) amortizedCuckoo
DeleteNot supportedO(1)Cuckoo
Space per item~10 bits (1% FP)~8 bits (0.26% FP)Cuckoo
False positive rate~1% typical~0.01–0.26%Cuckoo
Thread safetyNeeds external lockNeeds external lockTie
Max load factorNo limit~95% (then insert fails)Bloom
Decision: Cuckoo filter wins because the rolling window pattern requires periodic deletion of expired token JTIs. A Bloom filter would require full reconstruction on every rotation — O(n) vs O(1). The 8-bit-per-item space efficiency is a bonus.

3. Implementation in This Project

The Cuckoo filter is used inside the CSRF Service (Go 1.25) to enforce single-use CSRF tokens. Every HMAC-signed CSRF token contains a unique 128-bit JTI (JWT Token Identifier). After a token is validated once, its JTI is inserted into the Cuckoo filter. Any subsequent attempt to reuse the same token is rejected because the JTI is found during lookup.

Filter Wrapper (internal/cuckoo/filter.go)

A thread-safe wrapper around the github.com/seiflotfy/cuckoofilter library:

package cuckoo

import (
    "sync"
    cuckoo "github.com/seiflotfy/cuckoofilter"
)

// Filter wraps a Cuckoo filter with thread-safe operations.
type Filter struct {
    cf *cuckoo.Filter
    mu sync.RWMutex    // RWMutex: multiple concurrent readers, exclusive writer
}

// Insert adds a JTI to the consumed set.
func (f *Filter) Insert(jti []byte) bool {
    f.mu.Lock()          // Exclusive write lock
    defer f.mu.Unlock()
    return f.cf.Insert(jti)
}

// Lookup checks if a JTI has been consumed.
func (f *Filter) Lookup(jti []byte) bool {
    f.mu.RLock()         // Shared read lock (concurrent lookups allowed)
    defer f.mu.RUnlock()
    return f.cf.Lookup(jti)
}
Thread safety: sync.RWMutex allows multiple goroutines to perform concurrent Lookup operations (the hot path for ext_authz checks), while Insert and Delete take an exclusive lock. This is critical because Envoy sends hundreds of concurrent ext_authz requests.

Rolling Filter (internal/cuckoo/rolling.go)

A dual-filter structure that provides time-based expiration without a wall clock on individual entries:

// RollingFilter manages two Cuckoo filters with automatic rotation.
type RollingFilter struct {
    current  *Filter     // Receives new inserts
    previous *Filter     // Checked on lookups (grace period)
    capacity uint
    mu       sync.RWMutex
}

// Lookup checks BOTH filters (current OR previous).
func (rf *RollingFilter) Lookup(jti []byte) bool {
    rf.mu.RLock()
    defer rf.mu.RUnlock()
    return rf.current.Lookup(jti) || rf.previous.Lookup(jti)
}

// Rotate discards previous, moves current to previous, creates new current.
func (rf *RollingFilter) Rotate() {
    rf.mu.Lock()
    defer rf.mu.Unlock()
    rf.previous = rf.current           // Old current becomes grace-period filter
    rf.current = NewFilter(rf.capacity) // Fresh empty filter for new inserts
}

// StartAutoRotation begins periodic filter rotation (matches token TTL).
func (rf *RollingFilter) StartAutoRotation(interval time.Duration) func() {
    ticker := time.NewTicker(interval)
    done := make(chan struct{})
    go func() {
        for {
            select {
            case <-ticker.C: rf.Rotate()
            case <-done: ticker.Stop(); return
            }
        }
    }()
    return func() { close(done) } // Returns stop function
}

5. Three-Tier Validation Architecture (L1/L2/L3)

The Cuckoo filter is the L1 tier in a three-tier validation architecture. Each tier serves a specific purpose:

L1: Cuckoo Filter

~50ns

In-memory, per-pod
Single-use enforcement. Catches replay attacks instantly. No I/O, no network.

L2: HMAC-SHA256

~1µs

Stateless, cryptographic
Signature verification, TTL check, user binding, origin binding. Zero external dependencies.

L3: Redis Set

~1ms

Cross-pod dedup
Ensures single-use across all CSRF service replicas (2+ pods). Optional — fail-open if Redis is down.

Validation Order and Why It Matters

The tiers are checked in a specific order that optimizes for the common case (valid, first-use token) while catching attacks as early as possible:

Validate(ctx, userID, reqToken, origin) {
    // L2 first: cheapest rejection (no state needed)
    payload := hmac.Verify(reqToken)     // ~1µs, rejects tampered/expired
    if payload.Sub != userID { return false }  // User binding
    if payload.Org != origin { return false }  // Origin binding

    jti := payload.Jti[:]

    // L1 second: in-memory, catches same-pod replays (~50ns)
    if cuckooL1.Lookup(jti) { return false }  // Already consumed!

    // L3 third: cross-pod check (~1ms, optional)
    if redis.SIsMember("csrf:consumed", jti) {
        cuckooL1.Insert(jti)  // Promote to L1 for faster future rejection
        return false
    }

    // Valid! Mark consumed immediately.
    cuckooL1.Insert(jti)                    // L1: instant (~50ns)
    go redis.SAdd("csrf:consumed", jti)      // L3: async (fire-and-forget)

    return true
}
Why L2 before L1? HMAC verification is stateless and rejects the majority of invalid tokens (tampered, expired, wrong user) without touching the Cuckoo filter at all. Only cryptographically valid tokens reach the Cuckoo filter, keeping the filter's load low and focused on replay detection.

6. Rolling Window: Time-Based Expiration

Individual Cuckoo filter entries have no TTL. Instead, the RollingFilter uses a two-window rotation pattern to provide time-bounded expiration with a grace period:

Rolling Window Timeline (Token TTL = 10 minutes) ============================================================================ T=0min Startup current = FilterA (empty) previous = FilterB (empty) T=3min Token X validated, JTI inserted into FilterA (current) current = FilterA [X] previous = FilterB [] T=10min AUTO-ROTATION #1 current = FilterC (new, empty) ← fresh filter previous = FilterA [X] ← old current becomes grace period FilterB is garbage collected ← old previous discarded T=12min Replay of Token X attempted Lookup checks: FilterC? NO. FilterA? YES, found! → REJECTED (grace period still active) T=20min AUTO-ROTATION #2 current = FilterD (new, empty) previous = FilterC [] FilterA is garbage collected ← Token X's JTI is now gone T=22min Replay of Token X attempted Lookup checks: FilterD? NO. FilterC? NO. But L2 HMAC check: iat=T+3min, TTL=10min → EXPIRED! → REJECTED by HMAC TTL check (before reaching Cuckoo)
Defense in depth: Even after a JTI ages out of both Cuckoo filters (after ~2x TTL), the token is still rejected because the HMAC's embedded iat timestamp will have expired. The Cuckoo filter provides fast single-use enforcement within the TTL window; HMAC provides the ultimate time barrier.

Why Two Filters Instead of One?

Without a "previous" filter, tokens consumed just before rotation would immediately become re-usable. The grace period ensures that a token consumed at T=9:59 is still detected as consumed at T=10:01 (during the next window).

Auto-Rotation Lifecycle

In main.go, the rolling filter is started with the same interval as the token TTL:

cf := cuckoo.NewRollingFilter(cfg.CuckooCapacity)  // Default: 1,000,000
stopCuckooRotation = cf.StartAutoRotation(cfg.TokenTTL) // Default: 10 minutes

// On graceful shutdown:
stopCuckooRotation()  // Stops the ticker goroutine

7. Security Analysis at Every Layer

Tracing a mutating request through the entire stack, here is every security check and encryption layer:

1 Browser → Gateway: TLS 1.3

All traffic hits https://myecom.net:30000. TLS is terminated at the Istio Gateway using a cert-manager-issued certificate (self-signed CA, ECDSA P-256, 30-day rotation, 7-day renewBefore). HTTP on port 30080 returns a 301 redirect to HTTPS.

Security guarantee: Eavesdroppers cannot read the CSRF token in transit. MITM attacks are prevented by certificate pinning (for API clients) or CA trust (for browsers).

2 Gateway → ext_authz: Istio mTLS (HBONE)

Inside the mesh, ztunnel wraps all traffic in HBONE tunnels (HTTP/2 CONNECT over mutual TLS). The Gateway's SPIFFE identity (spiffe://cluster.local/ns/infra/sa/istio-ingressgateway) is verified by the destination ztunnel.

Security guarantee: Even if an attacker compromises a pod in another namespace, they cannot impersonate the gateway or intercept ext_authz traffic. PeerAuthentication enforces STRICT mTLS.

3 ext_authz: Origin Validation

The CSRF service checks the Origin header against a whitelist: https://myecom.net:30000, https://localhost:30000, https://idp.keycloak.net:30000. Requests from unknown origins are rejected with 403.

Security guarantee: Prevents cross-origin CSRF attacks from malicious websites. Even if an attacker has a valid CSRF token, they cannot use it from https://evil.com.

4 ext_authz: JWT Validation (Istio + Go)

JWT signatures are validated twice: first by Istio's RequestAuthentication policy (using Keycloak's JWKS endpoint), then the CSRF service extracts claims via base64 decode (trusting Istio's upstream verification). Optional introspection checks if the user account is still active.

Security guarantee: Stolen/forged JWTs are rejected. User binding ensures CSRF tokens are tied to specific users — Token A from User X cannot protect a request from User Y.

5 ext_authz: HMAC-SHA256 Verification (L2)

The CSRF token's cryptographic signature is verified against a rotating key ring. The token's embedded timestamp is checked against the 10-minute TTL. User and origin bindings are validated with constant-time comparison.

Security guarantee: Attackers cannot forge tokens without the HMAC secret key. Expired tokens are rejected. User/origin binding prevents token theft across users or origins.

6 ext_authz: Cuckoo Filter (L1) + Redis (L3)

The 128-bit JTI is checked against the in-memory Cuckoo filter and the cross-pod Redis set. If found in either, the token has already been used and is rejected.

Security guarantee: Single-use enforcement. Even if an attacker intercepts a valid token (via XSS, network sniffing of internal traffic, or log leakage), they have at most one use. Replay attacks are blocked.

7 ext_authz → Backend: mTLS + Spring Security

After CSRF validation passes, the original request is forwarded to the backend (ecom-service) via ztunnel mTLS. Spring Security performs its own JWT validation as defense-in-depth.

Security guarantee: Even if the CSRF service has a bug that allows an invalid request through, the backend independently validates the JWT. Two independent security systems must both fail for an attack to succeed.

8 Response: Auto-Regeneration on Failure

When validation fails (expired or consumed token) but the JWT is valid, the CSRF service generates a new token and includes it in the 403 response body. The UI reads it and retries automatically.

Security guarantee: New tokens are only issued to authenticated users. The 403 response itself is encrypted (TLS). Auto-regeneration prevents UX degradation without weakening security.

8. Attack Scenarios & How Cuckoo Defends

Token Replay Attacker captures a valid CSRF token and reuses it L1 Cuckoo Filter Blocked: JTI found in Cuckoo filter on second use (~50ns rejection)

9. Istio ext_authz Integration

The CSRF service is invoked by Envoy as an external authorization provider. This means every request to protected routes passes through the CSRF service before reaching the backend, regardless of which backend language (Java, Python, Go) handles it.

Istio Mesh Configuration

The CSRF service is registered as an extensionProvider in the Istio mesh config:

# Istio mesh config (istio ConfigMap)
extensionProviders:
  - name: csrf-ext-authz
    envoyExtAuthzHttp:
      service: csrf-service.infra.svc.cluster.local
      port: 8080
      headersToUpstreamOnAllow:
        - authorization
        - x-csrf-token
      headersToDownstreamOnDeny:
        - content-type
        - set-cookie
      includeRequestHeadersInCheck:
        - authorization
        - x-csrf-token
        - origin
        - referer

AuthorizationPolicy (CUSTOM Action)

An AuthorizationPolicy with CUSTOM action tells Envoy to consult the CSRF service for matching requests:

apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: csrf-ext-authz
  namespace: istio-system   # Root namespace = mesh-wide
spec:
  action: CUSTOM
  provider:
    name: csrf-ext-authz
  rules:
    - to:
        - operation:
            paths:
              - "/ecom/*"        # E-Commerce Service
              - "/inven/*"       # Inventory Service
              - "/csrf/token"    # CSRF token generation itself

Request Flow Through Envoy

Envoy ext_authz Decision Flow: Client Request | v Envoy (ztunnel / sidecar) | | Match AuthorizationPolicy? | Path matches /ecom/* → YES | Action = CUSTOM → call ext_authz provider | v csrf-service:8080/ (catch-all route = ExtAuthzCheck handler) | |—— 200 OK: Envoy forwards request to upstream (ecom-service) |—— 403 Forbidden: Envoy returns 403 directly to client |—— 503 Unavailable: Envoy returns 503 to client | v Upstream service (only reached on 200)
Key design point: The CSRF service uses a catch-all route (mux.HandleFunc("/", h.ExtAuthzCheck)) for ext_authz checks. Envoy sends the original request path (e.g., /ecom/cart) to the CSRF service, which only cares about the HTTP method and headers — not the path. The dedicated GET /csrf/token route handles token generation separately.

Summary: Why This Architecture Works

The Key Insight

Traditional CSRF protection requires a database write on every token generation and a database read on every validation. By combining HMAC-SHA256 (stateless cryptographic tokens) with a Cuckoo filter (in-memory single-use tracking), the BookStore platform achieves:

  • Zero I/O for token generation — HMAC tokens are computed in ~1µs with no Redis/DB write
  • Zero I/O for most validations — HMAC verify + Cuckoo lookup = ~1.05µs, no network
  • Redis-optional resilience — if Redis fails, L1 Cuckoo + L2 HMAC still provide full per-pod security
  • Bounded memory — 16 MB for 2 million token capacity, automatically cleaned by rolling rotation
  • Gateway-level protection — all backends (Java, Python, future services) are protected by a single Go service via Istio ext_authz
