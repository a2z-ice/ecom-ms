🔒

TLS Certificate Management

cert-manager v1.17.2 automates the full certificate lifecycle. Self-signed CA (10-year) issues leaf certificates (30-day) with 7-day pre-expiry renewal. ECDSA P-256 keys across the chain. Multi-SAN certificate covers all gateway hostnames. Cert Dashboard Operator provides real-time monitoring with SSE-powered renewal tracking.

🛡

Zero-Trust Network

Every namespace has a default-deny NetworkPolicy. 14 explicit policies whitelist only required traffic paths. Istio AuthorizationPolicies add L4 pod-level access control with SPIFFE identities. No service inherently trusts another — every request is authenticated and authorized independently.

🔑

Identity & Access

Keycloak 26.5.4 provides OIDC Authorization Code Flow with PKCE (S256). Tokens stored exclusively in memory — never localStorage. JWT validation occurs at both the Istio mesh layer (RequestAuthentication) and independently within each backend service (Spring Security, FastAPI middleware). CSRF tokens are server-side in Redis.

🧱

Container Hardening

Every container runs as non-root (UID 1000) with read-only root filesystem. All Linux capabilities dropped. Multi-stage Docker builds produce minimal runtime images. Secrets injected exclusively via Kubernetes Secrets with secretKeyRef — zero hardcoded values.

Reliability — Built to Last

Persistent storage, automated migrations, health checks, and idempotent operations ensure the platform survives restarts, failures, and redeployments.

💾

Persistent Storage

All 4 PostgreSQL instances use PersistentVolumeClaims backed by host directories via local-hostpath StorageClass. Data survives pod restarts and cluster recreation. Kafka data persisted on PVC mount at /var/lib/kafka/data.

🔄

Automated Migrations

Schema migrations run as Kubernetes init containers that must exit 0 before the application starts. Liquibase (Java) and Alembic (Python) ensure databases are always at the correct schema version. Zero manual intervention.

❤

Health Probes

Every service exposes health endpoints consumed by Kubernetes liveness and readiness probes. Kafka uses TCP socket probes. Debezium servers expose /q/health. Failed probes trigger automatic pod restarts.

🔧

Idempotent Scripts

All operational scripts are safe to run multiple times. up.sh auto-detects state (fresh/degraded/healthy) and applies the minimum required actions. No destructive side effects from re-runs.

📋

CDC Offset Tracking

Debezium Server stores WAL offsets in Kafka topics (debezium.ecom.offsets, debezium.inventory.offsets). On restart, processing resumes from the last committed offset — no data loss or duplication.

🔍

200+ E2E Tests

Playwright test suite covers authentication, cart, checkout, CDC pipeline, TLS certificates, Superset dashboards, Kiali topology, cert-dashboard renewal, and more. All tests pass against the live cluster.

Resiliency — Self-Healing Operations

The platform recovers automatically from Docker restarts, pod failures, and infrastructure disruptions through intelligent detection and ordered recovery.

🏗

Smart Recovery (up.sh)

The master startup script auto-detects the cluster state and applies the appropriate recovery strategy:

  • ✓ No cluster → Full bootstrap from scratch
  • ✓ Degraded → ztunnel restart + ordered pod recovery
  • ✓ Healthy → Verify connectors + smoke test
🔄

Post-Docker Recovery

After Docker Desktop restart, the platform executes a precise three-phase recovery:

  • 1 Restart ztunnel (Istio HBONE plumbing)
  • 2 Rolling restart all pods (dependency order: DBs first)
  • 3 Re-register Debezium connectors (Kafka topics rebuilt)

Recovery Flow Diagram

Automated Recovery Pipeline
Docker Restart → up.sh detects → ztunnel restart → DBs restart → Apps restart → Debezium re-register → Smoke test ✓

Resiliency Features

Certificate Auto-Renewal

cert-manager monitors certificate expiry and renews 7 days before expiration. The Cert Dashboard Operator provides visibility and manual renewal triggers via SSE-powered UI.

Kafka Offset Recovery

Debezium stores CDC offsets in dedicated Kafka topics. On pod restart, processing resumes from the last committed position. Flink SQL jobs maintain exactly-once semantics.

Init Container Guards

Application containers cannot start until database migrations complete successfully (exit 0). This prevents data corruption from schema mismatches during rolling updates.

15-Factor App — Full Compliance

The platform adheres to all 15 factors of the modern cloud-native application methodology, extending the original 12-Factor App with API-first design, telemetry, and security.

1

One Codebase, One Application

Single Git repository with clear service boundaries. Each microservice (ecom-service/, inventory-service/, ui/) has its own build, test, and deploy pipeline.

2

API First

RESTful APIs designed before implementation. OpenAPI specs via Spring Boot actuator and FastAPI auto-docs (/docs, /redoc). Gateway HTTPRoutes define the public API surface.

3

Dependency Management

Maven (pom.xml), Poetry (pyproject.toml), npm (package.json). All dependencies explicitly declared. Multi-stage Docker builds isolate build-time from runtime deps.

4

Design, Build, Release, Run

Strict separation: Kubernetes manifests (design), Docker multi-stage builds (build), kind image loading (release), kubectl apply (run). Init containers gate migrations before app start.

5

Configuration, Credentials & Code

All config via environment variables. Secrets via Kubernetes Secrets with secretKeyRef. Spring Boot application.yml uses ${ENV_VAR} references. Zero hardcoded values. VITE_ vars baked at Docker build time.

6

Logs

All services write to stdout only. OTel Collector captures structured logs and forwards to Loki. No local log files. Grafana provides centralized log querying with service_name, level, and namespace labels.

7

Disposability

Fast startup via lightweight containers. Graceful shutdown with signal handling. PVC-backed databases survive pod termination. Kafka offsets persist across Debezium restarts. Init containers ensure clean state.

8

Backing Services

PostgreSQL, Kafka, Redis, and Keycloak treated as attached resources. Each accessed via environment variables (connection strings, URIs). Swappable without code changes.

9

Environment Parity

Local kind cluster mirrors production topology: same container images, same Kubernetes manifests, same Istio mesh, same cert-manager. The only delta is scale (1 replica vs N).

10

Administrative Processes

Migrations run as init containers (Liquibase, Alembic). Keycloak realm import via Job. Superset bootstrap via Job. All admin tasks are automated, repeatable, and version-controlled.

11

Port Binding

Each service self-contained and port-bound: ecom-service (8080), inventory-service (8000), ui-service (80). Kubernetes Gateway API routes external traffic. NodePort maps to host for local access.

12

Stateless Processes

Application services are stateless. Session data in Redis. Cart state server-side in PostgreSQL. Token state in memory (per-request). Horizontal scaling requires no sticky sessions.

13

Concurrency

Process-model scaling: each service runs as independent pods. Spring Boot thread pool, FastAPI async workers, Flink task slots. Kubernetes manages pod lifecycle and scheduling across 3 nodes.

14

Telemetry

Full observability stack: Prometheus (metrics), Loki (logs), Tempo (traces), OTel Collector (pipeline), Grafana (dashboards), Kiali (mesh), Superset (BI). Every layer is instrumented.

15

Authentication & Authorization

Keycloak OIDC with PKCE. JWT validated at mesh (RequestAuthentication) and application (Spring Security, FastAPI middleware) layers. RBAC with realm roles. CSRF protection. mTLS for all internal traffic.
